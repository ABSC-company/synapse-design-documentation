# Contributing — правила ведения дизайн-документации Synapse

> Язык документации: русский.
> Все новые документы пишутся по правилам ниже и на примерах уже готовых модулей (эталоны: `1.3. Ambulant Reception`, `1.0. Layout elements`).

---

## 1. Структура проекта

```
synapse-design-documentation/
├── README.md                        — назначение репозитория, порядок работы
├── CHANGELOG.md                     — журнал согласованных изменений дизайна
├── CONTRIBUTING.md                  — этот документ
├── assets/                          — общие изображения (логотип и т.п.)
└── releases/
    └── v1.0.0/                      — документация конкретной версии системы
        ├── 0. Intro/                — общие положения версии
        │   ├── 0.0. Content.md              — оглавление: карта всех модулей и страниц
        │   ├── 0.1. Scope and goals.md      — scope, цели, проблема, KPI
        │   └── 0.2. Design road-map.md      — road-map и статусы готовности модулей
        ├── 1. Modules and pages/    — модули и их страницы/компоненты
        │   ├── 1.0. Layout elements/        — глобальные компоненты лейаута (Header, Sidebar)
        │   ├── 1.1. <Module>/               — папка модуля
        │   │   ├── 1.1.0. Module layout components/   — общие компоненты модуля (если есть)
        │   │   │   └── 1.1.0.1. <Component>/<Component>.md
        │   │   └── 1.1.1. <Page> Page/<Page>.md       — папка страницы + её спецификация
        │   └── ...
        └── 2. Custom components TR/ — ТЗ кастомных (нестандартных) компонентов
            └── 2.N. <Component>/
```

### 1.1. Правила нумерации и именования

1. **Каждая версия системы** — отдельная папка `releases/vX.Y.Z/`. Документы новой фичи кладутся в папку актуальной версии.
2. **Модуль** — папка вида `1.N. <Название модуля>` внутри `1. Modules and pages/`. Номер модуля закрепляется в `0.0. Content.md` и `0.2. Design road-map.md` и больше не меняется.
3. **Страница** — отдельная папка вида `1.N.M. <Название> Page` внутри папки модуля. Внутри лежит один основной файл спецификации `<Название>.md`. Дополнительные материалы страницы (design-spec, frontend-spec, изображения) кладутся в ту же папку, например:
   `1.5.4. Documents Editor Page/Document editor — Design spec.md`.
4. **Общие компоненты модуля** (используются несколькими страницами модуля) — папка `1.N.0. Module layout components/` с подпапками `1.N.0.K. <Component>`. Пример: `1.3.0.1. Page Header`, `1.3.0.2. Electronic Queue`.
5. **Глобальные компоненты** (видны во всей системе) — в `1.0. Layout elements/` (Header, Sidebar).
6. **Кастомные компоненты** (сложные UI-компоненты, требующие отдельного ТЗ на разработку: таблица, календарь, workflows и т.п.) — в `2. Custom components TR/` с нумерацией `2.N.`.
7. Страницы/модули, запланированные, но не входящие в текущий scope, помечаются суффиксом `(Soon)` в имени папки: `1.6.13. Recruiting (Soon)`.
8. Названия папок и файлов — на английском, содержимое документов — на русском.

### 1.2. Обязательные действия при добавлении модуля или страницы

1. Добавить пункт в оглавление `0. Intro/0.0. Content.md`.
2. Добавить/обновить строку модуля в `0. Intro/0.2. Design road-map.md` (чек-лист и таблица статусов: ТЗ → Техническая документация → Реализация).
3. После каждого согласованного изменения дизайна — запись в `CHANGELOG.md`.

## 2. Правила расписывания страниц

Каждая страница описывается в своём файле `<Название>.md` строго по следующей структуре (порядок разделов фиксированный):

1. **Спецификация страницы интерфейса** (заполняется отдельно для каждой страницы):
   - Страница: название
   - Модуль: к какому модулю относится
   - Позиция в модуле: место страницы в навигации/сценарии модуля
   - Тип страницы: операционная рабочая область / форма / справочник и т.п.
   - Основные роли: кто работает со страницей
   - Статус документа: Draft / Review / Approved
2. **Назначение страницы и сценарии использования** — зачем страница нужна, нумерованный список ключевых задач, общий layout страницы (какие блоки где расположены, со ссылками на компоненты).
3. **User Flows / Task Flows для страницы** — каждый flow отдельным подзаголовком `### Flow N. <Название>`, шаги нумерованным списком «пользователь делает → система отвечает».
4. **Структура: Wireframes** — структурные блоки страницы списком; табличные данные описываются markdown-таблицей «Колонка / Описание»; бизнес-правила отображения — нумерованными списками.
5. **Визуал: High-Fidelity mockups** — визуальные правила, ссылки на макеты.
6. **Состояния и edge cases** — error, empty, loading, offline, permissions.
7. **Interaction & Animation** — правила поведения, переходы, микроанимации.
8. **Зависимости и ограничения** — API, роли, feature flags, технические ограничения.
9. **Acceptance criteria** — проверяемые критерии приёмки.

Дополнительные правила:

- Заголовок документа (`# ...`) содержит номер страницы и её название: `# 1.3.1. Visits Page`.
- Если поведение элемента описано в компоненте — не дублировать, а ссылаться относительной ссылкой на его документ: `см. [Page Header.md](../1.3.0. Module layout components/1.3.0.1. Page Header/Page Header.md)`.
- Термины системы (ЭМК, визит, направление и т.д.) используются единообразно во всех документах.
- Эталонный пример оформления: `1.3. Ambulant Reception/1.3.1. Visits Page/Visits Page.md`.

## 3. Правила расписывания компонентов

Компоненты (глобальные, компоненты лейаута модуля, кастомные) описываются по структуре:

1. **Заголовок** — название компонента: `# Global Header`.
2. **Описание компонента** — что это, где расположен, на каких страницах отображается; список задач компонента («Компонент предназначен для: …»).
3. **Структура компонента** — перечень составных элементов.
4. **Раздел на каждый элемент** (`## <Название элемента>`), внутри которого:
   - расположение элемента внутри компонента;
   - назначение;
   - `### Поведение` — нумерованные правила поведения (клик, повторный клик, сохранение состояния и т.д.);
   - `### Task Flow` — для сложных сценариев, по этапам (`#### Этап N. <Название>`);
   - `### Пример` — при необходимости.
5. **Состояния и edge cases**, **Зависимости и ограничения** — по аналогии со страницами, если применимо.
6. Разделы верхнего уровня отделяются горизонтальной чертой `---`.

Куда класть компонент:

| Тип компонента | Расположение |
| --- | --- |
| Глобальный (виден во всей системе) | `1. Modules and pages/1.0. Layout elements/` |
| Общий для страниц одного модуля | `1.N.0. Module layout components/1.N.0.K. <Component>/` |
| Компонент-событие уровня системы (например, блокировка) | отдельный «модуль»: `1.N. <Event>/1.N.1. <Component>/` |
| Кастомный компонент с отдельным ТЗ на разработку | `2. Custom components TR/2.N. <Component>/` |

Эталонные примеры: `1.0. Layout elements/Header.md`, `1.3.0.1. Page Header/Page Header.md`.

---

## 4. Общие требования к оформлению

1. Формат — Markdown; таблицы, нумерованные списки и относительные ссылки предпочтительнее свободного текста.
2. Изображения, относящиеся к конкретной странице/компоненту, хранятся рядом с её `.md`-файлом; общие для репозитория — в `assets/`.
3. Один документ = одна страница или один компонент. Крупные спецификации разбиваются на файлы внутри той же папки (`— Design spec`, `— Frontend spec`).
4. Перед merge в `main` документ должен пройти вычитку; статус в шапке спецификации меняется с `Draft` на актуальный.
