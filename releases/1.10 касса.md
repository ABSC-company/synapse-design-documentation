Подмодуль «Касса» (`CASH`) отвечает за приём оплаты пациентом по итогам визита и за кассовые операции: смены, чеки, Z/X-отчёты. Работает поверх ЭМК/визита (`EMR`), но не хранит и не показывает кассиру клиническую часть ЭМК — только административные данные, нужные для оплаты (пациент, перечень услуг, сумма).

Раздел детализирован до уровня, достаточного для проектирования модели данных (бэк) и экранов (дизайн). Названия сущностей и полей — рабочие, для проработки, не окончательный нейминг API/БД.

## 2. Статусы и переходы

**Смена:**
| Из | В | Триггер |
|---|---|---|
| — | `открыта` | обязательное действие при первом входе в кассу за день/за рабочее место |
| `открыта` | `закрыта` | закрытие Z-отчёта |
| `закрыта` | `открыта` | восстановление старшим кассиром (см. §3) |

**X-отчёт:** `открыт` → `закрыт` — формируется по требованию (например, кассир уходит с рабочего места); не влияет на статус смены и не сбрасывает итоги Z-отчёта; за смену их может быть несколько.

**Z-отчёт:**
| Из | В | Триггер |
|---|---|---|
| — | `открыт` | создаётся автоматически вместе со сменой |
| `открыт` | `закрыт` | кассир/старший кассир закрывает смену в конце рабочего дня; фискальные итоги фиксируются |
| `закрыт` | `восстановлен` (≈ снова `открыт`) | старший кассир восстанавливает ошибочно закрытый Z-отчёт, указав причину; смена возвращается в `открыта` |

**Платёж:**
| Из | В | Триггер |
|---|---|---|
| — | `черновик` | кассир открыл визит к оплате, способ ещё не подтверждён |
| `черновик` | `отложен` | кассир нажал «Отложить» |
| `черновик` / `отложен` | `оплачен` | оплата наличными/терминалом подтверждена сразу |
| `черновик` / `отложен` | `ожидает_подтверждения` | инициирована онлайн-оплата (Payme/Click/Uzum), ждём колбэк |
| `ожидает_подтверждения` | `оплачен` | платёжная система подтвердила успех |
| `ожидает_подтверждения` | `отменён` | платёжная система вернула отказ/таймаут |
| `черновик` / `отложен` | `отменён` | кассир отменил вручную до оплаты |
| `оплачен` | `возвращён` | проведён Refund |

## 3. Бизнес-правила и валидации

| Правило | Проверка / поведение | Пример сообщения об ошибке |
|---|---|---|
| Обязательное открытие смены | Любое кассовое действие (поиск к оплате, приём платежа, отчёты) заблокировано, пока для текущего кассира/рабочего места нет смены в статусе `открыта` | «Откройте смену, чтобы продолжить работу с кассой» |
| Приём оплаты только в открытую смену/Z-отчёт | Проведение `Payment` в статус `оплачен` невозможно, если Shift/ZReport не в статусе `открыта`/`открыт` | «Смена закрыта, приём оплаты недоступен» |
| Оплата закрывает визит | После перехода платежа в `оплачен` — печатается чек и визит/ЭМК переводится в статус «оплачено» (событие наружу, см. §8) | — |
| Подтверждение онлайн-оплаты — только по колбэку | Статус `оплачен` для `онлайн_*` не выставляется по действию кассира — только по ответу платёжной системы; повторный колбэк с тем же `idempotency_key`/`транзакция_id` не создаёт второй платёж | «Ожидание подтверждения от платёжной системы» |
| Отложенный платёж не закрывает визит | Визит остаётся доступным для оплаты позже, кассир находит его через поиск/«Отложенные платежи» | — |
| Отмена — только до оплаты | Платёж в статусе `оплачен` нельзя отменить — только через `Refund` | «Платёж уже проведён, для отмены оформите возврат» |
| Возврат обязателен с причиной | `Refund` не создаётся без заполненного поля `причина` | «Укажите причину возврата» |
| Возврат не может превышать сумму платежа | Сумма всех `Refund` по одному `Payment` (нарастающим итогом) ≤ `сумма` платежа | «Сумма возврата превышает сумму платежа» |
| Разбивка отчётов по способам оплаты | X- и Z-отчёт агрегируют суммы отдельно по каждому способу (наличные / терминал / Payme / Click / Uzum) — для сверки с фактической выручкой | — |
| Закрытие X-отчёта не закрывает смену | Можно закрывать X-отчёт многократно за смену (например, при пересменке на одном рабочем месте) | — |
| Закрытие Z-отчёта закрывает смену | После закрытия дальнейший приём платежей по этой смене невозможен, пока смена не восстановлена | «Смена уже закрыта» |
| Восстановление Z-отчёта — только у старшего кассира | Требует указания причины, фиксируется в аудите как отдельное событие | «Недостаточно прав для восстановления Z-отчёта» |
| Кассиру не показывается клиническая часть ЭМК | Из ЭМК в кассовый интерфейс передаются только пациент, перечень оказанных услуг и сумма — принцип минимальных привилегий `[SYS-RBAC]`, `[SYS-PDATA]` | — |
| Цены фиксируются на момент выставления к оплате | `перечень_услуг_снимок` не пересчитывается, даже если прайс в `SVC` изменится до фактической оплаты | — |
| Аудит всех денежных операций | Открытие/закрытие смены, платежи, возвраты, восстановление Z-отчёта — под неизменяемый аудит-трейл `[SYS-AUDIT]` | — |

## 4. Роли и права
> В привязке к общей RBAC-модели (`AUTH`), без изобретения своей схемы прав (`[SYS-RBAC]`).

- **Кассир** — открытие смены, поиск визитов к оплате, приём оплаты (наличные/терминал/онлайн), отложить/отменить платёж (до оплаты), печать чека, закрытие X-отчёта.
- **Старший кассир** — все права кассира + закрытие Z-отчёта, восстановление ошибочно закрытого Z-отчёта, проведение возврата.
- **Администратор** *(не роль клиники, а настройка модуля)* — реквизиты платёжных систем (Payme/Click/Uzum), параметры кассовых рабочих мест.

## 5. Пользовательские сценарии

**1. Открытие смены**
- *Основной поток:* при первом входе в кассовый интерфейс система требует открыть смену (кассир указывает сумму размена наличных) → создаются `Shift(открыта)` и `ZReport(открыт)`.
- *Альтернативный поток:* попытка перейти к любому кассовому действию без открытой смены → блокирующее сообщение с предложением открыть смену.

**2. Поиск визита/ЭМК к оплате**
- *Основной поток:* кассир ищет по любому из параметров (ФИО, ПИНФЛ, номер ЭМК, номер визита, дата, отделение) → получает список визитов, ожидающих оплаты (включая отложенные).

**3. Приём оплаты**
- *Основной поток:* кассир открывает найденный визит → видит перечень оказанных услуг и итоговую сумму → выбирает способ оплаты:
  - *наличные* — ввод полученной суммы, расчёт сдачи, немедленное подтверждение;
  - *терминал* — инициируется операция на POS-терминале, ожидание подтверждения оплаты картой;
  - *онлайн (Payme/Click/Uzum)* — генерация ссылки/QR, платёж переходит в `ожидает_подтверждения` до колбэка от платёжной системы.
- *Результат:* платёж `оплачен` → авто-печать чека → визит/ЭМК закрывается по оплате (событие наружу).
- *Альтернативный поток:* платёжная система вернула отказ/таймаут → платёж `отменён`, кассир предлагает другой способ.

**4. Отложить оплату**
- *Основной поток:* кассир помечает платёж как `отложен` без закрытия визита (например, пациент вышел доплатить) → визит остаётся в «Отложенных платежах» текущей смены.

**5. Отменить платёж**
- *Основной поток:* кассир отменяет черновик/отложенный платёж с указанием причины, до перехода в `оплачен`.

**6. Возврат оплаты**
- *Предусловия:* платёж в статусе `оплачен`.
- *Основной поток:* старший кассир открывает оплаченный визит → инициирует возврат → указывает сумму и причину → чек аннулируется, платёж переходит в `возвращён`.

**7. Закрытие X-отчёта**
- *Основной поток:* кассир при уходе с рабочего места формирует промежуточный отчёт по текущим итогам смены → печатает → продолжает работу под той же сменой позже (фискальные итоги Z-отчёта не сбрасываются).

**8. Закрытие Z-отчёта (конец рабочего дня)**
- *Основной поток:* кассир/старший кассир закрывает Z-отчёт → система фиксирует итоги по способам оплаты, смена переходит в `закрыта`, дальнейший приём платежей по ней недоступен.

**9. Восстановление ошибочно закрытого Z-отчёта**
- *Предусловия:* Z-отчёт/смена закрыты преждевременно по ошибке кассира.
- *Основной поток:* старший кассир открывает «Управление сменами» → находит закрытую смену → «Восстановить» → указывает причину → смена и Z-отчёт возвращаются в статус `открыта`/`открыт`, работа продолжается; факт восстановления фиксируется в аудите.

**10. Обработка колбэка от платёжной системы (технический сценарий)**
- *Предусловия:* платёж в статусе `ожидает_подтверждения`, ранее отправлен запрос в Payme/Click/Uzum с `idempotency_key`.
- *Основной поток:* платёжная система присылает колбэк с результатом и `транзакция_id` → система находит `Payment` по `idempotency_key` → сверяет сумму → при успехе переводит платёж в `оплачен` (см. сценарий 3, «Результат») → при неуспехе — в `отменён` с фиксацией причины из ответа.
- *Альтернативный поток:* повторный колбэк с тем же `транзакция_id` (платёжные системы могут слать колбэк более одного раза) → обрабатывается идемпотентно, повторного перехода статуса и повторного чека не происходит.
- *Альтернативный поток:* колбэк не пришёл в течение таймаута → платёж остаётся в `ожидает_подтверждения`, кассиру доступно ручное «Проверить статус» (повторный запрос статуса у платёжной системы) или отмена и повтор оплаты другим способом.

**11. Сверка наличности при закрытии смены (инкассация)**
- *Основной поток:* при закрытии Z-отчёта система показывает расчётную сумму наличных (`сумма_наличных_расчётная` = размен + наличные платежи − наличные возвраты) → кассир/старший кассир пересчитывает физический денежный ящик и вводит `сумма_наличных_фактическая` → расхождение (если есть) отображается и фиксируется вместе с Z-отчётом.

## 6. Экраны (для дизайн-документации)

**«Открытие смены»**
- Поля: сумма размена наличных (число), рабочее место (авто по логину/выбор из списка), кассир (авто из сессии `AUTH`).
- Кнопка «Открыть смену»; пока не нажата — весь остальной интерфейс кассы заблокирован модальным окном.

**«Поиск визитов к оплате»**
- Поле поиска (единая строка, распознаёт ФИО, ПИНФЛ, номер ЭМК, номер визита) + фильтры: дата, отделение, статус (не оплачен / отложен).
- Колонки результатов: *Пациент, № визита, Отделение, Сумма к оплате, Статус*.
- Действие по строке — «Открыть к оплате».

**«Оплата визита»**
- Шапка: пациент (ФИО, ПИНФЛ), № визита, отделение/врач.
- Таблица услуг: *Услуга, Количество, Цена, Сумма*; итоговая строка «Итого к оплате».
- Вкладки способа оплаты:
  - *Наличные* — поле «Получено», авторасчёт «Сдача».
  - *Терминал* — кнопка «Отправить на терминал», индикатор ожидания ответа POS.
  - *Онлайн* — select платёжной системы (Payme/Click/Uzum), QR-код/ссылка, индикатор ожидания колбэка (см. сценарий 10), кнопка «Проверить статус».
- Кнопки: «Оплатить», «Отложить», «Отменить». Кнопка «Оплатить» неактивна, если сумма не выбрана/способ не подтверждён.

**«Чек»** — предпросмотр (перечень услуг, итог, способ оплаты, номер чека, дата/время, кассир) и печать сразу после успешной оплаты; кнопка «Печать повторно» из карточки платежа.

**«Отложенные платежи»** — список: *Пациент, № визита, Сумма, Кассир, Время откладывания*; действие «Вернуться к оплате».

**«X-отчёт»** — кнопка «Сформировать X-отчёт», предпросмотр с разбивкой по способам оплаты (таблица: способ, сумма, кол-во чеков), кнопка «Печать»; без влияния на статус смены.

**«Z-отчёт / Закрытие смены»**
- Сводка: разбивка по способам оплаты, общее количество чеков, расчётная сумма наличных (см. сценарий 11).
- Поле ввода фактической суммы наличных при закрытии, авто-подсветка расхождения.
- Кнопка «Закрыть смену» (недоступна повторно, если уже закрыта).

**«Управление сменами»** *(старший кассир)* — список: *Дата, Кассир, Рабочее место, Статус, Итог*; фильтры по дате/кассиру/рабочему месту; действие «Восстановить» (с обязательным полем «Причина») для закрытых смен.

**«Возврат»** *(старший кассир)* — выбор оплаченного платежа/чека → поле «Сумма возврата» (не более неоплаченного остатка), поле «Причина», кнопка «Провести возврат» → печать чека-возврата.

## 7. Оборудование и внешние интеграции

- **POS-терминал (эквайринг)** — оплата картой инициируется из «Оплаты визита», сумма передаётся на терминал; подтверждение — по ответу терминала/эквайера. Конкретный протокол/SDK — на этапе бэк-реализации.
- **Платёжные системы Payme / Click / Uzum** — интеграция через их API: создание счёта/ссылки на оплату (по `сумма`, `idempotency_key`), приём колбэка (webhook) с результатом (см. сценарий 10). Каждая система обрабатывается отдельным адаптером, но приводится к единой модели `Payment`.
- **Фискальный регистратор / онлайн-ККМ** — вероятно обязателен по законодательству РУз для формирования юридически значимого чека (см. «Открытые вопросы» и системный вопрос О-1); если требуется — `Receipt` печатается через фискальный регистратор, а не просто на обычном принтере, и `фискальный_номер` в `ZReport` — это номер фискального документа, а не внутренний.
- **Принтер чеков** — печать `Receipt`, `XReport`, `ZReport`; отделён от вопроса фискализации (обычный чек-принтер vs фискальный регистратор — см. открытый вопрос).

## 8. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| `CASH-FR-01` | Система требует открытия смены перед любым кассовым действием | must |
| `CASH-FR-02` | Открытие смены автоматически создаёт связанный Z-отчёт | must |
| `CASH-FR-03` | Кассир может найти визит/ЭМК к оплате по любому из параметров: ФИО, ПИНФЛ, номер ЭМК, номер визита, дата | must |
| `CASH-FR-04` | Кассиру доступны только административные данные ЭМК (пациент, услуги, сумма), клиническая часть скрыта | must |
| `CASH-FR-05` | Система поддерживает приём оплаты наличными, через терминал и через онлайн-системы (Payme, Click, Uzum) | must |
| `CASH-FR-06` | Статус оплаты для онлайн-систем подтверждается только колбэком платёжной системы, не действием кассира | must |
| `CASH-FR-07` | После успешной оплаты система автоматически формирует и печатает чек | must |
| `CASH-FR-08` | После успешной оплаты визит/ЭМК переводится в статус «оплачено» | must |
| `CASH-FR-09` | Кассир может отложить оплату визита без его закрытия | must |
| `CASH-FR-10` | Кассир может отменить платёж до его перехода в статус «оплачен» | must |
| `CASH-FR-11` | Возврат по оплаченному платежу доступен только старшему кассиру и требует указания причины | must |
| `CASH-FR-12` | Кассир может сформировать X-отчёт в любой момент открытой смены без влияния на её статус | must |
| `CASH-FR-13` | Закрытие Z-отчёта фиксирует итоги по способам оплаты и закрывает смену | must |
| `CASH-FR-14` | Старший кассир может восстановить ошибочно закрытый Z-отчёт с указанием причины | must |
| `CASH-FR-15` | Все денежные операции и восстановление Z-отчёта фиксируются в аудит-трейле | must |
| `CASH-FR-16` | Повторный колбэк платёжной системы с тем же идентификатором транзакции не создаёт повторного платежа/чека (идемпотентность) | must |
| `CASH-FR-17` | Перечень услуг и цены фиксируются в платеже на момент выставления к оплате и не пересчитываются позже | must |
| `CASH-FR-18` | Сумма возвратов по одному платежу не может превышать сумму самого платежа | must |
| `CASH-FR-19` | При закрытии смены система показывает расчётную сумму наличных и позволяет зафиксировать фактическую с отображением расхождения | should |

## 9. Публичный интерфейс (контракт с другими модулями)

**Синхронные методы:**

| Метод | Вход | Выход | Назначение |
|---|---|---|---|
| `ПолучитьСтатусОплаты` | visit_id/эмк_id | статус (не оплачен / отложен / оплачен / возвращён) | Для `EMR`/`REG`, чтобы знать, закрыт ли визит по оплате |

**Публикуемые доменные события:**

| Событие | Когда | Полезная нагрузка |
|---|---|---|
| `CASH.СменаОткрыта` / `CASH.СменаЗакрыта` | открытие/закрытие смены | смена_id, кассир_id, рабочее_место_id |
| `CASH.ОплатаПринята` | платёж перешёл в `оплачен` | visit_id, сумма, способ_оплаты, номер_чека |
| `CASH.ПлатёжВозвращён` | проведён возврат | visit_id, сумма_возврата, причина |
| `CASH.ZОтчётЗакрыт` | Z-отчёт закрыт | смена_id, итоги_по_способам_оплаты |
| `CASH.ZОтчётВосстановлен` | старший кассир восстановил Z-отчёт | смена_id, причина, кем |

**Вероятные зависимости от других модулей:** сумма к оплате и перечень услуг по визиту — читаются из модуля-владельца биллинга (см. «Открытые вопросы» — кандидаты `EMR`/`SVC`/`REG`); `HR` (кассир, старший кассир как сотрудники); `CLINIC` (рабочие места кассы); `AUTH` (роли и права); внешние интеграции — платёжные системы Payme/Click/Uzum и, вероятно, фискальный регистратор/онлайн-ККМ (не модуль системы, внешний сервис).

## Открытые вопросы

- Кто владеет расчётом суммы к оплате (перечень услуг визита и итог) — `SVC`, `EMR` или отдельная сущность «Счёт»? Нужно решить на этапе сверки контрактов (§6 системного ТЗ), прежде чем фиксировать точную схему обращения `CASH` за суммой.
- Нужна ли поддержка частичной оплаты (пациент оплачивает не всю сумму сразу)?
- Частичный возврат или только полный?
- Список визитов «к оплате» — показывать по всей клинике, только за сегодня, только по своему отделению/кассе?
- Требуется ли интеграция с фискальным регистратором / онлайн-ККМ по законодательству РУз — напрямую влияет на структуру и обязательность Z/X-отчётов (пересекается с системным открытым вопросом О-1).
- Может ли смена передаваться другому кассиру (пересменка на одном рабочем месте без закрытия Z-отчёта), или строго один кассир = одна смена от открытия до закрытия?
- Нужна ли сверка фактической наличности в кассе с расчётной при закрытии смены (инкассация)? *(в этой версии добавлена как поддерживаемая — см. §1.1, сценарий 11 — но нужно подтвердить, обязательна она или опциональна)*
- Как обрабатывается таймаут/неуспех онлайн-оплаты — автоматический повтор, ручная отмена, уведомление кассира (`NOTIF`)?
- Конкретный протокол/SDK интеграции с Payme/Click/Uzum и с эквайринговым POS-терминалом — на этапе бэк-реализации, влияет на формат `транзакция_id`/`idempotency_key`.
- Обязательна ли фискализация чеков через онлайн-ККМ по законодательству РУз для медицинских услуг — определяет, нужен ли отдельный фискальный регистратор или достаточно обычной печати чека (пересекается с системным вопросом О-1).
- Что делать с расхождением фактической и расчётной наличности при закрытии смены — просто фиксировать в Z-отчёте или требовать обязательного комментария/эскалации старшему кассиру при превышении порога?
