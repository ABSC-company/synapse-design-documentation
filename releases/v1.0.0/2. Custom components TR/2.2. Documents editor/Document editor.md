Редактор документов предназначен для построения переиспользуемой структуры документа - шаблона. Шаблоны документов используются для подготовки на их основе готовых документов, с подставленными значениями (в виде переменных; данных, полученных от пользователя через поля ввода; вычисленные результаты формул; вычисленные результаты условий) и отформатированным контекстом.

Технические требования для дизайна с описанием бизнес логике на Front-end:

- элементы документа (контентная часть):
    - текст
    - изображение
    - таблица
    - список
    - переменные
    - поля ввода
    - подписи
    - формула
    - результат по условию

- панель инструментов:
    - документ: колонки (2 колонки) или на одной строке (уровне) совмещать изображение с текстом
    - текст: жирный
    - текст: курсив
    - текст: зачёркнутый
    - текст: подчёркнутый
    - текст: цвет
    - текст: размер шрифта
    - текст: позиционирование (с правого/левого края; по центру; в ширину)
    - изображение: добавить/удалить изображение (с возможностью изменения размера)
        - drop area / file input / url input
    - таблицы: добавить/удалить таблицу
    - таблицы: добавить/убрать строку
    - таблицы: добавить/убрать столбец
    - таблицы: объединить ячейки
    - таблицы: закрасить ячейки
    - список: добавить/удалить список
    - переменная: вставить переменную (дальше список вычисляемых переменных)
        - текущая дата
        - текущая время
        - название клиники
        - логотип клиники
        - название филиала
        - название главного филиала
        - адрес филиала
        - адрес главного филиала
        - контакты филиала: соц. сети
        - контакты филиала: эл. почта
        - контакты филиала: номера телефона
        - контакты главного филиала: соц. сети
        - контакты главного филиала: эл. почта
        - контакты главного филиала: номера телефона
        - ФИО заполняющего документа (по отдельности, т.е. 3 переменные)
        - специализация заполняющего врача
        - категория заполняющего врача
        - отдел заполняющего
        - должность заполняющего
        - ФИО пациента (по отдельности, т.е. 3 переменные) (если участвует в документе)
        - дата рождения пациента (если участвует в документе)
        - пол пациента (если участвует в документе)
        - адрес пациента (если участвует в документе)
        - группа крови пациента (ABO) (если участвует в документе)
        - ресурс фактор крови пациента (Rh) (если участвует в документе)
        - рост пациента (если участвует в документе)
        - вес пациента (если участвует в документе)
        - ИМТ пациента (если участвует в документе)
        - аллергии пациента (если участвует в документе)
        - хронические заболевания пациента (если участвует в документе)
    - поля ввода: использовать/создать поле ввода
    - подписи: добавить подпись
    - формула: добавить формулу
    - результат по условию: добавить результат по условию

- хранение контента происходит в html формате (относительно библиотеки, используемой как готовое решение - TipTap).

- возможность форматировать текста (жирный, курсив, зачёркнутый, подчёркнутый, смена цвета, размер шрифта, по возможности сам шрифт)

- возможность использовать переменные из списка переменных, вместо которых в готовом итоговом документе подставляются значения. В html переменные выглядят согласно синтаксису: `{{ var__(variable id) }}` (например `{{ var__1 }}`). В редакторе необходимо отобразить лишь имя переменной (по свойству name) c подсветкой.

- переменные могут быть вычисляемые (на стороне бекенда) и статичные (данные из баз данных на бекенде). Вычисляемые переменные вычисляются относительно ситуации: кто на данный момент являются объектами документа (заполняющий - врач или иной сотрудник, текущий пациент) или считаются глобальными настройками и вычисляются на стороне бекенда. Статичные переменные - это переменные отражающие в себе данные из баз данных на стороне бекенда. Синтаксис вычисляемых переменных: `{{ var__(идентификатор переменной) }}` (например `{{ var__1 }}`). Синтаксис статичной переменной: `{{ var__(название таблицы)__(идентификатор записи в таблице) }}` (например `{{ var__employees__1 }}`).

- список переменных на фронт приходит сгруппированным по группам. Все вычисляемые переменные входят в группу "Основные", а остальные переменные раскиданы по группам по названию таблиц. Список групп приходит с бекенда. Реализовать отображение списка групп через компонент Accordion. Данные статичных переменных подгружать (get запрос) после раскрытия элемента (группы) аккордиона. Внутри раскрытого списка д.б. инпут поиска по данным.

- при удалении каких-либо данных из бд, которые участвуют в шаблоне документа в качестве статичных переменных перед удалением данных должно открываться модальное окно предупреждающее об использовании данных в шаблонах (список шаблонов) 

- возможность добавлять пользовательские поля ввода в документ, для того чтобы в итоговый документ попадали данные введённые во время заполнения документа по шаблону. В html поля ввода выглядят согласно синтаксису: `{{ input__(input id) }}` (например `{{ input__1 }}`). В редакторе необходимо отобразить лишь название поля ввода (по свойству name) c подсветкой.

- вставка изображения если drop/file input - сначала изображение загружается в систему (File Module (S3) - возвращает id файла из баккета) а в html вставляется ссылка на этот файл (статика) 

- поля ввода можно создавать. Созданные поля ввода можно переиспользовать в текущем или в другом шаблоне документа

- Json схема полей ввода (для фронта и бека):
    - `name` - название поля ввода
    - `id` - идентификатор (PK)
    - `value` - значение
        - если `type` = "input", - строка
        - если `type` = "number_input", - число
        - если `type` = "file_input", - FK (FileModule), id на файл
        - если `type` = "select", - value значения из списка
        - если `type` = "multi_select", - массив из value значения из списка
        - nullable
    - `type`: - тип поля ввода, возможные значения:
        - "input"
        - "number_input"
        - "file_input"
        - "select"
        - "multi_select"
    - `required` - обязательное поле ввода или нет (true/false), false - default
    - `default_value` - значение по умолчанию (nullable)
    - `data`
        - если `type` = "number_input", - массив из чисел максимального и минимального возможного значения (`[0, 0]`)
        - если `type` = "file_input", - mime тип файла который можно загрузить
        - если `type` = "select" || "multi_select", - массив из объектов которые отображаются в списке
        - nullable

```json
{
    "id": 0,
    "name": "",
    "value": null,
    "type": "",
    "required": false,
    "default_value": null,
    "data": null
}
```

- при создании поля ввода с типом `type` == "select" | "multi_select" в список значений добавлять новые можно вручную
- помимо пользовательских полей ввода будут и системные (готовые) поля ввода (с гидратацией данных со стороны бекенда из бд или иных хранилищ). Схема значений поля ввода с типом "select" и "multi_select", которые хранятся в бекенде и гидратации данных нет:
    -  `key` - название значения
    -  `value` - значение (м.б. в любом типе)
```json
"data": [
    {
        "key": "",
        "value": 0
    }
]
```
Схема значений поля ввода с типом "select" и "multi_select", которые хранятся в бекенде и есть гидратация:
```json
"data": {
    "src": "",
    "path": "",
    "query_keys": [""],
    "query_values": [""],
    "key": "",
    "value": ""
}
```
где:
    - `src` - тип ресурса. Возможные значения: sql, json
    - `path` - доп. свойство для уточнения пути данных (название таблицы в бд или название json файла)
    - `query_keys` - доп. свойство для выборки из таблицы БД (колонки таблиц)
    - `query_values` - доп. свойство для выборки из таблицы БД (значения колонок)
    - `key` - свойство (json) или колонка (sql/nosql), которое будет подставляться под `key`
    - `value` - свойство (json) или колонка (sql/nosql), которое будет подставляться под `value`

- при нажатии в панели инструментов на кнопку добавить подпись будет отображаться список используемых в шаблоне подписей и кнопка добавить новую подпись. В каждый элемент списка подписей вложено 5 переменных:
    - signs__signer_department__(sign queue) - отдел подписываемого
    - signs__signer_position__(sign queue) - должность подписываемого
    - signs__signer_fullname__(sign queue) - полное имя подписываемого
    - signs__sign__(sign queue) - его подпись
    - signs__sign_date__(sign queue) - дата подписания

- при добавлении новой подписи указывается необходимые данные. Json схема данных о подписи:
    - `queue` - локальный id (в рамках данного шаблона) подписи
    - `department_id` - FK (Structure Module) 
    - `position_id` - FK (Structure Module) 
    - `employee_id` - FK (HR Module) (nullable)

```json
{
    "queue": 0,
    "department_id": 0,
    "position_id": 0,
    "employee_id": null
}
```

- очерёдность подписания определяется по порядку расположения элементов в массиве signs шаблона

- если не указан `employee_id` документ уйдёт на подпись первому сотруднику найденному через выборку по указанным параметрам из бд.

- при нажатии в панели инструментов на кнопку добавить формулу будет отображаться список готовых формул и кнопка добавить новую формулу.
    - при добавлении новой формулы заполняются поля:
        - `name` - название формулы
        - `formula` - строка с арифметической операцией. При написании арифметической операции в ней можно использовать переменные, поля ввода (если используются в документе), результаты других арифметических операций - формулы (так же проверяются поля ввода в них, а так же нет ли зависимости ещё не вычисленной арифметической операции (проверяется очередность их использования если есть зависимость. Запрещены зависимые результатами друг от друга арифметические операции))
        - `dependencies` - для бекенда, зависимости:
            - `variables` - массив идентификаторов используемых переменных
            - `inputs` - массив идентификаторов используемых полей ввода
            - `formulas` - массив идентификаторов используемых формул
        - `id` - генерируется беком, идентификатор формулы 
    - формула записывается в инпут с упрощённым редактором (для похожего как в редакторе отображения переменных, полей ввода и формул)

- при нажатии в панели инструментов на кнопку добавить результат по условию будет отображаться список готовых результатов по условию и кнопка добавить новый результат по условию.
    - при добавлении нового результата по условию реализуется 
    - json схема результат по условию:
        - `name` - название условия
        - `conditions` - массив условий
            - `type` - тип условия (if, if else, else)
            - `condition` - объект данных о условии
                - `logical` - логический оператор (не обязателен) (AND, OR)
                - `rules` - правила условий
                    - `field` - операнда, которую необходимо проверить (м.б. переменной, полем ввода, формулой)
                    - `operator` - операторы сравнения
                    - `value` - операнда, с которой происходит сравнение (м.б. переменной, полем ввода, формулой)
            - `then` - итог сравнения, если результат правда (м.б. переменной, полем ввода, формулой)
        - `dependencies`
            - `variables` - массив идентификаторов используемых переменных
            - `inputs` - массив идентификаторов используемых полей ввода
            - `formulas` - массив идентификаторов используемых формул
        - `id` - генерируется беком, идентификатор результата по условию 
Свойства в поле `condition` через `rules` могут быть вложены сами в себя, если используется логический оператор "OR"
Пример json структуры, который подготавливает фронтенд:
```json
{
    "name": "",
    "conditions": [
        {
            "type": "if",
            "condition": {
                "logical": "OR",
                "rules": [
                    {
                        "logical": "AND",
                        "rules": [
                            { "field": "{{ var__2 }}", "operator": ">=", "value": "{{ var__6 }}" },
                            { "field": "{{ input__1 }}", "operator": "!=", "value": "some value" }
                        ]
                    },
                    {
                        "logical": "AND",
                        "rules": [
                            { "field": "{{ formula__2 }}", "operator": ">=", "value": 20 },
                            { "field": "{{ var__1 }}", "operator": "==", "value": "{{ var__24 }}" }
                        ]
                    }
                ]
            },
            "then": "{{ var__4 }}"
        },
        {
            "type": "else_if",
            "condition": {
                "logical": "AND",
                "rules": [
                    { "field": "{{ input__12 }}", "operator": "<=", "value": 2 },
                    { "field": "{{ formula__10 }}", "operator": "!=", "value": "some value" }
                ]
            },
            "then": "{{ formula__5 }}"
        },
        {
            "type": "else",
            "then": "{{ var__40 }}"
        }
    ]
}
```

- Схема итогового шаблона (фронт отправляет беку):
    - `name` - название шаблона документа
    - `description` - описание для шаблона документа
    - `pages` - массив страниц с html содержимым
        - `page` - идентификатор страницы
        - `content` - html с сырыми переменными, подписями и полями ввода
    - `dependencies` - списки зависимостей шаблона
        - `variables` - переменные, используемые в шаблоне. Массив из идентификаторов переменных (пример, `[1, 3, 6]`)
        - `inputs` - поля ввода, используемые в шаблоне.  Массив из идентификаторов полей ввода (пример, `[1, 23, 5]`)
        - `signs` - информация и по сути очередь (по порядку расположения в массиве) подписей. Массив из идентификаторов
        - `formulas` - информация и по сути очередь (по порядку расположения в массиве) формул. Массив из идентификаторов
        - `conditions` - список из результатов по условию, используемых в шаблоне документа. Массив из идентификаторов

```json
{
    "name": "",
    "description": "",
    "pages": [
        {
            "page": 1,
            "content": ""
        }
    ],
    "dependencies": {
        "variables": [],
        "inputs": [],
        "signs": [],
        "formulas": [],
        "conditions": []
    }
}
```

Редактор документов с Front-end части может отправить документ в ЭДО (на подписание или просто для уведомления) напрямую или сохранить как шаблон. 2 кнопки "Сохранить как шаблон" и "Отправить" доступны сразу после внесения любого контента в редактор (в документ).
При отправке указывается кому нужно отправить (можно нескольким людям) - эти люди имеют право на просмотр этого документа. Отдельно также назначается допуск к этому документу: общий (смогут смотреть все, у кого есть право получать документы с текущего баккета, так же настраивается права на редактирование и удаление файла) или определяется через список группа лиц, кому разрешено просматривать, редактировать, удалять файл. Если указаны подписи, то эти люди так же имеют право на просмотр файла
