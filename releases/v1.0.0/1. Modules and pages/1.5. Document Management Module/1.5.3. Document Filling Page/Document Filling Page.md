# 1.5.3 Document Filling Page

## Спецификация страницы интерфейса

* Название страницы: Заполнение документа
* Модуль: EDM (Electronic Document Management)
* Тип страницы: Рабочая страница заполнения документа по шаблону (split-layout: форма + превью)
* Основные роли: Администратор, Руководитель, Сотрудник (автор/заполняющий), Подписант
* Статус документа: New / Draft (документ ещё не отправлен в документооборот)
* Платформа: веб-приложение медицинской системы
* Связанные страницы:
  * ← 1.5.1 Documents Page (реестр документов / выбор шаблона)
  * → 1.5.4 Documents Editor Page (редактор шаблона — открывается из кнопки «Изменить шаблон»)
  * → 1.5.2 Document Page (открывается после успешной отправки документа)

---

## Назначение страницы и сценарии использования

Страница предназначена для **заполнения полей ввода документа, построенного по шаблону**, в виде формы и формирования итогового документа для отправки в систему документооборота (ЭДО).

Шаблон (см. 1.5.4 для контекста) содержит статичный контент, переменные (подставляются автоматически), **поля ввода** (заполняет пользователь), блоки подписей, формулы (вычисляемые значения) и результаты по условию (значения, зависящие от логических условий). На этой странице пользователь вводит данные в поля ввода, наблюдает за тем, как меняется итоговый документ в превью, и отправляет его.

Страница визуально разделена на два экрана (**split**):

* **Левая часть** — постраничное заполнение формы. Отображаются только поля ввода документа, сгруппированные по страницам шаблона.
* **Правая часть** — превью итогового документа в реальном времени с подставленными значениями.

С помощью страницы пользователь может:

1. Заполнять поля ввода документа (постранично).
2. Видеть итоговый документ в превью с мгновенным пересчётом переменных, формул и результатов по условию.
3. Перейти к изменению текущего шаблона в редакторе (кнопка «Изменить шаблон»).
4. Отправить документ в документооборот (кнопка «Отправить» или «Подписать и отправить», если в документе есть подпись текущего сотрудника).
5. Перед отправкой проверить и при необходимости изменить метаданные документа (название, описание, список получателей) в модальном окне.

### Основные сценарии использования

* **Сценарий A.** Сотрудник выбирает шаблон в реестре → попадает на страницу заполнения → заполняет поля → отправляет документ на подписание другим сотрудникам.
* **Сценарий B.** Сотрудник является первым/единственным подписантом → заполняет документ → нажимает «Подписать и отправить» → документ уходит дальше по очереди подписей уже подписанным им.
* **Сценарий C.** В процессе заполнения сотрудник обнаруживает, что шаблон требует правок → нажимает «Изменить шаблон» → переходит в редактор (1.5.4), правит, возвращается к заполнению.

## User Flows / Task Flows

### Flow 1. Заполнение документа по шаблону

1. Пользователь открывает страницу заполнения (из реестра по выбору шаблона).
2. Система загружает шаблон, строит форму полей ввода (левая часть) и рендерит превью (правая часть) с подставленными переменными.
3. Незаполненные поля ввода, невычисленные формулы и нерассчитанные результаты по условию отображаются в превью в виде плейсхолдеров (см. раздел про превью).
4. Пользователь заполняет поля на текущей странице формы.
5. По мере ввода превью обновляется: подставляются значения, пересчитываются формулы и результаты по условию.
6. Пользователь переходит к следующей странице формы и заполняет остальные поля.
7. Когда все обязательные поля заполнены — кнопка отправки становится активной.

---

### Flow 2. Отправка документа (без подписи текущего сотрудника)

1. Пользователь заполнил обязательные поля.
2. Нажимает кнопку «Отправить».
3. Открывается модальное окно проверки: название, описание, список получателей (предзаполнены из шаблона, если заданы).
4. Пользователь проверяет/редактирует данные.
5. Нажимает «Отправить» в модальном окне.
6. Система создаёт документ, отправляет его в документооборот.
7. Пользователь перенаправляется на страницу документа (→ 1.5.2) или в реестр с toast-уведомлением об успехе.

---

### Flow 3. Подписать и отправить (документ содержит подпись текущего сотрудника)

1. Документ построен по шаблону, в котором есть блок подписи, назначенный текущему сотруднику.
2. Кнопка отправки имеет подпись «Подписать и отправить».
3. Пользователь заполняет поля и нажимает «Подписать и отправить».
4. Открывается то же модальное окно проверки (название, описание, получатели).
5. После подтверждения система фиксирует подпись текущего сотрудника, создаёт документ и отправляет дальше по очереди подписантов.
6. Пользователь перенаправляется на страницу документа со статусом подписи «Подписано вами».

---

### Flow 4. Изменение шаблона

1. Пользователь в процессе заполнения нажимает «Изменить шаблон».
2. Если введены значения — система предупреждает о возможной потере/несоответствии введённых данных (модальное подтверждение).
3. Пользователь подтверждает переход.
4. Открывается редактор шаблона (→ 1.5.4) с текущим шаблоном.
5. После сохранения изменений пользователь возвращается к заполнению с обновлённой структурой полей.

### Header

Элементы:

* Кнопка «← Назад» — возврат в реестр (с подтверждением, если есть несохранённые данные).
* Заголовок: «Заполнение: "Название шаблона"».
* Кнопка «Изменить шаблон» — переход в редактор (1.5.4) для правки текущего шаблона.
* Кнопка отправки:
  * «Отправить» — по умолчанию.
  * «Подписать и отправить» — если в документе есть блок подписи текущего сотрудника.
  * Недоступна (disabled), пока не заполнены все обязательные поля.

---

### Левая часть — Форма заполнения

Форма строится из **полей ввода** шаблона, сгруппированных по страницам документа (постранично — соответствует разбивке шаблона).

Для каждого поля отображается:

* Название поля (`name` из схемы поля ввода) + маркер обязательности `*`.
* Контрол по типу поля:

| Тип поля        | Контрол в форме                                    |
| --------------- | -------------------------------------------------- |
| `input`         | Однострочное / многострочное текстовое поле        |
| `number_input`  | Числовое поле (с учётом min/max из схемы)           |
| `file_input`    | Загрузчик файла (drag & drop / выбор), фильтр типов |
| `select`        | Выпадающий список (один выбор)                       |
| `multi_select`  | Множественный выбор (чипы / чекбоксы)               |

* Подсказка/ошибка валидации под полем.
* Значение по умолчанию подставляется из схемы поля (если задано).

Навигация по форме:

* Кнопки «← Предыдущая страница» / «Следующая страница →».
* Индикатор прогресса по страницам (`Страница X из N`) и/или степпер-точки.
* Точка/индикатор отмечает страницы, на которых есть незаполненные обязательные поля.

> Поля, которые в шаблоне используются несколько раз (один и тот же чип поля ввода в разных местах), в форме показываются **один раз**; значение подставляется во все вхождения превью.

---

### Правая часть — Превью документа

Превью рендерит **итоговый документ** (HTML из шаблона) с подставленными значениями — белый «лист» A4 с тенью на нейтральном сером фоне (аналогично редактору 1.5.4 и странице документа 1.5.2).

В превью элементы шаблона отображаются так:

| Элемент шаблона          | Заполнено                              | Не заполнено / не вычислено                                   |
| ------------------------ | -------------------------------------- | ------------------------------------------------------------- |
| Переменная               | Подставленное значение (обычный текст) | Подставлена при инициализации (данные системы) — всегда заполнена |
| Поле ввода               | Введённое значение (обычный текст)     | Плейсхолдер: пунктирная «заливка» / `__________` приглушённого цвета |
| Формула                  | Вычисленный результат (обычный текст)  | Метка-заглушка `⟨будет вычислено⟩` приглушённого/курсивного стиля |
| Результат по условию     | Рассчитанное значение (обычный текст)  | Метка-заглушка `⟨зависит от данных⟩` приглушённого стиля       |
| Подпись                  | Блок подписи (ФИО, должность, дата)    | Зона подписи отображается как пустой плейсхолдер подписи        |

Принципы отображения незаполненного в превью:

1. **Поле ввода (пусто)** — на месте значения показывается приглушённый плейсхолдер (например, пунктирная подчёркнутая зона нейтрального цвета), чтобы пользователь видел, где появится текст. Опционально — лёгкая подсветка незаполненных зон, синхронизированная с подсветкой полей в форме.
2. **Формула (не вычислена)** — когда не хватает входных данных (одно из зависимых полей пусто), вместо результата выводится приглушённая метка-заглушка `⟨будет вычислено⟩`. Как только все зависимости заполнены — метка мгновенно заменяется результатом.
3. **Результат по условию (не рассчитан)** — пока условие не может быть однозначно вычислено (входные поля пусты), показывается приглушённая метка `⟨зависит от данных⟩`. После заполнения — подставляется значение соответствующей ветки (ЕСЛИ / ИНАЧЕ ЕСЛИ / ИНАЧЕ).
4. Заглушки **визуально отличаются** от реального контента (приглушённый цвет, курсив/пунктир), но не выглядят как ошибки — это нейтральное «ещё не заполнено».
5. При фокусе на поле в форме соответствующее место в превью может **подсвечиваться/прокручиваться** (scroll-to + highlight), чтобы связать форму и итоговый документ.

Навигация превью:

* `Страница X из N` + кнопки «←» / «→» (если документ многостраничный).
* Прокрутка превью синхронизируется с активной страницей формы (опционально).

## Состояния и Edge Cases

### Loading

* Загрузка шаблона: скелетон формы (слева) и скелетон листа (справа).
* Тяжёлый пересчёт формул/условий или загрузка файла в `file_input`: локальный индикатор у соответствующего поля/места превью, остальная страница интерактивна.

### Empty

* Шаблон без полей ввода (все данные — переменные/формулы): форма показывает информационное состояние «В этом документе нет полей для заполнения» и сразу активную кнопку отправки.
* Превью никогда не пустое — всегда рендерится лист документа (с плейсхолдерами для незаполненного).

### Error

* Шаблон не найден / удалён — страница с сообщением об ошибке и кнопкой «← Назад к документам».
* Ошибка загрузки шаблона — сообщение с кнопкой «Повторить».
* Ошибка валидации полей — поля с ошибкой подсвечиваются, под ними текст ошибки; кнопка отправки заблокирована; страницы с ошибками отмечены в навигации.
* Ошибка отправки — toast с текстом ошибки, состояние кнопки восстанавливается, введённые данные сохраняются.
* Ошибка загрузки файла в `file_input` — сообщение у поля + кнопка повтора.

### Offline

* При потере соединения — нефатальный баннер «Нет соединения. Изменения сохранены локально».
* Введённые значения сохраняются локально (черновик/draft) и не теряются.
* Отправка блокируется до восстановления соединения; при восстановлении — баннер исчезает, отправка снова доступна.

### Permissions

* У пользователя нет прав на создание документа по этому шаблону — страница 403 с кнопкой «← Назад».
* Нет прав на редактирование шаблона — кнопка «Изменить шаблон» скрыта/недоступна (заполнение по-прежнему доступно).
* Документ уже отправлен (повторное открытие черновика, который уже ушёл) — переход на страницу документа (1.5.2) в режиме просмотра.

### Несохранённые изменения

* При попытке покинуть страницу (Назад / Изменить шаблон / закрытие) с заполненными, но не отправленными полями — модальное подтверждение о возможной потере данных.

## Модальное окно: Отправка документа

Открывается по кнопке «Отправить» / «Подписать и отправить». Предназначено для финальной проверки и (при необходимости) корректировки метаданных перед отправкой.

```
┌──────────────────────────────────────────────────────┐
│  Отправка документа                              [✕]  │
├──────────────────────────────────────────────────────┤
│                                                      │
│  Название *                                          │
│  ┌────────────────────────────────────────────────┐  │
│  │  Эпикриз — Иванов И.И.                         │  │  ← предзаполнено
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  Описание                                            │
│  ┌────────────────────────────────────────────────┐  │
│  │  (многострочное поле)                          │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  Получатели *                                        │
│  ┌────────────────────────────────────────────────┐  │
│  │  🔍 Найти сотрудника...                        │  │
│  │  [Петрова А.С. ×]  [Сидоров К.Р. ×]            │  │  ← чипы выбранных
│  └────────────────────────────────────────────────┘  │
│  Указанные сотрудники получат документ.              │
│  Если в документе есть подписи — подписанты          │
│  автоматически добавляются как получатели.           │
│                                                      │
├──────────────────────────────────────────────────────┤
│            [Отмена]    [Подписать и отправить]        │
└──────────────────────────────────────────────────────┘
```

Требования:

* **Название** — обязательное, предзаполнено из шаблона/контекста, редактируемое.
* **Описание** — необязательное, редактируемое.
* **Получатели** — обязательное, множественный поиск сотрудников (чипы). Предзаполнены из настроек шаблона, если заданы.
* Если в документе есть подписи — подписанты автоматически входят в число получателей (с пометкой).
* Текст кнопки подтверждения совпадает с кнопкой страницы: «Отправить» или «Подписать и отправить».
* Кнопка подтверждения недоступна, пока не заполнены обязательные поля (название, минимум один получатель).
* «Отмена» закрывает модальное окно без отправки, введённые в форму данные сохраняются.

## Interaction & Animation

* **Реактивность превью.** Ввод в поле формы мгновенно (с лёгким debounce для числовых/текстовых полей) обновляет соответствующее место в превью. Подстановка значения — плавно (мягкий fade при замене плейсхолдера на значение, 100–150 мс).
* **Пересчёт формул/условий.** При изменении зависимого поля результат формулы/условия пересчитывается и обновляется в превью; переход «заглушка → значение» — плавный fade.
* **Связь форма ↔ превью.** Фокус на поле → подсветка + scroll-to соответствующего места в превью (плавная прокрутка, 150–200 мс).
* **Навигация по страницам формы.** Переключение страниц — горизонтальный slide или fade (150 мс); прогресс-индикатор обновляется.
* **Переключение split на табы (адаптив).** Плавная смена контента таба.
* **Открытие модального окна отправки.** Появление с fade + scale (100–150 мс); закрытие — fade-out (150 мс).
* **Disabled-кнопки.** Переход в активное состояние при заполнении обязательных полей — плавная смена стиля.
* **Загрузка/отправка.** Кнопка отправки переходит в состояние loading (spinner), повторный клик заблокирован.

## Зависимости и ограничения

### API / данные

* Загрузка шаблона (структура: контент, переменные, поля ввода, формулы, условия, подписи).
* Автоподстановка переменных (данные системы: клиника, заполняющий, пациент, дата/время) при инициализации.
* Вычисление формул и результатов по условию (на клиенте по схеме из шаблона либо запросом к бэкенду — согласовать).
* Создание документа и отправка в документооборот (с подписью текущего сотрудника, если применимо).
* Сохранение черновика (draft) при offline / уходе со страницы.
* Загрузка файлов для полей типа `file_input`.

### Роли и права

* Заполнение — право на создание документа по шаблону.
* «Изменить шаблон» — право на редактирование шаблона (иначе кнопка скрыта).
* «Подписать и отправить» — наличие блока подписи текущего сотрудника в документе.

### Технические ограничения

* Превью — рендер HTML итогового документа; вёрстка плейсхолдеров для незаполненного не должна «прыгать» при подстановке значений (резервировать место).
* Формулы не должны быть взаимозависимыми (циклы запрещены ещё в редакторе — 1.5.4); на этой странице при невозможности вычислить выводится заглушка.
* Поля ввода, встречающиеся несколько раз, вводятся один раз и синхронизируются во всех вхождениях.
* Поведение split / перетягиваемого разделителя и адаптив (split → табы) — в рамках единой дизайн-системы.

### Feature flags

* Перетягиваемый разделитель split — опционально (за флагом).
* Синхронная подсветка/scroll-to форма ↔ превью — опционально (за флагом).

## Визуал: High-Fidelity Mockups

Визуальные требования:

1. Split-экран: слева — форма, справа — превью документа; чёткая визуальная граница между частями.
2. Превью: белый лист A4 с тенью на нейтральном сером фоне — единообразно с редактором (1.5.4) и страницей документа (1.5.2).
3. Незаполненные поля ввода в превью — приглушённый плейсхолдер (пунктир/подчёркивание), визуально отличимый от реального текста, но не «ошибочный».
4. Невычисленные формулы — приглушённая метка `⟨будет вычислено⟩`; результаты по условию — `⟨зависит от данных⟩`.
5. Поля формы сгруппированы по страницам; обязательные помечены `*`; ошибки валидации видны под полем.
6. Индикатор прогресса по страницам формы отмечает страницы с незаполненными обязательными полями.
7. Кнопка отправки меняет подпись на «Подписать и отправить» при наличии подписи текущего сотрудника.
8. Подстановка значений и пересчёт формул/условий в превью — плавные, без резких скачков вёрстки.
9. Все состояния (loading, empty, error, offline, permissions) оформлены и единообразны с дизайн-системой.
10. На узких экранах split корректно превращается в табы [Форма] / [Превью].

## Acceptance Criteria

* Страница построена как split: слева форма полей ввода, справа превью итогового документа.
* Форма строится из полей ввода шаблона, сгруппированных постранично; навигация между страницами работает.
* Каждый тип поля ввода (`input`, `number_input`, `file_input`, `select`, `multi_select`) отображается корректным контролом с учётом схемы (min/max, типы файлов, обязательность, значение по умолчанию).
* Поле, встречающееся в шаблоне несколько раз, вводится один раз и синхронизируется во всех вхождениях превью.
* Превью рендерит итоговый документ с подставленными переменными.
* Незаполненные поля ввода отображаются в превью как приглушённые плейсхолдеры.
* Невычисленные формулы отображаются как `⟨будет вычислено⟩`; нерассчитанные результаты по условию — как `⟨зависит от данных⟩`.
* По мере заполнения полей превью обновляется в реальном времени: подставляются значения, пересчитываются формулы и результаты по условию.
* Кнопка «Изменить шаблон» открывает редактор (1.5.4) с текущим шаблоном; при наличии введённых данных показывается предупреждение о возможной потере.
* Кнопка отправки недоступна, пока не заполнены все обязательные поля.
* При наличии в документе подписи текущего сотрудника кнопка имеет подпись «Подписать и отправить».
* По нажатию кнопки отправки открывается модальное окно проверки с полями: название (обязательное, предзаполнено), описание (необязательное), получатели (обязательное, минимум один).
* Подписанты автоматически добавляются в список получателей.
* После подтверждения отправки документ создаётся и отправляется в документооборот; при «Подписать и отправить» фиксируется подпись текущего сотрудника.
* Состояния Loading, Empty, Error, Offline и Permissions обрабатываются и отображаются корректно.
* При уходе со страницы с незаполненными, но не отправленными данными показывается подтверждение.
* На узких экранах split адаптируется в табы [Форма] / [Превью].
