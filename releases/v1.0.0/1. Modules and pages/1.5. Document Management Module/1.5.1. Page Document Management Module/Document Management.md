# 1.5.1 Documents Page

## Спецификация страницы интерфейса

* Название страницы: Документы
* Модуль: EDM (Electronic Document Management)
* Тип страницы: Реестр документов
* Основные роли: Администратор, Руководитель, Сотрудник
* Статус документа: Draft
* Платформа: веб-приложение медицинской системы

---

## Назначение страницы и сценарии использования

Страница предназначена для просмотра, поиска, согласования, подписания и контроля документов внутри клиники.

С помощью страницы пользователь может:

1. Просматривать список документов.
2. Создавать новые документы.
3. Открывать карточку документа.
4. Отслеживать статус документооборота.
5. Выбирать режим подписания документа.
6. Просматривать прогресс подписания.
7. Подписывать документы.
8. Отказывать в подписании документов.
9. Просматривать историю подписания.
10. Добавлять комментарии.
11. Отслеживать документы, ожидающие его действий.

---

## Режимы подписания документа

Система поддерживает два режима подписания документа. Режим выбирается автором документа при отправке документа на подписание и не может быть изменён после отправки.

### Последовательное подписание

* Подписанты подписывают документ строго по очереди.
* Каждый следующий подписант получает доступ к документу только после того, как предыдущий подписант подписал документ.
* В блоке подписантов отображается порядок подписания.
* Текущий подписант визуально выделяется.

### Параллельное подписание

* Все подписанты получают доступ к документу одновременно, сразу после отправки документа на подписание.
* Порядок подписания не предусмотрен и не отображается.
* Подписанты могут подписывать документ в любой момент и в любой последовательности.
* Документ получает статус «Подписан» после того, как подпись поставили все подписанты, независимо от порядка.

## User Flows / Task Flows

### Flow 1. Просмотр списка документов

1. Пользователь открывает раздел «Документы».
2. Система отображает список доступных документов.
3. Пользователь просматривает информацию по документам.
4. Пользователь может открыть документ для просмотра.

---

### Flow 2. Создание документа

1. Пользователь нажимает кнопку «Создать документ».
2. Заполняет данные документа.
3. Сохраняет документ.
4. Документ появляется в списке со статусом «Черновик».

---

### Flow 3. Отправка документа на подписание

1. Автор документа открывает документ.
2. Выбирает режим подписания: последовательное или параллельное.
3. Настраивает список подписантов.
4. Указывает получателей документа.
5. Нажимает кнопку «Отправить на подписание».
6. Документ получает статус «Ожидает подписи».

Поведение в зависимости от режима:

**Последовательный режим:**
- доступ к документу получает только первый подписант в очереди;
- остальные подписанты не имеют доступа до завершения предыдущих этапов.

**Параллельный режим:**
- доступ к документу получают одновременно все указанные подписанты;
- порядок подписания не задаётся.

---

### Flow 4. Подписание документа

1. Подписант открывает документ.
2. Ознакамливается с содержимым.
3. Нажимает кнопку «Подписать».
4. Система сохраняет подпись.
5. Дальнейшее поведение зависит от режима подписания:
   * **Последовательный режим** — документ переходит следующему подписанту в очереди либо, если подписант был последним, получает статус «Подписан».
   * **Параллельный режим** — система проверяет, подписали ли документ все подписанты; если да, документ получает статус «Подписан», если нет — документ остаётся доступен оставшимся подписантам без изменения очереди.

---

### Flow 5. Отказ в подписании документа

1. Подписант открывает документ.
2. Нажимает кнопку «Отказать».
3. Система открывает форму ввода комментария.
4. Пользователь указывает причину отказа.
5. Подтверждает действие.
6. Статус документа меняется на «Отказан».
7. Процесс подписания останавливается независимо от режима подписания (последовательного или параллельного).
8. Автор документа получает информацию о причине отказа.

## Логика подписания документа

### Последовательное подписание

Система поддерживает последовательное подписание документа.

Подписанты подписывают документ строго в порядке очереди.

Пример:

Очередь подписания:

1. Пётр
2. Иван
3. Алексей

После отправки документа:

Документ виден:

* Автору документа.
* Петру.

Документ не виден:

* Ивану.
* Алексею.

После подписания Петром:

Документ становится виден:

* Автору документа.
* Петру.
* Ивану.

Документ всё ещё не виден:

* Алексею.

После подписания Иваном:

Документ становится виден:

* Автору документа.
* Петру.
* Ивану.
* Алексею.

После подписания Алексеем:

Документ получает статус «Подписан».

---

### Параллельное подписание

Система поддерживает параллельное подписание документа.

Все подписанты получают доступ к документу одновременно, без очереди.

Пример:

Подписанты: Пётр, Иван, Алексей.

После отправки документа:

Документ виден:

* Автору документа.
* Петру.
* Ивану.
* Алексею.

Каждый подписант может подписать документ независимо от действий остальных, в любом порядке.

После того как подпись поставили все три подписанта (Пётр, Иван и Алексей — в любой последовательности), документ получает статус «Подписан».

Если хотя бы один из подписантов отказывает в подписании, документ получает статус «Отказан», а процесс подписания останавливается для всех оставшихся подписантов.

---

## Блок подписантов

Отображается отдельным блоком внутри карточки документа.

Состав отображаемых полей зависит от режима подписания документа.

### Последовательный режим

Для каждого подписанта отображается:

* ФИО.
* Должность.
* Порядок подписания.
* Статус подписания.

### Параллельный режим

Для каждого подписанта отображается:

* ФИО.
* Должность.
* Статус подписания.

Порядок подписания не отображается.

---

Возможные статусы подписанта:

* Ожидает подписи.
* Текущий подписант (применимо только для последовательного режима).
* Подписано.
* Отказано.

В последовательном режиме текущий подписант должен визуально выделяться.

В параллельном режиме визуально выделяются все подписанты, ожидающие подписи, без указания текущей очереди.

---

## Подписание документа

Если документ ожидает подписи текущего пользователя:

Отображается блок действий.

Элементы:

* Кнопка «Подписать».
* Кнопка «Отказать».

Кнопка «Отказать» располагается под кнопкой «Подписать».

Условие доступности блока действий зависит от режима подписания:

* **Последовательный режим** — блок действий доступен только текущему подписанту в очереди.
* **Параллельный режим** — блок действий доступен любому подписанту, который ещё не подписал и не отказал.

---

## Отказ в подписании

При нажатии кнопки «Отказать»:

Открывается форма.

Элементы:

* Текстовое поле комментария.
* Кнопка подтверждения отказа.

Требования:

* Комментарий обязателен.
* Кнопка подтверждения недоступна пока комментарий не заполнен.
* После отказа документ получает статус «Отказан».
* Процесс подписания останавливается независимо от режима (последовательного или параллельного) — остальным подписантам доступ к подписанию закрывается.
* Автор документа видит комментарий причины отказа.

---

## Статусы документа

### Draft

Черновик.

Особенности:

* Документ создан.
* Доступен только автору.
* Разрешено редактирование.

---

### Pending

Ожидает первую подпись.

Особенности зависят от режима подписания:

* **Последовательный режим** — документ виден автору и текущему подписанту.
* **Параллельный режим** — документ виден автору и всем подписантам одновременно.

---

### Processed

В процессе подписания.

Особенности:

* Получена минимум одна подпись.
* Отображается прогресс подписания.
* В последовательном режиме доступ переходит следующему подписанту по очереди.
* В параллельном режиме доступ остаётся у всех подписантов, ещё не подписавших документ.

Пример:

2 из 5 подписей.

---

### Signed

Подписан.

Особенности:

* Все подписи получены.
* Документооборот завершён.
* Применимо одинаково для последовательного и параллельного режима.

---

### Delivered

Отправлен на ознакомление.

Особенности:

* Подписание не требуется.
* Документ доступен получателям для просмотра.

---

### Rejected

Отказан.

Особенности:

* Один из подписантов отказал в подписании.
* Отображается причина отказа.
* Документ возвращён автору.
* Процесс подписания останавливается полностью независимо от режима подписания.

---

### Aborted

Отозван.

Особенности:

* Автор отменил процесс подписания.
* Документооборот остановлен.

---

## Визуал: High-Fidelity Mockups

Визуальные требования:

1. Статус документа должен быть заметен во всех представлениях.
2. Режим подписания (последовательный или параллельный) должен быть визуально различим.
3. Прогресс подписания должен отображаться визуально.
4. Текущий подписант должен быть выделен в последовательном режиме.
5. В параллельном режиме все ожидающие подписанты должны быть выделены одновременно, без указания очереди.
6. Подписанные этапы должны отображаться отдельно от ожидающих.
7. Причина отказа должна отображаться в карточке документа.
8. Документы, ожидающие подписи текущего пользователя, должны иметь визуальный индикатор.
9. Пользователь должен видеть количество оставшихся подписей.
10. Интерфейс должен поддерживать большое количество документов и подписантов.
11. Все действия должны соответствовать единой дизайн-системе продукта.

## Состояния и Edge Cases

### Empty

* Документы отсутствуют.
* Отображается Empty State.
* Отображается кнопка создания документа.

### Loading

* Загрузка списка документов.
* Загрузка содержимого документа.
* Выполнение подписания.
* Отправка документа.

### Error

* Документ не найден.
* Ошибка подписания.
* Ошибка загрузки документа.
* Ошибка сохранения.
* Ошибка при выборе режима подписания.

### Permissions

* Пользователь не имеет доступа к документу.
* Пользователь не является текущим подписантом (последовательный режим).
* Пользователь не может подписать документ вне своей очереди (последовательный режим).
* Пользователь не может подписать документ, если он не входит в список подписантов (параллельный режим).

### Locked

Для документов в статусе:

* Signed
* Aborted
* Rejected

Редактирование документа недоступно.

Режим подписания недоступен для изменения после отправки документа на подписание, независимо от текущего статуса.
