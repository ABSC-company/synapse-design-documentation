# 1.5.1 Documents Page

## Спецификация страницы интерфейса

* Название страницы: Документы
* Модуль: EDM (Electronic Document Management)
* Тип страницы: Реестр документов
* Основные роли: Администратор, Руководитель, Сотрудник
* Статус документа: Draft
* Платформа: веб-приложение медицинской системы

---

## Назначение страницы и сценарии использования

Страница предназначена для просмотра, поиска, согласования, подписания и контроля документов внутри клиники.

С помощью страницы пользователь может:

1. Просматривать список документов.
2. Создавать новые документы.
3. Открывать карточку документа.
4. Отслеживать статус документооборота.
5. Просматривать прогресс подписания.
6. Подписывать документы.
7. Отказывать в подписании документов.
8. Просматривать историю подписания.
9. Добавлять комментарии.
10. Отслеживать документы, ожидающие его действий.

---

## User Flows / Task Flows

### Flow 1. Просмотр списка документов

1. Пользователь открывает раздел «Документы».
2. Система отображает список доступных документов.
3. Пользователь просматривает информацию по документам.
4. Пользователь может открыть документ для просмотра.

---

### Flow 2. Создание документа

1. Пользователь нажимает кнопку «Создать документ».
2. Заполняет данные документа.
3. Сохраняет документ.
4. Документ появляется в списке со статусом «Черновик».

---

### Flow 3. Отправка документа на подписание

1. Автор документа открывает документ.
2. Выбирает режим подписания.
3. Настраивает список подписантов.
4. Указывает получателей документа.
5. Нажимает кнопку «Отправить на подписание».
6. Документ получает статус «Ожидает подписи».

Для последовательного режима:
- доступ получает первый подписант.

Для параллельного режима:
- доступ одновременно получают все подписанты.

---

### Flow 4. Подписание документа

1. Подписант открывает документ.
2. Ознакамливается с содержимым.
3. Нажимает кнопку «Подписать».
4. Система сохраняет подпись.
5. Документ переходит следующему подписанту либо получает статус «Подписан».

---

### Flow 5. Отказ в подписании документа

1. Подписант открывает документ.
2. Нажимает кнопку «Отказать».
3. Система открывает форму ввода комментария.
4. Пользователь указывает причину отказа.
5. Подтверждает действие.
6. Статус документа меняется на «Отказан».
7. Автор документа получает информацию о причине отказа.

## Карточка документа

При открытии документа пользователь переходит на страницу просмотра документа.

Страница состоит из следующих блоков:

### Информация о документе

Отображается:

* Название документа.
* Автор документа.
* Категория.
* Статус документа.
* Дата создания.
* Дата обновления.
* Тип доступа.
* Количество просмотров.

---

### Просмотр содержимого документа

Отображается:

* Содержимое документа.
* Страницы документа.
* Прикреплённые файлы.
* История комментариев.
* История подписания.

---

### Панель действий

Набор действий зависит от роли пользователя и статуса документа.

Возможные действия:

* Подписать.
* Отказать.
* Скачать документ.
* Распечатать документ.
* Добавить комментарий.

---

## Логика подписания документа

Система поддерживает последовательное подписание документа.

Подписанты подписывают документ строго в порядке очереди.

Пример:

Очередь подписания:

1. Пётр
2. Иван
3. Алексей

После отправки документа:

Документ виден:

* Автору документа.
* Петру.

Документ не виден:

* Ивану.
* Алексею.

После подписания Петром:

Документ становится виден:

* Автору документа.
* Петру.
* Ивану.

Документ всё ещё не виден:

* Алексею.

После подписания Иваном:

Документ становится виден:

* Автору документа.
* Петру.
* Ивану.
* Алексею.

После подписания Алексеем:

Документ получает статус «Подписан».

---

## Блок подписантов

Отображается отдельным блоком внутри карточки документа.

Для каждого подписанта отображается:

* ФИО.
* Должность.
* Порядок подписания.
* Статус подписания.

Возможные статусы:

* Ожидает подписи.
* Текущий подписант.
* Подписано.
* Отказано.

Текущий подписант должен визуально выделяться.

---

## Подписание документа

Если документ ожидает подписи текущего пользователя:

Отображается блок действий.

Элементы:

* Кнопка «Подписать».
* Кнопка «Отказать».

Кнопка «Отказать» располагается под кнопкой «Подписать».

---

## Отказ в подписании

При нажатии кнопки «Отказать»:

Открывается форма.

Элементы:

* Текстовое поле комментария.
* Кнопка подтверждения отказа.

Требования:

* Комментарий обязателен.
* Кнопка подтверждения недоступна пока комментарий не заполнен.
* После отказа документ получает статус «Отказан».
* Автор документа видит комментарий причины отказа.

---

## Статусы документа

### Draft

Черновик.

Особенности:

* Документ создан.
* Доступен только автору.
* Разрешено редактирование.

---

### Pending

Ожидает первую подпись.

Особенности:

* Документ отправлен на согласование.
* Виден автору и текущему подписанту.

---

### Processed

В процессе подписания.

Особенности:

* Получена минимум одна подпись.
* Отображается прогресс подписания.

Пример:

2 из 5 подписей.

---

### Signed

Подписан.

Особенности:

* Все подписи получены.
* Документооборот завершён.

---

### Delivered

Отправлен на ознакомление.

Особенности:

* Подписание не требуется.
* Документ доступен получателям для просмотра.

---

### Rejected

Отказан.

Особенности:

* Один из подписантов отказал в подписании.
* Отображается причина отказа.
* Документ возвращён автору.

---

### Aborted

Отозван.

Особенности:

* Автор отменил процесс подписания.
* Документооборот остановлен.

---

## Визуал: High-Fidelity Mockups

Визуальные требования:

1. Статус документа должен быть заметен во всех представлениях.
2. Прогресс подписания должен отображаться визуально.
3. Текущий подписант должен быть выделен.
4. Подписанные этапы должны отображаться отдельно от ожидающих.
5. Причина отказа должна отображаться в карточке документа.
6. Документы, ожидающие подписи текущего пользователя, должны иметь визуальный индикатор.
7. Пользователь должен видеть количество оставшихся подписей.
8. Интерфейс должен поддерживать большое количество документов и подписантов.
9. Все действия должны соответствовать единой дизайн-системе продукта.

## Состояния и Edge Cases

### Empty

* Документы отсутствуют.
* Отображается Empty State.
* Отображается кнопка создания документа.

### Loading

* Загрузка списка документов.
* Загрузка содержимого документа.
* Выполнение подписания.
* Отправка документа.

### Error

* Документ не найден.
* Ошибка подписания.
* Ошибка загрузки документа.
* Ошибка сохранения.

### Permissions

* Пользователь не имеет доступа к документу.
* Пользователь не является текущим подписантом.
* Пользователь не может подписать документ вне своей очереди.

### Locked

Для документов в статусе:

* Signed
* Aborted
* Rejected

Редактирование документа недоступно.

## Acceptance Criteria

* Пользователь может просматривать список документов.
* Пользователь может открыть документ.
* Пользователь может видеть автора документа.
* Пользователь может видеть статус документа.
* Пользователь может видеть прогресс подписания.
* Пользователь может видеть очередь подписантов.
* Пользователь может подписать документ.
* Пользователь может отказаться от подписания документа.
* Отказ невозможен без комментария.
* Автор документа получает причину отказа.
* Документ отображается только участникам текущего этапа подписания.
* Следующий подписант получает доступ только после завершения предыдущего этапа.
* Все статусы отображаются корректно.
* Все состояния отображаются корректно.
