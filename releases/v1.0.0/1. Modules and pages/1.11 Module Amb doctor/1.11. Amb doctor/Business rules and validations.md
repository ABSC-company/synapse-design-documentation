## Бизнес-правила и валидации

### Видимость ЭМК и поиск пациента

| Правило | Проверка / поведение |
|---|---|
| ЭМК не видна содержательно до первого обращения | Пока `PatientRecord.дата_первого_обращения` не заполнена, пациент виден в списках (поиск находит его), но содержательные данные ЭМК показываются скрыто/размыто |
| Медицинская история пуста до первой подписанной консультации | `дата_первой_консультации` может отсутствовать даже при заполненной `дата_первого_обращения` (пациент пришёл, но приём не состоялся) — лента истории в этом случае просто пуста |
| Поиск пациента — по всем параметрам | ФИО, номер ЭМК, ПИНФЛ — не только по одному из полей |

### Доступ к ЭМК до начала приёма

| Правило | Проверка / поведение |
|---|---|
| Открытие ЭМК до начала приёма — только по праву на пациента | Врач может открыть ЭМК пациента до начала своего приёма только если пациент: (а) записан к нему через регистратуру, (б) находится в его очереди (талон `ожидает_врача`/`ожидает_вызова` на этого врача), либо (в) направлен к нему в рамках текущего визита |
| Экстренный доступ к чужой ЭМК | Врач может открыть ЭМК вне обычных прав, только указав причину экстренности; действие обязательно фиксируется в аудит-трейле `[SYS-AUDIT]` отдельной пометкой «экстренный доступ» |
| Аллергии всегда на виду | Список аллергий из `PatientRecord` отображается в «шапке» экрана консультации на каждом приёме, а не только в общей ЭМК |
| Доступ к ЭМК — по функциональной необходимости | Принцип минимальных привилегий `[SYS-RBAC]`, `[SYS-PDATA]`; любое открытие/изменение ЭМК — под аудит-трейл `[SYS-AUDIT]` |

### Экстренный приём и триаж

| Правило | Проверка / поведение |
|---|---|
| Регистратура фиксирует, но не решает медицинский вопрос | Регистратура запускает сценарий вне очереди административно (`инициатор_экстренного_приёма_id`, `причина_экстренности`, `время_инициации`) — это не медицинское решение, а фиксация обращения как потенциально экстренного |
| Медицинское подтверждение — отдельная роль | Приём фактически считается экстренным (переход в `идёт_приём` минуя обычный вызов) только после подтверждения дежурным врачом, старшей медсестрой или иным уполномоченным медицинским сотрудником по внутреннему регламенту клиники |
| До подтверждения — приём не в приоритете | Пока триаж не подтверждён, `Encounter` остаётся в `ожидает_вызова`; экстренный приоритет в очереди начинает действовать только после подтверждения |
| Приём начинается по вызову из очереди либо по подтверждённому триажу | В плановом случае — через «Вызвать пациента»; в экстренном — после подтверждения триажа приём переходит в `идёт_приём` напрямую |

### Направление к другому врачу (типы)

| Тип направления | Что происходит с исходным `Encounter` | Что происходит с талоном / новым `Encounter` | Когда применять |
|---|---|---|---|
| `внутри_текущего_визита` | Переходит в `завершён` | У целевого врача создаётся новый `Encounter(ожидает_вызова)` с тем же `visit_id` и тем же `talon_id` — номерок переадресуется | Пациенту нужна консультация другого специалиста прямо сейчас, в рамках того же визита |
| `плановое_на_другой_день` | Не меняется (остаётся `завершён` по завершении текущего приёма как обычно) | Талон/новый `Encounter` не создаются — формируется рекомендация/запись на другой день | Пациенту нужна консультация специалиста, но не в рамках этого визита |
| `с_возвратом_к_текущему_врачу` | Переходит в `приостановлен` (не завершается) | У целевого врача создаётся новый `Encounter` с тем же `visit_id`/`talon_id`; после его завершения `исходный_приём_id` возвращается в `идёт_приём` | Пациент временно уходит на консультацию/исследование и должен вернуться к текущему врачу |
| `без_создания_очереди` | Не меняется | Ничего не создаётся — это просто рекомендация в тексте консультации/документе | Направление носит рекомендательный характер |

| Правило | Проверка / поведение |
|---|---|
| Направление не закрывает визит в клинику | Ни один из типов направления не закрывает `visit_id` — закрывается (или приостанавливается) только конкретный `Encounter` |
| Направление — только на конкретного врача | `Referral.целевой_врач_id` обязателен для всех типов, требующих создания приёма; варианта «только специализация» нет |
| Счёт визита дорастает | Для типов, создающих новый оплачиваемый приём в рамках визита, `Invoice` в `CASH` «дорастает» новой услугой того же визита |

### Логика оплаты

| Правило | Проверка / поведение |
|---|---|
| Платный приём открывается только после подтверждения оплаты | Если направление (`требует_оплаты = true`) или заказ исследования относится к платной услуге, целевой `Encounter` создаётся/остаётся в статусе `ожидает_оплаты` и не участвует в обычном вызове очереди, пока не наступит один из разрешающих случаев ниже |
| Разрешающие случаи | Оплата подтверждена (`CASH.ОплатаПринята`) → `оплачено`; действует страховое покрытие/депозит/корпоративный договор, проверенные `CASH`, → `оплачено`; ответственное лицо явно разрешает продолжение без оплаты, с обязательной причиной, → `разрешено_без_оплаты` |
| PHYS не владеет оплатой | Модуль не хранит суммы, платежи или чеки — только `статус_оплаты` как ссылку на результат проверки в `CASH`; сама проверка и приём оплаты происходят в `CASH` |
| Отложенная оплата — через существующий механизм CASH | Если пациент оплатит позже, `Encounter` остаётся в `ожидает_оплаты`, пока `CASH` не сообщит об изменении статуса счёта |

### Консультация, диагноз, назначения

| Правило | Проверка / поведение |
|---|---|
| Консультация неизменяема после подписания | Правка подписанной `Consultation` создаёт новую версию (`версия + 1`) либо оформляется как `ConsultationAddendum`; старая версия/исходная запись сохраняется целиком, не удаляется `[SYS-TRACE]` |
| Предварительный диагноз допустим при подписании | Врач может подписать консультацию, заполнив только `диагноз_предварительный`, если окончательный диагноз ещё не установлен |
| Уточнение диагноза — только через дополнение | Как только диагноз уточняется/становится заключительным после подписания, это оформляется `ConsultationAddendum`, а не правкой исходной консультации |
| Дополнение не изменяет исходную запись | `ConsultationAddendum` — самостоятельная подписанная запись, связанная с исходной консультацией и видимая в истории ЭМК как её продолжение |
| Структурированные назначения — отдельно от плана лечения | `MedicationOrder`, `ProcedureOrder`, `Recommendation` — машиночитаемые сущности консультации; `план_лечения_рекомендации` остаётся как свободный текст-резюме, не заменяет их |
| Назначение препарата — клиническое, не складское | `MedicationOrder` фиксирует, что врач прописал, а не остаток/выдачу препарата |

### Исследования и очередь

| Правило | Проверка / поведение |
|---|---|
| Заказ на исследование — не через собственную таблицу | Назначение лабораторного/инструментального исследования всегда идёт через публичный метод `LAB`/`INSTR` — этот модуль не хранит заказы и результаты напрямую, только ссылку в консультации |
| Результат исследования подтягивается автоматически | Как только `LAB`/`INSTR` публикует событие о готовом результате, он становится виден в ЭМК и в консультации, без ручного «импорта» врачом; если консультация уже подписана — результат становится основанием для `ConsultationAddendum` |
| Повторный вызов пациента | Врач может повторно вызвать пациента до исчерпания настраиваемого количества попыток; после этого талон переходит в `неявка`/`пропущен_врач`; регистратура может вручную вернуть пациента в очередь |
| Порог аномальной длительности — опциональный, per-клиника | Клиника может не включать это ограничение вовсе; если включено — значение задаёт администратор этой клиники |
| Аномально долгая консультация — предупреждение и обязательная причина | Если приём (`идёт_приём`) длится дольше порога, система показывает врачу предупреждение; подписать консультацию сверх порога можно только заполнив `причина_длительной_консультации` |
