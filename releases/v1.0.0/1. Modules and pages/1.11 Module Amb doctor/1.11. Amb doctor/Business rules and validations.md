## Бизнес-правила и валидации

| Правило | Проверка / поведение |
|---|---|
| ЭМК не видна до первого визита | Пока `PatientRecord.дата_первого_визита` не заполнена, пациент виден в списках (поиск находит его), но содержательные данные ЭМК показываются скрыто/размыто |
| Поиск пациента — по всем параметрам | ФИО, номер ЭМК, ПИНФЛ — не только по одному из полей |
| Приём начинается только по вызову из очереди, кроме экстренного триажа | Врач видит и открывает приём через «Вызвать пациента»; исключение — `триаж_уровень = экстренный`, когда приём открывается сразу, минуя обычный порядок вызова |
| Экстренный приём инициирует регистратура (в амбулаторном сегменте) | Решение о том, что пациенту нужен приём вне очереди, принимает регистратура при обращении пациента (`REG.ОткрытьЭкстренныйПриём`) — не сам врач и не медсестра; в стационаре аналогичную роль будет играть приёмный покой |
| Порог аномальной длительности — опциональный, per-клиника | Клиника может не включать это ограничение вовсе; если включено — конкретное значение `порог_аномальной_длительности_приёма_минут` задаёт администратор этой клиники, единого значения по умолчанию для всех клиник нет |
| Аллергии всегда на виду | Список аллергий из `PatientRecord` отображается в «шапке» экрана консультации на каждом приёме, а не только в общей ЭМК |
| Консультация неизменяема после подписания | Правка подписанной `Consultation` создаёт новую версию (`версия + 1`); старая версия сохраняется целиком, не удаляется `[SYS-TRACE]` |
| Направление не закрывает визит в клинику, только текущий приём | Оформление `Referral` завершает текущий `Encounter`, но `visit_id` остаётся открытым; у целевого врача создаётся новый `Encounter` с тем же `visit_id` и тем же `talon_id` — номерок переадресуется, новый не выдаётся (счёт в `CASH` «дорастает» новой услугой того же визита) |
| Направление — только на конкретного врача | `Referral.целевой_врач_id` обязателен; варианта «только специализация, распределить по свободным врачам» нет |
| Аномально долгая консультация — предупреждение и обязательная причина | Если приём (`идёт_приём`) длится дольше `порог_аномальной_длительности_приёма_минут`, система показывает врачу предупреждение; подписать консультацию сверх порога можно только заполнив `причина_длительной_консультации` |
| Заказ на исследование — не через собственную таблицу | Назначение лабораторного/инструментального исследования всегда идёт через публичный метод `LAB`/`INSTR` — этот модуль не хранит заказы и результаты напрямую, только ссылку в консультации |
| Результат исследования подтягивается автоматически | Как только `LAB`/`INSTR` публикует событие о готовом результате, он становится виден в ЭМК и в консультации, без ручного «импорта» врачом |
| Доступ к ЭМК — по функциональной необходимости | Принцип минимальных привилегий `[SYS-RBAC]`, `[SYS-PDATA]`; любое открытие/изменение ЭМК — под аудит-трейл `[SYS-AUDIT]` |
