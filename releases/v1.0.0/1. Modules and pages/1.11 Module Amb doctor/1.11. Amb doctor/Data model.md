## Модель данных

### Владеет `EMR`

#### 1.1. ЭМК (PatientRecord)

> Единая карта на пациента, создаётся один раз. Пока пациент ни разу не пришёл в клинику, его ЭМК не показывается врачу содержательно — пациент виден в списке, но детали скрыты/размыты. Поиск — по всем параметрам, включая ПИНФЛ, а не только ФИО/номер ЭМК.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | номер ЭМК |
| пациент_id | FK Пациент (`REG`/`MGMT`) | да | |
| дата_создания_эмк | date | да | момент заведения карты — может произойти до фактической консультации (например, при первичной регистрации в базе) |
| дата_первого_обращения | date | нет | момент, когда пациент физически впервые пришёл в клинику (визит создан/начался); от этой даты зависит правило видимости ЭМК (см. [Business rules and validations.md](Business%20rules%20and%20validations.md)) — пациент мог прийти, но приём мог не состояться (неявка/отмена) |
| дата_первой_консультации | date | нет | момент подписания первой консультации; именно с этого момента начинается содержательная медицинская история пациента |
| аллергии | JSON, массив (наименование, реакция, тяжесть) | нет | выводится в «шапке» ЭМК на каждом экране врача — критично для безопасности |
| хронические_заболевания | JSON, массив (диагноз МКБ-10, дата установления) | нет | |
| группа_крови, резус | string | нет | |
| статус | enum: активна | да | ЭМК не «закрывается» и не удаляется, только версионируется |

> Разница между тремя датами: карта (`дата_создания_эмк`) может существовать раньше, чем пациент физически появится в клинике; пациент может прийти (`дата_первого_обращения`), но по любой причине не попасть на приём; медицинская история как таковая (жалобы, диагнозы, назначения) появляется только после `дата_первой_консультации`.

#### 1.2. Консультация (Consultation)

> Клиническая запись одного приёма. Неизменяема после подписания врачом — правки только новой версией или дополнением (`ConsultationAddendum`), история не затирается `[SYS-TRACE]`.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| эмк_id | FK PatientRecord | да | |
| приём_id | FK Encounter (`PHYS`) | да | какой приём породил эту запись |
| врач_id | FK Сотрудник (`HR`) | да | |
| дата_время | datetime | да | |
| причина_обращения | string/enum | нет | краткая структурированная категория обращения, отдельно от свободного текста жалоб — удобно для отчётности |
| жалобы | text | нет | |
| анамнез_заболевания, анамнез_жизни | text | нет | |
| объективный_осмотр | text/структурированные показатели (АД, пульс, температура и т.д.) | нет | |
| диагноз_предварительный | код МКБ-10 + текст | да, если `статус_диагноза = предварительный` | врач может подписать консультацию с предварительным диагнозом, если окончательный ещё не установлен (например, ждут результат анализа) |
| диагноз_заключительный | код МКБ-10 + текст | да, если `статус_диагноза = заключительный` | |
| диагноз_сопутствующий | array<код МКБ-10 + текст> | нет | |
| статус_диагноза | enum: предварительный / уточнённый / заключительный | да | «уточнённый» проставляется, когда диагноз скорректирован через `ConsultationAddendum`, не меняя исходную запись |
| план_лечения_рекомендации | text | нет | свободный текст-резюме плана; структурированные назначения — отдельными сущностями ниже |
| статус | enum: черновик / подписана | да | см. [Statuses and transitions.md](Statuses%20and%20transitions.md) |
| версия | integer | да | при правке подписанной записи создаётся новая версия, старая сохраняется |

> МКБ-10 — часть медицинского документа, заполняется врачом при консультации; справочник кодов — внешний международный стандарт, не бизнес-специфичный справочник клиники.

#### 1.3. Назначение лекарственного препарата (MedicationOrder)

> Часть консультации/ЭМК — клиническое назначение (что врач прописал), а не складские/аптечные данные. `PHYS`/`EMR` не становится владельцем складских остатков препаратов (это зона `WMS`, если в клинике есть аптечный склад).

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| консультация_id | FK Consultation | да | |
| препарат | string | да | наименование (в перспективе — ссылка на фармакологический справочник) |
| дозировка | string | да | |
| кратность | string | да | например, «2 раза в день» |
| длительность | string | да | например, «7 дней» |
| путь_введения | enum: перорально / внутривенно / внутримышечно / местно / прочее | да | |
| комментарий_врача | text | нет | |
| статус | enum: активно / отменено | да | отмена — с указанием причины, не физическое удаление |

#### 1.6. Дополнение к консультации (ConsultationAddendum)

> Решает задачу «результат анализа пришёл после подписания приёма»: подписанная консультация не редактируется, но врач может добавить дополнение отдельной записью, которая становится частью истории ЭМК.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| consultation_id | FK Consultation | да | к какой подписанной консультации относится |
| врач_id | FK Сотрудник (`HR`) | да | |
| дата_время | datetime | да | |
| текст_дополнения | text | да | |
| причина_дополнения | text | да | например, «получен результат анализа, диагноз уточнён» |
| linked_result_id | FK (результат `LAB`/`INSTR`) | нет | если дополнение вызвано конкретным результатом исследования |
| диагноз_уточнённый | код МКБ-10 + текст | нет | опционально уточняет диагноз; при заполнении `Consultation.статус_диагноза` помечается как `уточнённый`, но сама запись `Consultation` не переписывается |
| подпись | boolean/метка подписи врача | да | дополнение подписывается так же, как и обычная консультация |

> Правило: дополнение не изменяет исходную подписанную `Consultation` — это независимая, тоже неизменяемая после подписания запись, связанная с исходной консультацией и видимая в ленте истории ЭМК как её продолжение.

### Владеет `PHYS`

#### 1.8. Приём (Encounter)

> Рабочая обёртка вокруг клинической консультации: кто, когда, в каком статусе. Создаётся, когда талон электронной очереди вызван к этому врачу, при направлении от другого врача, либо напрямую — при экстренном приёме по триажу.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| visit_id | FK Визит (`REG`) | да | визит в клинику — один на весь путь пациента за день, общий для всех `Encounter` этого визита |
| talon_id | FK QueueTicket (`REG`, компонент «Электронная очередь») | нет | тот же номерок, что и у визита в клинику; при направлении внутри визита переиспользуется, не создаётся заново |
| врач_id | FK Сотрудник (`HR`) | да | |
| кабинет_id | FK `CLINIC` | да | |
| пациент_id | FK Пациент | да | |
| статус | enum: `ожидает_вызова` / `вызван` / `идёт_приём` / `приостановлен` / `ожидает_оплаты` / `завершён` / `неявка` / `отменён` | да | см. [Statuses and transitions.md](Statuses%20and%20transitions.md) |
| триаж_уровень | enum: плановый / экстренный | да | `экстренный` — приём вне очереди, минуя обычный порядок вызова |
| инициатор_экстренного_приёма_id | FK Сотрудник (`HR`) | обязательно для `триаж_уровень = экстренный` | сотрудник регистратуры, зафиксировавший обращение как потенциально экстренное |
| подтвердил_триаж_id | FK Сотрудник (`HR`) | обязательно для `триаж_уровень = экстренный` | дежурный врач / старшая медсестра / иной уполномоченный медицинский сотрудник, подтвердивший медицинскую экстренность — не регистратор |
| причина_экстренности | text | обязательно для `триаж_уровень = экстренный` | |
| время_инициации | datetime | обязательно для `триаж_уровень = экстренный` | когда регистратура зафиксировала обращение |
| время_подтверждения | datetime | нет | когда уполномоченный медицинский сотрудник подтвердил триаж; до подтверждения приём считается предварительно открытым |
| статус_оплаты | enum: не_требуется / ожидает_оплаты / оплачено / разрешено_без_оплаты | да | заполняется, если приём порождён направлением/заказом, требующим оплаты |
| дата_время_начала / дата_время_окончания | datetime | нет / нет | |
| причина_длительной_консультации | text | обязательно, если порог включён клиникой и длительность превысила `порог_аномальной_длительности_приёма_минут` | порог — опциональная настройка, каждая клиника задаёт (или отключает) по-своему |
| consultation_id | FK Consultation (`EMR`) | нет | заполняется, когда врач создаёт клиническую запись |

#### 1.9. Направление (Referral)

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| консультация_id | FK Consultation | да | из какой консультации создано |
| направляющий_врач_id | FK Сотрудник (`HR`) | да | |
| целевой_врач_id | FK Сотрудник (`HR`) | да | направление всегда на конкретного врача, без варианта «только специализация» |
| тип_направления | enum: `внутри_текущего_визита` / `плановое_на_другой_день` / `с_возвратом_к_текущему_врачу` / `без_создания_очереди` | да | см. [Business rules and validations.md](Business%20rules%20and%20validations.md), таблица типов направления |
| причина | text | да | |
| требует_оплаты | boolean | да | определяется услугой целевого приёма; при `true` целевой `Encounter` создаётся в статусе `ожидает_оплаты` |
| статус_оплаты | enum: не_требуется / ожидает_оплаты / оплачено / разрешено_без_оплаты | да | зеркалирует статус оплаты целевого приёма |
| запланированная_дата | date | обязательно для `плановое_на_другой_день` | |
| исходный_приём_id | FK Encounter | нет | приём, который приостановлен и должен возобновиться после направления (для `с_возвратом_к_текущему_врачу`) |
| статус | enum: создано / в_очереди / выполнено / отменено | да | см. [Statuses and transitions.md](Statuses%20and%20transitions.md) |
| новый_приём_id | FK Encounter | нет | заполняется для типов, создающих приём у целевого врача (`внутри_текущего_визита`, `с_возвратом_к_текущему_врачу`); не заполняется для `плановое_на_другой_день` и `без_создания_очереди` |

> Заказы на лабораторные и инструментальные исследования этот документ не владеет — они принадлежат будущим модулям `LAB`/`INSTR`. Врач инициирует их через публичный интерфейс этих модулей (см. [Public interface.md](../Public%20interface.md)), а в консультации хранится только ссылка на заказ и на готовый результат. Правило платности в равной мере применимо к таким заказам — конкретная реализация гейта на стороне `LAB`/`INSTR` фиксируется как открытый вопрос.
