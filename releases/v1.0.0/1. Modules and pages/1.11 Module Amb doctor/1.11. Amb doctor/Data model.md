## Модель данных

### Владеет `EMR`

#### 1.1. ЭМК (PatientRecord)

> Единая карта на пациента, создаётся один раз. Пока пациент ни разу не записался/не пришёл в клинику, его ЭМК не показывается врачу содержательно — пациент виден в списке, но детали скрыты/размыты. Поиск — по всем параметрам, включая ПИНФЛ, а не только ФИО/номер ЭМК.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | номер ЭМК |
| пациент_id | FK Пациент (`REG`/`MGMT`) | да | |
| дата_первого_визита | date | нет | заполняется по факту первого визита; до этого момента карта видна ограниченно (см. Business rules) |
| аллергии | JSON, массив (наименование, реакция, тяжесть) | нет | выводится в «шапке» ЭМК на каждом экране врача — критично для безопасности |
| хронические_заболевания | JSON, массив (диагноз МКБ-10, дата установления) | нет | |
| группа_крови, резус | string | нет | |
| статус | enum: активна | да | ЭМК не «закрывается» и не удаляется, только версионируется |

#### 1.2. Консультация (Consultation)

> Клиническая запись одного приёма. Неизменяема после подписания врачом — правки только новой версией/дополнением, история не затирается `[SYS-TRACE]`.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| эмк_id | FK PatientRecord | да | |
| приём_id | FK Encounter (`PHYS`) | да | какой приём породил эту запись |
| врач_id | FK Сотрудник (`HR`) | да | |
| дата_время | datetime | да | |
| жалобы | text | нет | |
| анамнез_заболевания, анамнез_жизни | text | нет | |
| объективный_осмотр | text/структурированные показатели (АД, пульс, температура и т.д.) | нет | |
| диагноз_основной | код МКБ-10 + текст | да, для подписанной консультации | МКБ-10 — часть медицинского документа, заполняется врачом при консультации; справочник кодов — внешний международный стандарт, не бизнес-специфичный справочник клиники |
| диагноз_сопутствующий | array<код МКБ-10 + текст> | нет | |
| план_лечения_рекомендации | text | нет | |
| статус | enum: черновик / подписана | да | см. [Statuses and transitions.md](Statuses%20and%20transitions.md) |
| версия | integer | да | при правке подписанной записи создаётся новая версия, старая сохраняется |

#### 1.3. Документ ЭМК (ClinicalDocument)

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| эмк_id | FK PatientRecord | да | |
| консультация_id | FK Consultation | нет | документ может быть привязан к конкретной консультации (заключение) или существовать отдельно (справка) |
| тип | string | да | шаблон документа — ссылка на модуль шаблонов документов `DOCT` |
| файл_id | FK File (`FILE`) | нет | сформированный печатный документ |
| дата_создания, создал_id | datetime, FK Сотрудник (`HR`) | да | |

### Владеет `PHYS`

#### 1.4. Приём (Encounter)

> Рабочая обёртка вокруг клинической консультации: кто, когда, в каком статусе. Создаётся, когда талон электронной очереди вызван к этому врачу, либо напрямую — при экстренном приёме по триажу.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| visit_id | FK Визит (`REG`) | да | визит в клинику — один на весь путь пациента за день, общий для всех `Encounter` этого визита |
| talon_id | FK QueueTicket (`REG`, «Электронная очередь») | нет | тот же номерок, что и у визита в клинику; при направлении к другому врачу переиспользуется, не создаётся заново |
| врач_id | FK Сотрудник (`HR`) | да | |
| кабинет_id | FK `CLINIC` | да | |
| пациент_id | FK Пациент | да | |
| статус | enum: ожидает / идёт_приём / завершён / отменён / неявка | да | см. [Statuses and transitions.md](Statuses%20and%20transitions.md) |
| триаж_уровень | enum: плановый / экстренный | да | `экстренный` — приём вне очереди, минуя обычный порядок вызова; инициируется регистратурой в амбулаторном сегменте |
| дата_время_начала / дата_время_окончания | datetime | нет / нет | |
| причина_длительной_консультации | text | обязательно, если порог включён клиникой и длительность превысила `порог_аномальной_длительности_приёма_минут` | порог — опциональная настройка, каждая клиника задаёт (или отключает) по-своему |
| consultation_id | FK Consultation (`EMR`) | нет | заполняется, когда врач создаёт клиническую запись |

#### 1.5. Направление (Referral)

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| консультация_id | FK Consultation | да | из какой консультации создано |
| направляющий_врач_id | FK Сотрудник (`HR`) | да | |
| целевой_врач_id | FK Сотрудник (`HR`) | да | направление всегда на конкретного врача, без варианта «только специализация» |
| причина | text | да | |
| статус | enum: создано / в_очереди / выполнено / отменено | да | см. [Statuses and transitions.md](Statuses%20and%20transitions.md) |
| новый_приём_id | FK Encounter | нет | новый `Encounter` того же `visit_id` и того же `talon_id` — заполняется при создании приёма у целевого врача |

> Заказы на лабораторные и инструментальные исследования этот модуль не владеет — они принадлежат будущим модулям `LAB`/`INSTR`. Врач инициирует их через публичный интерфейс этих модулей (см. [Public interface.md](../Public%20interface.md)), а в консультации хранится только ссылка на заказ и на готовый результат.
