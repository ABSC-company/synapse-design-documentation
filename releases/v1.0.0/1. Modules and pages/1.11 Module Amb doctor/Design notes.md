## Что важно для макетов

1. Аллергии — самый критичный по безопасности элемент интерфейса всего модуля: должны быть видны без действий пользователя на каждом экране, где есть пациент. Не прятать за вкладку или сворачиваемую секцию.
2. Печатный медицинский документ (заключение, справка) — этот модуль не проектирует свою собственную форму заполнения документа: кнопка «Сформировать документ» вызывает `DOCT.ПолучитьШаблон` и открывает документ как уже готовый шаблон с полями для заполнения — сам экран заполнения/редактирования документа спроектирован в модуле документооборота (см. `1.5. Document Management Module`, в частности `1.5.3. Document Filling Page` и `1.5.4. Documents Editor Page`). В макетах «Амбулаторного врача» это действие — переход/встраивание готового компонента DOCT, а не отдельно нарисованная форма.
3. Направление к врачу и назначение исследования — оба воспринимаются врачом как «побочные», не основные действия консультации: не должны визуально конкурировать с главным CTA «Подписать и завершить приём» по весу и расположению, но должны быть легко доступны в один клик.
4. Статусы приёма (восемь значений: ожидает вызова / вызван / идёт приём / приостановлен / ожидает оплаты / завершён / неявка / отменён) и направления (создано / в очереди / выполнено / отменено) — переиспользовать цветовой язык статусов, уже принятый в модуле амбулаторной регистратуры (Visits Page, Electronic Queue), не изобретать новый для врача.
5. Экстренный триаж — двухшаговый процесс (регистратура фиксирует → медицинский сотрудник подтверждает) должен визуально различать «зафиксировано, ждёт подтверждения» и «подтверждено, экстренный приоритет» — это разные состояния на разных экранах (Emergency Encounter Modal у регистратуры, Triage Confirmation Page у медицинского сотрудника, индикатор на Рабочем столе врача), одинаковым способом маркированные во всех трёх местах.
6. Логика оплаты (`ожидает_оплаты` / `оплачено` / `разрешено_без_оплаты`) — сквозной статус, влияющий сразу на несколько экранов (Рабочий стол, Консультация, Направление, Назначение исследования); индикатор платности и статуса оплаты должен быть визуально одинаковым во всех них, не изобретаться заново для каждого экрана.
7. Боковая панель сокращённой ЭМК внутри консультации и полноценная страница «ЭМК пациента» — один и тот же набор данных в двух представлениях; вёрстка компонентов (шапка, лента) должна проектироваться как переиспользуемая, а не как два независимых макета.
8. Поиск по МКБ-10 в консультации — это внешний международный справочник, не редактируемый в этом модуле; в макетах не закладывать возможность добавления/редактирования кодов, только поиск и выбор. Предварительный/заключительный статус диагноза — переключатель, а не два одновременно обязательных поля.
9. Структурированные назначения (препараты/процедуры/рекомендации) — три отдельные секции с построчным добавлением, визуально отделённые от свободнотекстового «Плана лечения»; не сворачивать в одно универсальное поле.
10. Дополнение к консультации — не альтернатива редактированию, а отдельная сущность истории: в ленте ЭМК отображается вложенно под исходной консультацией, не как независимая запись того же уровня.
11. Экстренный доступ к чужой ЭМК — кнопка появляется только тогда, когда у врача действительно нет обычного права на пациента; форма всегда требует причину и явно предупреждает об аудите, чтобы не превращаться в обходной путь для рутинного просмотра чужих карт.
12. Настройки модуля (порог длительности приёма, число попыток вызова) — часть общей панели администратора, стилистически и по расположению согласованы с остальными вкладками настроек (`1.9 Module admin`), а не отдельным изолированным экраном.
