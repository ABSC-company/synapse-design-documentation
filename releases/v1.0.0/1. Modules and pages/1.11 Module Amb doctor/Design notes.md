## Что важно для макетов

1. Аллергии — самый критичный по безопасности элемент интерфейса всего модуля: должны быть видны без действий пользователя на каждом экране, где есть пациент (Рабочий стол — при наведении/раскрытии строки, Консультация — в шапке, ЭМК — в шапке). Не прятать за вкладку или сворачиваемую секцию.
2. Направление к врачу и назначение исследования — оба воспринимаются врачом как «побочные», не основные действия консультации: они не должны визуально конкурировать с главным CTA «Подписать и завершить приём» по весу и расположению, но должны быть легко доступны в один клик.
3. Статусы приёма (`ожидает` / `идёт_приём` / `завершён` / `отменён` / `неявка`) и направления (`создано` / `в_очереди` / `выполнено` / `отменено`) — переиспользовать цветовой язык статусов, уже принятый в модуле амбулаторной регистратуры (Visits Page, Electronic Queue), не изобретать новый для врача.
4. Экстренный триаж — визуальный маркер должен читаться на всех экранах, где может появиться такой приём (Рабочий стол врача, Консультация), одинаковым способом — это сквозной статус, а не локальная особенность одного экрана.
5. Боковая панель сокращённой ЭМК внутри консультации и полноценная страница «ЭМК пациента» — один и тот же набор данных в двух представлениях; вёрстка компонентов (шапка, лента) должна проектироваться как переиспользуемая, а не как два независимых макета.
6. Поиск по МКБ-10 в консультации — это внешний международный справочник, не редактируемый в этом модуле; в макетах не закладывать возможность добавления/редактирования кодов, только поиск и выбор.
7. Настройки модуля (порог длительности приёма) — часть общей панели администратора, стилистически и по расположению должны быть согласованы с остальными вкладками настроек (`1.9 Module admin`), а не отдельным изолированным экраном.
