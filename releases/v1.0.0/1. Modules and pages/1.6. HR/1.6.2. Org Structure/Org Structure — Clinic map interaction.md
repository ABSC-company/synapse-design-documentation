# 1.6.2. Org Structure — Взаимодействие с орг-структурой (карта клиники)

## Спецификация

- Документ: описание взаимодействия фронтенда с подмодулем структуры (Structure Submodule) в части карты клиники: здания, этажи, кабинеты, рабочие места и привязка к орг-структуре.
- Основание: [back-end/StructureModule.md](./back-end/StructureModule.md), разделы 5.7–5.11 (доменная модель), 9.9–9.10 (API), ТЗ фронта ([TS. Org structure.md](./TS.%20Org%20structure.md)).
- Связанные документы: [Org Structure.md](./Org%20Structure.md) (дерево отделов), `1.6.5. Staffing Schedule` (ставки), `1.3.4. Clinic map Page` (просмотр карты в регистратуре).
- Основные роли: Администратор системы, HR-менеджер.
- Статус документа: Draft

## Модель данных (со стороны бекенда)

Иерархия сущностей карты клиники:

```
Branch (филиал)
└── ClinicMap (одна SVG-карта филиала)
    ├── Building (здание — логическая группировка этажей, своей графики нет)
    │   └── Floor (этаж — слой path на SVG филиала + собственная SVG этажа)
    │       └── Room (кабинет — слой path на SVG этажа, своей графики нет)
    │           └── Workplace (рабочее место, 1:1 с кабинетом; department_id, capacity)
    │               └── Сотрудники (привязка через HR Module)
    └── Floor (этажи до объединения в здание: building_id = NULL)
```

Ключевые правила бекенда:

1. SVG-файлы хранятся в File Module; Structure Module хранит только `file_id`.
2. Анализ SVG выполняет сервер: все `path` **зарезервированного цвета** (системная константа, согласуется с фронтом) трактуются как этажи (на карте филиала) или кабинеты (на SVG этажа).
3. Идентификаторы созданных записей сервер прописывает прямо в `path` возвращаемой SVG (`floor_id` — на карте филиала, `room_id` — на SVG этажа); отдельное сопоставление `path → id` фронту не передаётся — фронт берёт идентификаторы из `path`.
4. Анализ и создание записей атомарны: при ошибке ни карта, ни этажи/кабинеты не создаются частично.
5. При создании кабинета автоматически создаётся его рабочее место: `department_id = NULL`, `capacity = 1`. Кабинет и рабочее место соотносятся **1:1** — кабинет и есть рабочее место; дополнительные рабочие места в кабинете не создаются.
6. Назначение сотрудника выполняется на рабочее место кабинета; число сотрудников ограничено `capacity` (на сколько сотрудников рассчитан кабинет).

## Flow 1. Загрузка карты филиала и получение этажей

1. Пользователь открывает страницу карты клиники. Пока карта не создана, страница пустая: по центру — empty state блок с кнопкой «Создать карту филиала».
2. Клик по кнопке открывает модальное окно, через которое пользователь загружает общую SVG филиала.
3. Фронт отправляет `POST /api/v1/structure/clinic-maps` (файл + `branch_id`; название карты можно не указывать — подставится название филиала).
4. Сервер сохраняет файл через File Module, анализирует графику, по `path` зарезервированного цвета создаёт записи в `floors` (без привязки к зданию) и прописывает `floor_id` в соответствующие `path` итоговой SVG.
5. В ответе фронт получает SVG **без отдельного сопоставления** `path.id → floor_id`: `floor_id` уже содержится в `path`, и фронт использует его во всех дальнейших операциях с этажами (выделение, объединение в здания, загрузка SVG этажа).
6. Фронт отрисовывает SVG; каждый `path` с `floor_id` становится интерактивным (hover-подсветка, выделение кликом).

Ограничения:

- У филиала может быть только одна карта (уникальность `branch_id`). Повторная загрузка — через удаление/замену карты (`DELETE`/`PATCH /clinic-maps/{id}`).
- Если сервер не нашёл ни одного `path` зарезервированного цвета — фронт показывает ошибку валидации SVG с подсказкой о требованиях к файлу.

## Flow 2. Объединение этажей в здания (фронт)

Слои, выявленные на карте филиала, — это **этажи**. Зданий на SVG нет: здание — логическая группировка, которую задаёт пользователь.

1. Пользователь переходит в режим «Объединение в здания».
2. Мультивыбор слоёв: клик/ctrl-клик (или рамка выделения) по `path` этажей на SVG — выбранные слои подсвечиваются.
3. Пользователь нажимает «Объединить в здание» и вводит название здания.
4. Фронт отправляет `POST /api/v1/structure/buildings` с `clinic_map_id`, `floor_ids[]` и названием.
5. Сервер создаёт здание и проставляет `building_id` выбранным этажам; фронт визуально группирует слои (общий цвет/контур группы, подпись здания).
6. Пользователь повторяет шаги 2–5 для остальных зданий.

Правила и edge cases:

- Этаж может принадлежать только одному зданию; уже сгруппированные слои недоступны для выбора в новое здание (визуально погашены).
- Названия этажей внутри здания задаются через `PATCH /floors/{id}` (например «1 этаж», «2 этаж») — сразу в мастере или позже.
- Разгруппировка/правка состава здания — `PATCH /buildings/{id}`; удаление здания (`DELETE`) возвращает этажи в состояние `building_id = NULL`, но не удаляет их.
- Незавершённое состояние допустимо: этажи без здания хранятся на сервере; мастер показывает счётчик «не распределено этажей: N».

## Flow 3. Загрузка SVG этажа и получение кабинетов

1. Пользователь выбирает этаж (на карте филиала или в списке этажей здания) и загружает отдельную SVG этажа.
2. Фронт отправляет `POST /api/v1/structure/floors/{id}/svg`.
3. Сервер сохраняет файл, анализирует графику, по `path` зарезервированного цвета создаёт кабинеты в `rooms` с привязкой к этажу и прописывает `room_id` в соответствующие `path` SVG этажа. Вместе с каждым кабинетом автоматически создаётся одно рабочее место (`department_id = NULL`, `capacity = 1`).
4. В ответе фронт получает SVG этажа без отдельного сопоставления: `room_id` берётся из `path` и используется во всех дальнейших операциях с кабинетом.
5. Фронт отрисовывает план этажа; каждый `path` с `room_id` — интерактивный слой.

## Flow 4. Настройка кабинета (фронт)

Пользователь кликает по кабинету на плане этажа — открывается панель настройки кабинета со следующими полями:

| Поле | Куда пишется | API |
| --- | --- | --- |
| Название кабинета | `rooms.name` | `PATCH /api/v1/structure/rooms/{id}` |
| Отдел (по орг-структуре) | `workplaces.department_id` рабочего места кабинета | `PATCH /api/v1/structure/workplaces/{id}` |
| Кол-во сотрудников (на сколько рассчитан кабинет) | `workplaces.capacity` | `PATCH /api/v1/structure/workplaces/{id}` |
| Есть склад (склад отдела) | признак кабинета-склада отдела | см. «Открытые вопросы» |

Поведение:

1. **Название** — свободный ввод, сохраняется по blur/кнопке «Сохранить».
2. **Отдел** — select с деревом отделов филиала (`GET /api/v1/structure/departments/{branch_id}`). При выборе отдела фронт проставляет `department_id` рабочему месту кабинета; название рабочего места при этом автоматически обновляется на название отдела (правило бекенда).
3. **Кол-во сотрудников** — числовое поле; пишется в `capacity` рабочего места кабинета и задаёт, на сколько сотрудников рассчитан кабинет.
4. Кабинет без настройки (без названия/отдела) валиден: сервер допускает `name = NULL`, `department_id = NULL`; мастер помечает такие кабинеты как «не настроен».

## Flow 5. Назначение сотрудника на кабинет (рабочее место)

Кабинет и есть рабочее место (1:1): пользователь открывает кабинет → видит его отдел, вместимость и назначенных сотрудников → назначает сотрудника.

1. Пользователь открывает кабинет на плане этажа.
2. Панель кабинета показывает: отдел, занято/вместимость (`capacity`), список назначенных сотрудников.
3. Пользователь нажимает «Назначить сотрудника» и выбирает сотрудника (поиск по сотрудникам HR Module; приоритетно — сотрудники отдела кабинета).
4. Привязка выполняется через интеграцию Structure ↔ HR Module:
   - если у рабочего места **нет отдела** (`department_id = NULL`), при привязке сотрудника рабочему месту автоматически назначается отдел этого сотрудника из его кадровой записи (правило бекенда);
   - если отдел у рабочего места есть, а сотрудник из другого отдела — фронт предупреждает о несоответствии (кадровый перевод выполняется на странице [Org Structure.md](./Org%20Structure.md) через приказ, не здесь).
5. Контроль вместимости: число сотрудников кабинета не может превышать `capacity`; при достижении лимита кнопка назначения недоступна с подсказкой.
6. Снятие сотрудника с рабочего места — действие в том же списке (204 No Content).

## Состояния и edge cases

- **Empty**: карта филиала не создана — по центру страницы empty state блок с кнопкой «Создать карту филиала»; клик открывает модальное окно загрузки SVG (Flow 1).
- **Loading**: анализ SVG на сервере может занимать время — прогресс-состояние «Анализируем карту…»; UI блокируется до ответа.
- **Error**: ошибка анализа SVG — записи не создаются (атомарность), фронт показывает ошибку и оставляет форму загрузки; `422` при нарушении бизнес-ограничений — тост с текстом ошибки.
- **Permissions**: операции требуют соответствующих прав Structure Module (`3401–3432` — карта/здания/этажи/кабинеты, `3501–3505` — рабочие места, `3900 STRUCTURE_MANAGE` — полный доступ). Без прав на изменение карта доступна read-only.
- **Удаление**: удаление этажа/кабинета с назначенными сотрудниками — предупреждение и требование предварительно снять привязки (валидация через HR Module).
- **Повторная загрузка SVG этажа**: поведение при перезаливке (пересоздание кабинетов vs. дозаливка) — см. «Открытые вопросы».

## Зависимости и ограничения

- **Structure Module** — источник данных карты, зданий, этажей, кабинетов, рабочих мест; серверный анализ SVG.
- **File Module** — хранение SVG-файлов; фронт получает файлы по `file_id`.
- **HR Module** — кадровые записи сотрудников; привязка сотрудника к рабочему месту и определение его отдела.
- Цвет `path`, маркирующий этажи и кабинеты, — системная константа, согласованная с бекендом; требования к подготовке SVG должны попасть в инструкцию для пользователя (подсказка в мастере загрузки).
- Все запросы фильтруются по `branch_id` (правило бекенда).

## Открытые вопросы (расхождения с документацией бекенда)

1. **«Есть ли склад (склад отдела)»** — в таблице `rooms` (раздел 5.10 бекенд-дока) нет признака склада. Требуется расширение бекенда: либо поле `rooms.is_warehouse` (+ связь с модулем Склад 1.7), либо склад отдела заводится в Warehouse Module с ссылкой на `room_id`. До решения поле в UI проектируется как чекбокс «Склад отдела», контракт — TBD.
2. **Кабинет = рабочее место (1:1)** — принятое продуктовое решение: у кабинета ровно одно рабочее место, отдел и вместимость кабинета хранятся на нём (`workplaces.department_id`, `workplaces.capacity`); отдельное поле `rooms.department_id` не требуется. Бекенд при этом допускает создание дополнительных рабочих мест (`POST /workplaces`, раздел 9.10) — нужно ограничение 1:1 на бекенде либо договорённость, что фронт этот эндпоинт не использует.
3. **Перезаливка SVG этажа** — бекенд-док не описывает поведение `POST /floors/{id}/svg` для этажа, у которого уже есть кабинеты (пересоздание? сопоставление?). Нужно согласовать: рекомендация — сопоставлять существующие кабинеты по `room_id` в `path` (если перезаливаемая SVG сохранила проставленные сервером идентификаторы) с сохранением настроек, новые — создавать, отсутствующие — помечать к удалению с подтверждением.
4. **Эндпоинт привязки сотрудника к рабочему месту** — в разделе 9.10 бекенд-дока нет явного метода assign/unassign (упомянут только в интеграции с HR, раздел 11.3). Нужно зафиксировать контракт (например, `POST /api/v1/structure/workplaces/{id}/employees` / `DELETE .../employees/{employee_id}`) и схему `WorkplaceEmployee` из раздела 14.
