# 1.6.10 HR Document Management (Кадровое делопроизводство)

## Спецификация страницы интерфейса

* **Название страницы:** Кадровое делопроизводство
* **Модуль:** HR (Human Resources) — представление поверх EDM Module
* **Тип страницы:** Реестр документов (ЭДО), отфильтрованный по кадровой принадлежности
* **Основные роли:** Администратор / HR-менеджер, Руководитель
* **Права доступа:** `HR_EMPLOYEE_GET` + права EDM на просмотр/подписание документов категории `hr`
* **Маршрут:** `/hr/documents`
* **Платформа:** веб-приложение медицинской системы
* **Статус документа:** Draft

> Базовая механика реестра/просмотра/подписания документов описана в модуле **1.5. Document Management Module** (страницы 1.5.1–1.5.4). Здесь — кадровая «обёртка»: HR Module является bucket-owner документов категории `hr`. Эта страница **не дублирует** редактор и реестр ЭДО, а задаёт кадровый контекст и точки интеграции.

---

## Назначение страницы и сценарии использования

Централизованный доступ к кадровым приказам и документам клиники (приём, перевод, отпуск, командировка, увольнение и пр.), реализованным в ЭДО. Страница — это реестр документов 1.5, преднастроенный на кадровую папку/категорию, с группировкой по типам приказов и привязкой к сотрудникам.

С помощью страницы пользователь может:

1. Просматривать кадровые документы, сгруппированные по типам приказов (folders).
2. Фильтровать по типу приказа, сотруднику, статусу подписания, периоду.
3. Открывать документ для просмотра/подписания (→ 1.5.2).
4. Создавать кадровый приказ (→ редактор/шаблон 1.5.4 / 1.5.3).
5. Переходить из приказа в эл. дело связанного сотрудника.

---

## User Flows / Task Flows

### Flow 1. Навигация по кадровым приказам
1. Пользователь открывает «Кадровое делопроизводство».
2. Система показывает кадровую папку ЭДО с подпапками по типам приказов.
3. Пользователь открывает подпапку (например, «Приёмы»).

### Flow 2. Открытие/подписание документа
1. Клик по документу → карточка/страница документа (механика 1.5.2): метаданные, подписанты, история.
2. Подписание/отказ — по правилам ЭДО.

### Flow 3. Создание приказа
1. «+ Создать приказ» → выбор типа (приём/перевод/отпуск/командировка/увольнение) → редактор/шаблон ЭДО (1.5.4 / 1.5.3) с предзаполненной кадровой категорией.

### Flow 4. Переход к сотруднику
1. В карточке приказа — ссылка «Сотрудник» → эл. дело (1.6.4).

## Структура: Wireframes

### Общий layout (реестр ЭДО, кадровый контекст)
```
┌─────────────────────────────────────────────────────────────────┐
│  Кадровое делопроизводство                      [+ Создать приказ]│
├─────────────────────────────────────────────────────────────────┤
│  Папки приказов:                                                 │
│  [📁 Приёмы] [📁 Переводы] [📁 Отпуска] [📁 Командировки] [📁 Увольнения]│
│  🔍 Поиск...                                        [Фильтр ▾]   │
├──────────────────┬───────────┬────────────┬────────────┬────────┤
│ Документ         │ Сотрудник │ Тип        │ Дата       │ Статус │
│ Приказ о приёме… │ Иванов И. │ Приём      │ 12.01.2026 │●Подписан│
│ Приказ об отпуске│ Петрова А.│ Отпуск     │ 01.06.2026 │●Ожидает │
└──────────────────┴───────────┴────────────┴────────────┴────────┘
```

### Типы приказов (folders из HR/EDM)
| Folder | Кадровый тип |
|--------|--------------|
| `hire` | Приём |
| `transfer` | Перевод / назначение / снятие должности |
| `leave` | Отпуск |
| `business_trip` | Командировка |
| `dismissal` | Увольнение |

### Фильтры (Drawer)
* Тип приказа (folder), Сотрудник (поиск по ФИО), Статус подписания (Draft/Pending/Processed/Signed/Delivered/Rejected/Aborted — см. 1.5), Период, Автор.

### Карточка/строка документа
* Те же поля и действия, что в 1.5 (название, автор, даты, статус, прогресс подписания), плюс кадровая привязка: связанный сотрудник и тип приказа.
* Клик — переход к документу (1.5.2); ссылка на сотрудника — в эл. дело (1.6.4).

### Создание приказа
* «+ Создать приказ» → выбор типа → редактор (1.5.4) или шаблон категории `hr` (1.5.3). Контекст (сотрудник, тип) передаётся в редактор, если приказ инициирован из кадрового действия (приём/перевод/увольнение).

## Визуал: High-Fidelity mockups

1. Папки-«вкладки» по типам приказов сверху реестра; счётчики ожидающих подписания.
2. Таблица документов в стиле реестра 1.5 + колонки «Сотрудник» и «Тип приказа».
3. Статусы подписания — бейджи из дизайн-системы ЭДО (1.5).
4. CTA «+ Создать приказ» с выбором типа. Единая дизайн-система (консистентно с 1.5).

## Состояния и edge cases

* **Empty:** «Кадровых документов нет» / пустая папка приказов конкретного типа.
* **Loading:** скелетон списка (как в 1.5).
* **Error:** блок ошибки + «Повторить»; toast при сбое действий.
* **Offline:** реестр из кэша; создание/подписание заблокировано, toast.
* **Permissions:** документы и действия видны по правам EDM; без `HR_EMPLOYEE_GET`/прав ЭДО на категорию `hr` — раздел/документы скрыты.
* **Edge cases:** приказ без привязки к сотруднику (общий кадровый документ) — колонка «Сотрудник» пуста; кадровое действие, для которого приказ ещё не создан — отражается как «требуется приказ» в эл. деле (1.6.4), но не блокирует действие.

## Interaction & Animation

* Переключение папок приказов — fade-смена списка.
* Поведение реестра, карточки и подписания — согласно 1.5 (модалки, drawer фильтров, статус-бейджи).
* Toast — выезд, авто-скрытие 3–5 с.

---

## Зависимости и ограничения

* **EDM Module (1.5)** — базовая механика реестра/документа/редактора/шаблонов; HR Module — bucket-owner категории `hr`.
* **API:** реестр документов ЭДО с фильтром по кадровой папке/категории; связь документа с `employee_id` (`hr_documents`, folders hire/transfer/leave/business_trip/dismissal).
* Интеграция с кадровыми действиями: формы приёма/перевода/отпуска/командировки/увольнения (1.6.3/1.6.4/1.6.2/1.6.9) открывают создание приказа в ЭДО.
* **Роли:** `HR_EMPLOYEE_GET` + права ЭДО на просмотр/подписание категории `hr`.
* **Ограничения:** редактор и детальная механика документов — в 1.5, здесь не дублируются; привязка приказа к кадровому действию не обязательна для завершения действия.

---

## Acceptance criteria

* Раздел показывает кадровые документы ЭДО, сгруппированные по типам приказов (приём, перевод, отпуск, командировка, увольнение).
* Фильтрация по типу приказа, сотруднику, статусу подписания и периоду работает.
* Открытие документа использует механику просмотра/подписания из 1.5.2; создание — редактор/шаблон 1.5.4/1.5.3 с категорией `hr`.
* Из карточки приказа доступен переход в эл. дело связанного сотрудника.
* Раздел не дублирует редактор ЭДО, а переиспользует его.
* Видимость документов и действий определяется правами EDM.
* Состояния empty / loading / error / offline / permissions обрабатываются корректно.
