# 1.1.1. Base Authorization Page

## Спецификация страницы интерфейса

- Название страницы: Базовая авторизация
- Модуль: Аутентификация
- Тип страницы: Вход в систему по логину и паролю
- Основные роли: любой пользователь, имеющий аккаунт в системе
- Статус документа: Draft
- Платформа: веб-приложение для медицинской регистратуры

## Назначение страницы и сценарии использования

Страница служит для безопасного входа пользователя в систему с помощью логина и пароля.
Основная задача — предоставить понятный, доступный и визуально спокойный интерфейс для ввода учетных данных.

Сценарии использования:

1. Пользователь открывает страницу приложения и хочет войти в учетную запись.
2. Пользователь вводит логин и пароль, затем нажимает кнопку "Войти".
3. Система проверяет учетные данные и в случае валидности направляет пользователя на главную страницу.
4. При невалидных данных пользователь получает сообщение об ошибке и может повторить ввод.
5. Страница должна поддерживать переходы между разными состояниями: загрузка, ошибка, offline.

## User Flows / Task Flows для страницы

### Flow 1. Вход по логину и паролю

1. Пользователь открывает страницу авторизации.
2. Страница отображает центральный контейнер, визуально делящийся на левую и правую части.
3. В левой части показывается изображение специалиста и его цитата.
4. В правой части отображается форма входа с полями:
   - логин
   - пароль
   - кнопка "Войти"
5. Пользователь вводит логин.
6. Пользователь вводит пароль.
7. Пользователь нажимает кнопку "Войти".
8. Система отправляет данные на аутентификационный API.
9. Если данные верны, пользователь перенаправляется в приложение.
10. Если данные неверны, отображается сообщение об ошибке и фокус возвращается на форму.

### Flow 2. Обновление страницы и смена цитаты

1. Пользователь обновляет страницу браузера.
2. Система загружает новую пару: изображение деятеля и цитату.
3. Левая часть обновляется автоматически.
4. Правая часть с формой входа остаётся доступной для ввода.

### Flow 3. Ошибка валидации или неверные данные

1. Пользователь вводит логин и/или пароль.
2. Нажимает кнопку "Войти".
3. Система получает ошибку от API.
4. На странице появляется сообщение с указанием причины (например, "Неверный логин или пароль").
5. Поля остаются заполненными, пользователь может скорректировать данные.
6. Далее пользователь повторно отправляет форму.

### Flow 4. Offline / отсутствие сети

1. Пользователь открывает страницу без сети.
2. Система обнаруживает offline-состояние.
3. Отображается уведомление о недоступности сети.
4. Кнопка "Войти" становится неактивной.
5. Пользователь может проверить подключение и обновить страницу.

## Структура: Wireframes

Страница состоит из одного центрального контейнера.

Лейаут:

- Центральный контейнер центрируется по вертикали и горизонтали.
- Контейнер визуально разделён на две части:
  - левая часть: иллюстрация и цитата.
  - правая часть: форма входа.
- Между частями может быть тонкая разделительная линия или градиентный переход.
- Контейнер должен иметь лёгкую тень и скруглённые углы.

Левая часть:

- Изображение круглой формы человека, внёсшего вклад в развитие медицины.
- Цитата под изображением или рядом с ним.
- Текст цитаты должен быть читаемым и лаконичным.
- При каждом обновлении страницы отображается новая пара изображение + цитата.

Правая часть:

- Заголовок: "Вход в систему".
- Инпут введения логина.
- Инпут введения пароля.
- Кнопка "Войти".

## Визуал: High-Fidelity mockups

Визуальные правила:

1. Цветовая схема должна быть спокойной, с акцентом на доверие и профессионализм.
2. Центральный контейнер расположен на нейтральном фоне.
3. Левая часть имеет визуальный акцент на лице деятеля и цитате.
4. Правая часть оформлена в виде карточки с четкими полями ввода.
5. Поля ввода имеют понятные подписи и плейсхолдеры.
6. Кнопка "Войти" выделяется цветом, но не должна быть слишком агрессивной.
7. Ошибки отображаются красным текстом под соответствующими полями или в общем баннере.
8. Все элементы выровнены по сетке, содержащей одинаковые отступы.
9. Шрифты — легко читаемые, без засечек или с минимальными засечками.
10. Иконка или иллюстрация в левой части — круглая, гармоничная, без резких элементов.

Обязательные визуальные компоненты:

- Центральный контейнер с двумя панелями.
- Иллюстрация деятеля медицины.
- Цитата под иллюстрацией.
- Заголовок формы "Вход в систему".
- Поле логина.
- Поле пароля.
- Кнопка "Войти".
- Сообщение об ошибке.
- Состояние loading для кнопки или обработки запроса.
- Состояние offline с уведомлением.

## Состояния и edge cases

### error

- Неверный логин или пароль.
- Сервер вернул ошибку 401 / 403.
- Сообщение выводится в зоне формы.
- Кнопка "Войти" становится доступной после корректировки.

### empty

- Если поля пусты, кнопка "Войти" должна быть неактивной.
- При попытке отправки пустой формы показывается подсказка об обязательных полях.

### loading

- При отправке формы кнопка переходит в состояние загрузки.
- Поля могут быть временно заблокированы.
- Отображается индикатор загрузки на кнопке или рядом.

### offline

- При отсутствии сети отображается баннер "Нет подключения к интернету".
- Кнопка "Войти" отключена.
- Форма остаётся видимой, но недоступной для отправки.

### permissions

- Страница не должна содержать элементы, требующие аутентифицированной роли.
- Если пользователь уже авторизован, перенаправить на главную страницу.

## Interaction & Animation

Правила поведения и переходы:

- Поля ввода получают фокус с мягким подсветом.
- При наведении на кнопку "Войти" меняется оттенок фона.
- При нажатии на кнопку она слегка уменьшается по масштабу.
- Ошибка валидации отображается плавно, без резкого мерцания.
- Смена цитаты и изображения при обновлении страницы может происходить без анимации или с лёгким fade-in для всего левого блока.
- Загрузка формы сопровождается индикатором на кнопке.
- При появлении баннера offline он выезжает сверху или появляется через fade-in.
- Переходы между состояниями формы должны быть быстрыми и плавными (150-200 мс).

## Зависимости и ограничения

- API:
  - POST /auth/login — аутентификация по логину и паролю.
  - GET /auth/quote — получение случайной пары: изображение деятеля и цитата.
- Роли:
  - доступ к странице имеет любой пользователь без авторизации.
  - при авторизованном состоянии редирект на основную страницу.
- Feature flags:
  - флаг `auth_new_ui` для включения обновленного интерфейса (если реализован).
- Технические ограничения:
  - форма должна корректно работать в современных браузерах (Chrome, Edge, Firefox).
  - адаптивный layout: центральный контейнер масштабируется для мобильных устройств и планшетов.
  - изображение деятеля и цитата должны загружаться асинхронно.

## Acceptance criteria

- Центральный контейнер выровнен по центру экрана.
- Левая и правая части четко разделены визуально.
- Левая часть содержит изображение деятеля медицины и цитату.
- При обновлении страницы изображение и цитата меняются.
- Правая часть содержит форму с полями логина и пароля и кнопкой "Войти".
- Поле логина и поле пароля валидируются как обязательные.
- При успешном вводе и отправке пользователь перенаправляется дальше.
- При неверных данных отображается понятное сообщение об ошибке.
- При отсутствии сети показывается offline-баннер и кнопка отключается.
- Все состояния отображаются корректно и логично.
