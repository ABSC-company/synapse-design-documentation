# 1.2.2. Booking Page

## Спецификация страницы интерфейса

- Страница: Записи пациентов
- Модуль: Амбулаторная регистратура
- Позиция в модуле: Вторая страница интерфейса сотрудника регистратуры
- Тип страницы: Операционная рабочая область расписания
- Основные роли: Регистратор, старший регистратор, администратор регистратуры
- Статус документа: Draft

## Назначение страницы и сценарии использования

Страница предназначена для управления расписанием записей пациентов на прием и исследования в формате графика (Booking Scheduler).

Ключевые задачи страницы:

1. Просмотр расписания записей на выбранный день.
2. Переключение контекста расписания по типам услуг через Segment Control.
3. Поиск по ФИО врача и ФИО пациента с подсветкой найденных совпадений.
4. Фильтрация расписания через Drawer по времени и услугам.
5. Создание новой записи на прием через кнопку Добавить запись.
6. Создание записи через выделение временного диапазона на строке врача.

Общий layout страницы:

1. Слева: Sidebar в свернутом состоянии.
2. Сверху: **Page Header** — заголовочный блок с кнопками управления (см. [Page Header.md](../1.2.0. Module layout components/1.2.0.1. Page Header/Page Header.md)).
3. Справа: **Electronic Queue** — блок электронной очереди по правому краю (см. [Electronic Queue.md](../1.2.0. Module layout components/1.2.0.2. Electronic Queue/Electronic queue.md)) вместе с календарём (влияет на таблицу визитов).
4. Контентная область: компонент Booking Scheduler.

**Page Header содержит две основные кнопки действий:**
- **Зарегистрировать пациента** — открывает модаль регистрации нового пациента
- **Запись пациента** — открывает модаль создания записи пациента к врачу

Обе кнопки работают аналогично странице Визиты и открывают те же модальные сценарии (см. подробнее в [Page Header.md](../1.2.0. Module layout components/1.2.0.1. Page Header/Page Header.md)).

## User Flows / Task Flows для страницы

### Flow 1. Просмотр расписания на выбранную дату

1. Пользователь открывает страницу Записи пациентов.
2. Система отображает текущую дату в горизонтальном календаре.
3. Система загружает Booking Scheduler на выбранную дату.
4. Пользователь просматривает слоты записи по строкам врачей.

### Flow 2. Переключение сегмента расписания

1. Пользователь выбирает параметр в Segment Control.
2. Доступные параметры:
    - Консультация врачей.
    - Инструментальные исследования.
    - Лабораторные исследования.
3. Система переключает контент графика на выбранный тип расписания.

### Flow 3. Поиск и фильтрация в Scheduler

1. Пользователь вводит запрос в инпут поиска.
2. Поиск работает по ФИО врача и ФИО пациента.
3. Совпадения в графике подсвечиваются через Highlight.
4. Пользователь нажимает кнопку Фильтр.
5. Открывается Drawer фильтрации.
6. Пользователь настраивает фильтры по времени и услугам.
7. Система обновляет отображение графика согласно фильтрам.

### Flow 4. Добавление записи через кнопку

1. Пользователь нажимает кнопку **Добавить запись** рядом с поиском и фильтром.
2. Открывается модальное окно создания записи пациента к врачу (см. [Page Header.md — «Кнопка «Запись пациента к врачу»**](../1.2.0. Module layout components/1.2.0.1. Page Header/Page Header.md)).
3. Пользователь заполняет модальную форму и сохраняет запись.
4. После успешного сохранения график обновляется.

### Flow 5. Добавление записи через выделение слота

1. Пользователь мышью выделяет временной диапазон в строке конкретного врача.
2. Система открывает модальное окно записи пациента.
3. В модале автоматически подставляются врач и выбранное время.
4. Пользователь подтверждает запись.
5. Система отображает новую запись в соответствующем слоте графика.

## Структура: Wireframes

Структурные блоки страницы:

1. Header модуля:
    - Кнопка Зарегистрировать пациента.
    - Кнопка Запись пациента.
2. Контент:
    - Горизонтальный календарь выбора дня.
    - Toolbar Scheduler:
        - Segment Control слева над графиком.
        - Инпут поиска справа.
        - Кнопка Фильтр справа.
        - Кнопка Добавить запись рядом с поиском и фильтром.
        - График Booking Scheduler.

Структура Booking Scheduler:

| Элемент           | Описание                                                |
| ----------------- | ------------------------------------------------------- |
| Заголовки графика | Врачи или кабинеты в зависимости от выбранного сегмента |
| Сетка времени     | Горизонтальная шкала времени, разбитая на ячейки        |
| Ячейка графика    | 1 ячейка = среднее время приема                         |
| Слот записи       | Визуальный блок записи пациента                         |
| Область выделения | Интерактивное выделение времени мышью                   |

Правило позиционирования контролов:

1. Segment Control расположен в левом верхнем углу области графика.
2. Поиск, Фильтр и Добавить запись расположены справа на одной линии с Segment Control.

## Визуал: High-Fidelity mockups

Визуальные правила:

1. Scheduler является доминирующим элементом страницы.
2. Сегмент переключения должен быть визуально связан с контентом графика.
3. Highlight найденных совпадений должен быть отчетливо заметен.
4. Drawer фильтрации открывается поверх контента справа.
5. Выделение диапазона времени мышью должно иметь явную визуальную индикацию.

Визуальные компоненты, обязательные к отрисовке в mockups:

1. Состояние закрытого Sidebar.
2. Header с 2 кнопками, как на странице Визиты.
3. Горизонтальный календарь.
4. Segment Control с 3 параметрами.
5. Поиск с примером подсветки результатов.
6. Кнопка Фильтр и открытый Drawer.
7. Кнопка Добавить запись.
8. Scheduler с заполненными слотами.
9. Состояние выделения диапазона времени на строке врача.
10. Открытый модал записи с автоподстановкой врача и времени.

## Состояния и edge cases

### Базовые состояния страницы

| Состояние   | Ожидаемое поведение                                                                                     |
| ----------- | ------------------------------------------------------------------------------------------------------- |
| loading     | Skeleton календаря, toolbar и сетки scheduler, блокировка интерактивных действий до завершения загрузки |
| empty       | Нет записей на выбранный день, отображается пустое состояние с CTA смены даты или сброса фильтров       |
| error       | Ошибка загрузки данных scheduler с кнопкой Повторить                                                    |
| offline     | Баннер отсутствия сети, невозможность создать или изменить запись, восстановление после reconnect       |
| permissions | Скрытие или disable действий создания/изменения согласно роли                                           |

### Edge cases: Scheduler и сегменты

1. Для выбранного сегмента нет доступных врачей или кабинетов.
2. Для выбранной даты нет ни одного свободного или занятого слота.
3. Сегмент переключен, но данные нового сегмента не загрузились.
4. Пользователь быстро переключает сегменты, ответы API приходят в разном порядке.
5. Длительность услуги не кратна размеру ячейки графика.
6. В расписании есть пересекающиеся записи у одного врача.
7. Врач временно недоступен, но отображается в заголовке графика.
8. При смене даты сохраняется старый сегмент, но фильтры становятся невалидными.

### Edge cases: поиск, highlight и фильтр

1. Поиск пустой, отображаются все данные согласно сегменту и фильтрам.
2. Поиск по ФИО врача не находит совпадений.
3. Поиск по ФИО пациента не находит совпадений.
4. Несколько совпадений в одной строке врача, подсветка не должна конфликтовать.
5. Поиск возвращает слишком много совпадений, интерфейс должен оставаться читаемым.
6. Drawer фильтра не открывается из-за ошибки UI-состояния.
7. В Drawer выбран диапазон времени вне рабочего графика.
8. Выбранные услуги отсутствуют в текущем сегменте.
9. После применения фильтра список пуст.
10. После сброса фильтра исходные данные не восстановились.

### Edge cases: добавление записи

1. Кнопка Добавить запись недоступна из-за прав роли.
2. Модал записи не открылся из-за ошибки состояния.
3. Запись пытаются создать в уже занятом временном слоте.
4. За время заполнения модала слот занял другой оператор.
5. Автоподстановка врача и времени при выделении работает некорректно.
6. Выделен слишком маленький диапазон времени, меньше минимальной длительности услуги.
7. Выделен диапазон, выходящий за границы рабочего времени врача.
8. Пользователь выделил диапазон в сегменте лаборатории, но выбран врач из сегмента консультаций.
9. Модал закрыт без сохранения после выделения, временное выделение должно быть снято.
10. После успешного создания записи новый слот не отобразился без перезагрузки.

### Edge cases: интеграция с модалом записи из Visits Page

1. Модал из Booking Page и модал из Visits Page расходятся по полям или правилам.
2. Feature flag для создания записи отключен, но кнопка Добавить запись видима.
3. В Header доступна кнопка Запись пациента, но кнопка Добавить запись в Scheduler скрыта.
4. Модал открыт из выделения слота, но предзаполненные врач/время перезаписаны некорректно.

## Interaction & Animation

Правила поведения:

1. Смена даты в календаре инициирует перезагрузку Scheduler на выбранный день.
2. Переключение Segment Control мгновенно меняет контент графика.
3. Поиск применяется без перехода на другую страницу.
4. Фильтр открывается в Drawer и применяется без полной перезагрузки страницы.
5. Кнопка Добавить запись и выделение слота открывают один и тот же модал записи.
6. После успешного сохранения записи график обновляется точечно.

Переходы и анимации:

1. Переключение сегментов сопровождается коротким cross-fade контента графика.
2. Drawer фильтра открывается/закрывается плавным slide справа.
3. Выделение диапазона в графике анимируется полупрозрачным слоем.
4. Highlight поисковых совпадений появляется без рывков и не меняет геометрию слотов.

## Зависимости и ограничения

### API

1. API загрузки расписания на дату и сегмент.
2. API загрузки врачей или кабинетов для текущего сегмента.
3. API поиска по ФИО врача и ФИО пациента.
4. API фильтрации по времени и услугам.
5. API создания записи пациента на прием.
6. API проверки доступности слота в реальном времени.
7. API, используемые модалом записи из страницы 1.2.1 Visits Page.

### Роли и права

1. Регистратор: просмотр scheduler, поиск, фильтр, создание записи при наличии права.
2. Старший регистратор: расширенные права управления записями.
3. Администратор регистратуры: полный доступ к настройкам и действиям.
4. Ограничения на создание записи и доступ к отдельным сегментам по роли.

### Feature flags

1. Включение Booking Scheduler для модуля.
2. Включение сегментов лабораторных исследований.
3. Включение сегментов инструментальных исследований.
4. Включение поиска с Highlight.
5. Включение режима создания записи через выделение временной области.

### Технические ограничения

1. Точность слотов зависит от настроенного среднего времени приема.
2. При больших объемах данных требуется виртуализация сетки графика.
3. Корректность расписания зависит от синхронизации с графиками врачей и кабинетов.
4. Производительность поиска и highlight зависит от количества одновременно отображаемых слотов.

## Acceptance criteria

1. Страница Записи пациентов отображается как вторая страница модуля регистратуры.
2. Страница использует общий layout: слева свернутый Sidebar, сверху Header.
3. В Header доступны 2 кнопки, идентичные странице Визиты.
4. Контентная область содержит горизонтальный календарь и Booking Scheduler.
5. Segment Control отображает 3 параметра: консультации, инструментальные исследования, лабораторные исследования.
6. Переключение сегмента обновляет соответствующий контент графика.
7. Поиск работает по ФИО врача и ФИО пациента.
8. Найденные совпадения в графике подсвечиваются через Highlight.
9. Кнопка Фильтр открывает Drawer с настройками по времени и услугам.
10. Кнопка Добавить запись открывает тот же модал записи пациента, что и на странице 1.2.1 Visits Page.
11. График реализован горизонтальной сеткой, где 1 ячейка соответствует среднему времени приема.
12. Заголовки графика соответствуют врачам или кабинетам в рамках выбранного сегмента.
13. Выделение области времени мышью на строке врача открывает модал записи.
14. При открытии модала из выделения автоматически подставляются врач и выбранное время.
15. Состояния loading, empty, error, offline, permissions реализованы и проверены.
16. Edge cases для сегментов, поиска, фильтров и добавления записи покрыты интерфейсно и системно.
