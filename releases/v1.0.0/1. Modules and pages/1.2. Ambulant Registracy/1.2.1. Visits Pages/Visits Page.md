# 1.2.1. Visits Page

## Спецификация страницы интерфейса

- Страница: Визиты
- Модуль: Амбулаторная регистратура
- Позиция в модуле: Первая страница интерфейса сотрудника регистратуры
- Тип страницы: Операционная рабочая область
- Основные роли: Регистратор, старший регистратор, администратор регистратуры
- Статус документа: Draft

## Назначение страницы и сценарии использования

Страница предназначена для ежедневной работы регистратуры с потоком визитов пациентов в клинику и вложенными визитами к врачам по услугам.

Ключевые задачи страницы:

1. Просмотр всех визитов в клинику на выбранную дату.
2. Быстрый поиск и фильтрация визитов.
3. Управление визитом в клинику через контекстные действия.
4. Управление вложенными визитами к врачам.
5. Регистрация нового пациента.
6. Создание новой записи пациента к врачу.

Общий layout страницы:

1. Слева: Sidebar в закрытом состоянии.
2. Сверху: Header модуля регистратуры.
3. Основная область контента: горизонтальный календарь и таблица визитов.

## User Flows / Task Flows для страницы

### Flow 1. Просмотр визитов на выбранный день

1. Пользователь открывает страницу Визиты.
2. Система отображает текущую дату в горизонтальном календаре.
3. Система загружает таблицу общих визитов в клинику на выбранный день.
4. Пользователь раскрывает строку визита в клинику и просматривает вложенные визиты к врачам.

### Flow 2. Поиск и фильтрация визитов

1. Пользователь вводит запрос в поиск.
2. Поиск выполняется по: ФИО пациента, внутренний ID пациента, направление, ФИО врача, услуга.
3. Пользователь открывает фильтр и применяет критерии: направление, ФИО врача, услуга и цена.
4. Таблица обновляется с учетом поиска и фильтров.

### Flow 3. Действия по визиту в клинику

1. Пользователь открывает контекстное меню строки визита через кнопку Действие или ПКМ.
2. Пользователь выбирает действие:
    - Отменить визит: открывается модальное окно подтверждения отмены.
    - Скачать все результаты: открывается модальное окно со списком документов для скачивания.
    - Распечатать все результаты: открывается нативное окно печати браузера.

### Flow 4. Действия по вложенному визиту к врачу

1. Пользователь раскрывает строку визита в клинику.
2. В подтаблице выбирает визит к врачу.
3. Открывает контекстное меню действия по кнопке или ПКМ.
4. Доступные действия: изменить, отменить, оплатить, скачать результаты, распечатать результаты.
5. Если результатов нет, система показывает Toast о неготовности результатов.

### Flow 5. Регистрация пациента

1. Пользователь нажимает кнопку Зарегистрировать пациента в Header.
2. Открывается модальное окно регистрации.
3. Пользователь выбирает тип гражданства: гражданин РУз или иностранный гражданин.
4. Пользователь заполняет 3 раздела формы через Tab.
    - Над компонентом Tab:
        - Инпут поиска по: если гражданин РУз - по PINFL или FaceID; если гражданин иностранного государства - MRZ (Machine Readable Zone, попробовать реализовать сканер документов) - реализация поиска совпадений через дебаунсинг
    - Содержание раздела 1 Персональные данные:
        - MRZ паспорта (если сканировался при поиске - автозаполняется. Из MRZ можно распарсить иные данные, включая: страну, пол, фамилию, имя) (не обязательно)
        - идентификационный номер/серия паспорта (номер/серия паспорта/загран. паспорта/метрики) (не обязательно)
        - Ф.И.О (3 отдельных инпута, отчество не обязательное)
        - дата рождения
        - пол
        - страна
        - город/провинция
        - адрес (только для гражданина РУз)
        - контактная информация - тип контактной информации (список типов: номер телефона, почта, соц. сеть); контактная информация (номер, адрес почты, ссылка на профиль) - минимум 1 контактная информация
        - фото пациента (необязательно)
        - категория пациента (из списка) (необязательно) 
    - Содержание раздела 2 Физиологические данные (полностью необязательно):
        - рост
        - вес
        - индекс массы тела (автоматически высчитывается исходя из роста и веса)
        - группа крови
    - Содержание раздела 3 Особые данные (полностью необязательно):
        - хронические заболевания
        - инвалидность
        - аллергии
5. Система автоматически выполняет обязательный поиск дубликатов до подтверждения формы.
6. Если дубликатов нет, система генерирует документ "Соглашение о обработке персональных данных", которую пациент подписывает через свою графическую подпись (на планшете, смартфоне).
7. После подписания, система подтверждает регистрацию и создает ЭМК пациента.
8. Если дубликат найден или пациент не подписал в течении 10 минут соглашение о обработке ПД, подтверждение блокируется, регистрация не выполняется и показывается Toast.

### Flow 6. Запись пациента к врачу

1. Пользователь нажимает кнопку Запись пациента в Header.
2. Пользователь может также нажать кнопку Создать визит в шапке таблицы рядом с поиском.
3. Обе кнопки открывают одно и то же модальное окно создания визита пациента.
4. Пользователь ищет пациента по ФИО, ID пациента, PINFL, FaceID, QR (если иностранный гражданин то и по MRZ).
5. Система показывает активные незавершенные визиты в клинику, если они есть.
6. Если выбран активный визит, разблокируются формы визита к врачу и загрузки файлов.
7. Если активного визита нет или нужен новый, разблокируется форма визита в клинику.
8. Пользователь заполняет форму и сохраняет запись:
    - Форма визита к врачу:

## Структура: Wireframes

Структурные блоки страницы:

1. Header модуля:
    - Кнопка Зарегистрировать пациента.
    - Кнопка Запись пациента.
2. Контент:
    - Горизонтальный календарь выбора дня.
    - Таблица визитов в клинику:
        - Панель поиска и фильтров таблицы.
        - Кнопка Создать визит рядом с поиском в шапке таблицы.
        - Подтаблица визитов к врачам внутри каждой строки визита в клинику.

Структура таблицы визитов в клинику:

| Колонка           | Описание                                  |
| ----------------- | ----------------------------------------- |
| Дата и время      | Дата и время визита в клинику             |
| Пациент           | ФИО пациента                              |
| Направление       | Причина обращения или профиль специалиста |
| Степень срочности | Статус триажа                             |
| Тип визита        | Амбулаторный или стационарный             |
| Статус            | Закончен или не закончен                  |
| Действие          | Контекстное меню действий                 |


Правило статуса визита в клинику:

1. Визит считается законченным, если все вложенные визиты к врачу имеют статус завершен или отменен.
2. В иных случаях статус визита в клинику не закончен.

Структура подтаблицы визитов к врачу:

| Колонка           | Описание                                             |
| ----------------- | ---------------------------------------------------- |
| Дата и время      | Дата и время приема у врача                          |
| Врач              | ФИО и должность врача                                |
| Направление врача | Медицинское направление врача                        |
| Услуга            | Название услуги и стоимость                          |
| Статус            | Предварительно, оплачен, завершен, отсрочен, отменен |
| Действие          | Контекстное меню действий                            |

## Визуал: High-Fidelity mockups

Визуальные правила:

1. Таблица является приоритетным визуальным элементом страницы.
2. Календарь расположен над таблицей и влияет на данные таблицы.
3. Контекстные действия визуально доступны через кнопку и через ПКМ.
4. Модальные окна используются для подтверждений, регистрации, записи и работы с результатами.
5. Toast используется для короткой обратной связи, в том числе при отсутствии результатов.

Визуальные компоненты, обязательные к отрисовке в mockups:

1. Состояние закрытого Sidebar.
2. Header с двумя основными CTA-кнопками.
3. Таблица с раскрытой подтаблицей.
4. Контекстные меню для обеих таблиц.
5. Модальное окно отмены визита.
6. Модальное окно скачивания всех результатов с таблицей документов.
7. Модальное окно оплаты услуги.
8. Модальное окно регистрации пациента.
9. Модальное окно записи пациента.

## Состояния и edge cases

### Базовые состояния страницы

| Состояние   | Ожидаемое поведение                                                                       |
| ----------- | ----------------------------------------------------------------------------------------- |
| loading     | Skeleton календаря и таблицы, блокировка действий до завершения загрузки                  |
| empty       | Пустое состояние с пояснением и CTA сброса фильтров или смены даты                        |
| error       | Ошибка загрузки с кодом и кнопкой Повторить                                               |
| offline     | Баннер отсутствия сети, недоступность мутаций, повторная синхронизация при восстановлении |
| permissions | Скрытие или дизейбл недоступных действий согласно роли                                    |

### Edge cases: таблица визитов в клинику

1. На выбранную дату нет визитов.
2. Таблица визитов за выбранный день загружается целиком; возможно состояние, когда родительские визиты уже отображены, а вложенные визиты к врачам еще догружаются отдельным запросом.
3. Визит есть, но вложенные визиты к врачам не успели загрузиться или загрузились с ошибкой.
4. Не удалось раскрыть подтаблицу из-за ошибки API.
5. Статусы вложенных визитов изменились в фоне и конфликтуют с отображаемым статусом родительского визита.
6. Отмена визита недоступна из-за отсутствия прав.
7. Попытка отмены уже завершенного визита.
8. Попытка отмены визита, по которому уже проведена оплата по одной или нескольким услугам.
9. Контекстное меню вызвано на записи, которая недоступна для действий из-за статуса или ограничений прав, при этом визиты не удаляются и не архивируются.
10. Скачать все результаты: документов нет.
11. Скачать все результаты: часть документов недоступна из-за прав.
12. Скачать все результаты: выбран неподдерживаемый формат хотя бы для одного документа.
13. Печать всех результатов недоступна из-за блокировки popup браузером.

### Edge cases: вложенный визит к врачу

1. Изменить визит недоступно, так как статус завершен.
2. Изменить визит недоступно, так как врач недоступен в выбранное время.
3. Выбранная услуга больше не предоставляется врачом.
4. Отмена вложенного визита запрещена после фискализации оплаты.
5. Оплата недоступна для роли при отключенном праве приема платежей в регистратуре.
6. Оплата отклонена терминалом.
7. Терминал недоступен или офлайн.
8. Оплата прошла в терминале, но не сохранилась в системе.
9. Скачать результаты по услуге: результатов нет, показывается Toast.
10. Печать результатов по услуге: документ поврежден или отсутствует.

### Edge cases: поиск и фильтры

1. Пустой поисковый запрос.
2. Слишком короткий запрос.
3. Слишком длинный запрос.
4. Ввод спецсимволов.
5. Совпадений нет.
6. Фильтры взаимоисключающие и дают пустой набор.
7. Сброс фильтров не возвращает исходный список.
8. Потеря состояния фильтров при смене даты.

### Edge cases: регистрация пациента

1. Не выбран тип гражданства.
2. Для гражданина РУз не введен PINFL.
3. PINFL невалидного формата.
4. OneID недоступен.
5. OneID вернул неполные данные.
6. OneID вернул данные, но пользователь изменил обязательные поля на невалидные.
7. В системе уже есть пациент с тем же PINFL.
8. Для иностранного гражданина найден дубль по ФИО и дате рождения.
9. Дубликат найден на обязательном этапе проверки до подтверждения формы.
10. Фото загружено в неподдерживаемом формате.
11. Фото превышает лимит размера.
12. Контактная информация заполнена некорректно.
13. Пользователь закрыл модальное окно с несохраненными данными.

### Edge cases: запись пациента

1. Пациент не найден по ФИО.
2. Пациент не найден по ID.
3. Пациент не найден по PINFL.
4. Пациент не найден по FaceID.
5. QR не читается.
6. У пациента есть несколько активных незавершенных визитов.
7. Пользователь выбрал активный визит, но он завершился другим оператором до сохранения.
8. Нет активных визитов, но форма визита в клинику не разблокировалась.
9. Обязательные поля визита в клинику не заполнены.
10. Не найдено свободных окон букинга у врача на текущий день.
11. Направление выбрано первым и не нашлось врачей.
12. Врач выбран первым и система не смогла определить направление.
13. Попытка прикрепить файл недопустимого типа.
14. Попытка прикрепить файл сверх лимита.
15. Ошибка сохранения после успешной загрузки файлов.

## Interaction & Animation

Правила поведения:

1. Выбор даты в календаре мгновенно перезагружает таблицу визитов.
2. Раскрытие строки визита в клинику выполняется без перехода на другую страницу.
3. Контекстное меню записи доступно по кнопке действия и по ПКМ.
4. Критичные действия отмены всегда требуют подтверждения в модальном окне.
5. После успешного действия строка таблицы обновляется без полного перезагрузки страницы.

Переходы и анимации:

1. Открытие модальных окон с короткой fade и scale анимацией.
2. Раскрытие подтаблицы с вертикальной анимацией высоты.
3. Toast появляется в правом верхнем углу и автоматически скрывается.
4. Кнопки в loading получают визуальный индикатор выполнения.

## Зависимости и ограничения

### API

1. API списка визитов в клинику по дате.
2. API вложенных визитов к врачам.
3. API отмены визита в клинику.
4. API изменения, отмены, оплаты визита к врачу.
5. API получения и скачивания медицинских документов и результатов.
6. API регистрации пациента.
7. API проверки дубликатов пациентов.
8. API поиска пациента по ФИО, ID, PINFL, FaceID, QR.
9. API интеграции с OneID для автозаполнения данных гражданина РУз.
10. API интеграции с платежным терминалом и фискализацией чеков.

### Роли и права

1. Регистратор может просматривать визиты и выполнять базовые действия.
2. Право приема оплаты в регистратуре включается глобальной настройкой.
3. Доступ к скачиванию и печати документов зависит от медицинских прав доступа.
4. Доступ к отмене визитов ограничен ролью и статусом визита.

### Feature flags

1. Включение приема оплаты в регистратуре.
2. Включение поиска по FaceID.
3. Включение поиска по QR коду приложения пациента.
4. Включение интеграции OneID.
5. Включение интеграции с платежным терминалом.

### Технические ограничения

1. Печать зависит от нативного механизма браузера и системных настроек пользователя.
2. Скорость загрузки таблицы зависит от объема визитов за выбранный день.
3. Для корректного вычисления статуса родительского визита нужна консистентность статусов вложенных визитов.
4. Поддерживаемые форматы документов и размер файлов ограничены серверной конфигурацией.

## Acceptance criteria

1. Страница Визиты открывается первой в модуле амбулаторной регистратуры и отображает общий layout с закрытым Sidebar и Header.
2. Выбор даты в горизонтальном календаре обновляет таблицу визитов на выбранный день.
3. Таблица визитов в клинику содержит обязательные колонки и корректно рассчитывает статус визита.
4. Каждая запись визита в клинику раскрывает подтаблицу визитов к врачам с обязательными колонками.
5. Контекстное меню строки визита в клинику открывается по кнопке действия и по ПКМ.
6. Кнопка Создать визит в шапке таблицы открывает тот же модал создания визита пациента, что и CTA Визит пациента в Header.
7. Действие Отменить визит открывает модальное подтверждение и корректно завершает сценарий отмены.
8. Действие Скачать все результаты открывает модальное окно с таблицей документов и выбором формата.
9. Действие Распечатать все результаты открывает нативное окно печати.
10. Контекстное меню вложенного визита к врачу содержит изменить, отменить, оплатить, скачать результаты, распечатать результаты.
11. При отсутствии результатов по услуге показывается Toast о неготовности результатов.
12. Поиск работает по ФИО пациента, внутреннему ID, направлению, ФИО врача и услуге.
13. Фильтры работают по направлению, ФИО врача и услуге с ценой.
