## Описание

Панель администратора — не отдельная страница, а набор экранов поверх двух модулей: `AUTH` (пользователи, роли, права, сессии) и `MGMT` (организация, оргструктура, системные настройки). Плюс — аудит и техподдержка, которые пронизывают всю систему.

Пока не пройден мастер первичной настройки, клиника считается несконфигурированной, и рабочие модули (Регистратура, ЭМК, Склад, Касса и т.д.) недоступны для реальной работы.

Экраны панели:

1. **Мастер первичной настройки** — обязателен при первом запуске.
2. **Пользователи** — список и карточки учётных записей.
3. **Роли и права** — матрица доступа по модулям.
4. **Организация / Оргструктура** — реквизиты и дерево филиалов/этажей/отделений.
5. **Системные настройки** — параметры, влияющие на несколько модулей сразу.
6. **Подключение 2FA** — экран для любого пользователя, которому по роли требуется второй фактор.
7. **Аудит-лог** — просмотр журнала изменений по всей системе.
8. **Отчёты и статистика**, включая отдельный «Журнал откатов статусов».
9. **Техподдержка** — подключение к сессии пользователя («Войти как»).

---

## Сценарий использования

При первом запуске система не пускает дальше мастера настройки: сначала данные организации, затем первый администратор — сразу с подключением 2FA, затем базовая оргструктура. Отдельным опциональным шагом можно сразу выбрать, какие типы документов подпадают под правило «редактирует только автор в течение 24 часов» — но этот шаг можно пропустить и вернуться к нему позже в системных настройках.

В повседневной работе администратор заводит роли (набор прав, сгруппированных по модулям), затем создаёт пользователей и назначает им роли. Пользователь без роли доступа не получает — это единственный способ создать учётную запись.

Если пользователю нужна помощь, администратор открывает «Техподдержку», указывает причину и подключается к его сессии («Войти как»). Пользователь в этот момент обязательно получает уведомление — без исключений на срочность. Пока подключение активно, это видно и администратору, и пользователю.

Если что-то пошло не так на уровне данных (например, кассир по ошибке закрыл смену не тем отчётом), администратор может откатить статус сущности — но только через операцию, которую явно предоставляет модуль-владелец этих данных. Панель никогда не правит чужие данные напрямую. Каждый такой откат обязательно виден в аудит-логе и отдельно — в «Журнале откатов статусов», с указанием причины.

Подозрительную или зависшую сессию администратор может завершить принудительно — но не обрывом на полуслове: пользователь видит предупреждение с обратным отсчётом и успевает сохраниться, прежде чем сессия закроется.
