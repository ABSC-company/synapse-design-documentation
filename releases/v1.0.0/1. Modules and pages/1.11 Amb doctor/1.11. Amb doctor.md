# 1.11. Амбулаторный врач (+ ЭМК)

Рабочее место амбулаторного врача объединяет два модуля из карты системного ТЗ: `PHYS` («Амбулаторный врач» — приёмы, назначения, заключения) и `EMR` («ЭМК» — электронная медицинская карта пациента, клинические записи). В этом релизе они описываются одним документом, но владение данными разделено по сущностям явно (`EMR` / `PHYS`) — без третьего «теневого» владельца.

Функции врача, от которых оттолкнулось это ТЗ:
1. Приём пациентов, записанных через регистратуру, — вызов пациента из электронной очереди.
2. Консультация с созданием медицинского документа.
3. Направление пациентов к другим врачам.
4. Назначение лабораторных и инструментальных исследований.
5. Просмотр истории болезни пациента.
6. ЭМК должна быть максимально информативной для врача.

Раздел детализирован до уровня, достаточного для проектирования модели данных (бэк) и экранов (дизайн). Названия сущностей и полей — рабочие, для проработки, не окончательный нейминг API/БД.

> Дизайн-документация (экраны, кнопки, wireframes) по этому ТЗ — в [1.11 Module Amb doctor](../1.11%20Module%20Amb%20doctor/Amb%20Doctor.md).
>
> Важно: печатный медицинский документ (заключение/справка) в дизайн-документации не расписывается отдельной формой — он открывается как готовый шаблон с полями для заполнения через модуль документооборота `DOCT` (см. `1.5. Document Management Module`).

### Терминология: «визит в клинику» vs «приём у врача»

> Решено: это два разных уровня, которые нельзя путать.
> - **Визит в клинику** (`Визит`, владеет `REG`) — верхнеуровневая сущность на весь путь пациента по клинике за день: от получения номерка на киоске до ухода. Номерок электронной очереди (`QueueTicket`) присваивается именно визиту в клинику и живёт с ним всё время, пока пациент перемещается между врачами/лабораторией/инструментальными исследованиями.
> - **Приём у врача** (`Encounter`, владеет `PHYS`, §1.2) — один конкретный «заход» в кабинет одного врача внутри визита в клинику. Направление к другому врачу закрывает текущий `Encounter`, но не закрывает визит в клинику и не создаёт новый талон — тот же номерок перенаправляется на нового врача.

## 1. Модель данных

### 1.1. Владеет `EMR`

#### ЭМК (PatientRecord)

> Единая карта на пациента, создаётся один раз. Согласно уже зафиксированной в проекте заметке (см. `1.3.3 EMC Base.md`): пока пациент ни разу не записался/не пришёл в клинику, его ЭМК не показывается врачу содержательно — пациент виден в списке, но детали скрыты/размыты. Поиск — по всем параметрам, включая ПИНФЛ, а не только ФИО/номер ЭМК.

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | номер ЭМК |
| пациент_id | FK Пациент (`REG`/`MGMT`) | да | |
| дата_первого_визита | date | нет | заполняется по факту первого визита; до этого момента карта видна ограниченно (см. §3) |
| аллергии | JSON, массив (наименование, реакция, тяжесть) | нет | выводится в «шапке» ЭМК на каждом экране врача — критично для безопасности |
| хронические_заболевания | JSON, массив (диагноз МКБ-10, дата установления) | нет | |
| группа_крови, резус | string | нет | |
| статус | enum: активна | да | ЭМК не «закрывается» и не удаляется, только версионируется |

| приём_id | FK Encounter (`PHYS`, §1.2) | да | какой приём породил эту запись |
| врач_id | FK Сотрудник (`HR`) | да | |
| дата_время | datetime | да | |
| жалобы | text | нет | |
| анамнез_заболевания, анамнез_жизни | text | нет | |
| объективный_осмотр | text/структурированные показатели (АД, пульс, температура и т.д.) | нет | |
| диагноз_основной | код МКБ-10 + текст | да, для подписанной консультации | решено: МКБ-10 — часть медицинского документа, заполняется врачом при консультации; справочник кодов — внешний международный стандарт (не бизнес-специфичный справочник клиники), поэтому спора о владении по образцу номенклатуры товара нет — `EMR` просто использует стандартный классификатор как источник значений поля |

### 1.2. Владеет `PHYS`

#### Приём (Encounter)

> Рабочая обёртка вокруг клинической консультации: кто, когда, в каком статусе. Создаётся, когда талон электронной очереди вызван к этому врачу (стык с компонентом «Электронная очередь», см. `Electronic Queue.md`), либо напрямую — при экстренном приёме по триажу (см. ниже).

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| visit_id | FK Визит (`REG`) | да | визит в клинику — один на весь путь пациента за день, общий для всех `Encounter` этого визита (см. «Терминология») |
| talon_id | FK QueueTicket (`REG`, компонент «Электронная очередь») | нет | тот же номерок, что и у визита в клинику; при направлении к другому врачу переиспользуется, не создаётся заново |
| врач_id | FK Сотрудник (`HR`) | да | |
| кабинет_id | FK `CLINIC` | да | |
| пациент_id | FK Пациент | да | |
| статус | enum: ожидает / идёт_приём / завершён / отменён / неявка | да | см. §2 |
| триаж_уровень | enum: плановый / экстренный | да | `экстренный` — приём вне очереди, минуя обычный порядок вызова; решено — инициируется регистратурой (`REG`) в амбулаторном сегменте, приёмным покоем в стационаре (вне области этого документа, будет описано вместе со стационарным модулем), см. §3, §8 |
| дата_время_начала / дата_время_окончания | datetime | нет / нет | |
| причина_длительной_консультации | text | обязательно, если порог включён клиникой и длительность превысила `порог_аномальной_длительности_приёма_минут` | порог — опциональная настройка, каждая клиника задаёт (или отключает) по-своему, см. §3, §6 |
| consultation_id | FK Consultation (`EMR`) | нет | заполняется, когда врач создаёт клиническую запись |

#### Направление (Referral)

| Поле | Тип | Обязательное | Примечание |
|---|---|---|---|
| id | UUID | да | |
| консультация_id | FK Consultation | да | из какой консультации создано |
| направляющий_врач_id | FK Сотрудник (`HR`) | да | |
| целевой_врач_id | FK Сотрудник (`HR`) | да | решено: направление всегда на конкретного врача, без варианта «только специализация» — по аналогии с тем, что и первичная запись в регистратуре всегда идёт к конкретному врачу |
| причина | text | да | |
| статус | enum: создано / в_очереди / выполнено / отменено | да | см. §2 |
| новый_приём_id | FK Encounter | нет | новый `Encounter` того же `visit_id` и того же `talon_id` — заполняется, когда для направления создан приём у целевого врача |

> Заказы на лабораторные и инструментальные исследования этот документ не владеет — они принадлежат будущим модулям `LAB`/`INSTR` («Заказы и результаты …» по карте модулей системного ТЗ). Врач инициирует их через публичный интерфейс этих модулей (§8), а в самой консультации хранится только ссылка на созданный заказ и на готовый результат.

## 2. Статусы и переходы

**Консультация:** `черновик` (врач заполняет) → `подписана` (врач завершил приём и подписал запись; дальнейшие правки — только новой версией, старая версия остаётся видна в истории).

**Приём (Encounter):**
| Из | В | Триггер |
|---|---|---|
| — | `ожидает` | создаётся по событию `REG.ТалонВызванКВрачу` |
| — | `идёт_приём` *(минуя `ожидает`)* | экстренный приём по триажу (`триаж_уровень = экстренный`) — врач/медсестра открывает приём сразу, без ожидания обычного вызова из очереди |
| `ожидает` | `идёт_приём` | врач подтверждает начало приёма (пациент физически зашёл) |
| `идёт_приём` | `завершён` | врач подписал консультацию |
| `ожидает` | `неявка` | пациент не зашёл за отведённое время (стык со статусом талона `пропущен_врач` в компоненте «Электронная очередь») |
| `ожидает` / `идёт_приём` | `отменён` | приём отменён вручную (редкий случай) |

**Направление:** `создано` → `в_очереди` (текущий `Encounter` переходит в `завершён`, но `visit_id`/`talon_id` не закрываются; у целевого врача создаётся новый `Encounter(ожидает)`, талон переадресуется — его `кабинет_id`/`врач_id` обновляются на нового врача, статус талона возвращается в `ожидает_врача`) → `выполнено` (целевой врач подписал свою консультацию) / `отменено`.

## 3. Бизнес-правила и валидации

| Правило | Проверка / поведение |
|---|---|
| ЭМК не видна до первого визита | Пока `PatientRecord.дата_первого_визита` не заполнена, пациент виден в списках (поиск находит его), но содержательные данные ЭМК показываются скрыто/размыто — согласовано с ранее зафиксированной заметкой проекта |
| Поиск пациента — по всем параметрам | ФИО, номер ЭМК, ПИНФЛ — не только по одному из полей |
| Приём начинается только по вызову из очереди, кроме экстренного триажа | Врач видит и открывает приём через «Вызвать пациента» (см. `Electronic Queue.md`); исключение — `триаж_уровень = экстренный`, когда приём открывается сразу, минуя обычный порядок вызова |
| Экстренный приём инициирует регистратура (в амбулаторном сегменте) | Решение о том, что пациенту нужен приём вне очереди, принимает регистратура при обращении пациента (`REG.ОткрытьЭкстренныйПриём`, см. §8) — не сам врач и не медсестра; в стационаре аналогичную роль будет играть приёмный покой (отдельный будущий модуль) |
| Порог аномальной длительности — опциональный, per-клиника | Клиника может не включать это ограничение вовсе; если включено — конкретное значение `порог_аномальной_длительности_приёма_минут` задаёт администратор этой клиники, единого значения по умолчанию для всех клиник нет |
| Аллергии всегда на виду | Список аллергий из `PatientRecord` отображается в «шапке» экрана консультации на каждом приёме, а не только в общей ЭМК |
| Консультация неизменяема после подписания | Правка подписанной `Consultation` создаёт новую версию (`версия + 1`); старая версия сохраняется целиком, не удаляется `[SYS-TRACE]` |
| Направление не закрывает визит в клинику, только текущий приём | Оформление `Referral` завершает текущий `Encounter`, но `visit_id` (визит в клинику) остаётся открытым; у целевого врача создаётся новый `Encounter` с тем же `visit_id` и тем же `talon_id` — номерок переадресуется, новый не выдаётся (счёт `Invoice` в `CASH` «дорастает» новой услугой того же визита — уже согласовано в ТЗ «Касса») |
| Направление — только на конкретного врача | `Referral.целевой_врач_id` обязателен; варианта «только специализация, распределить по свободным врачам» нет — согласуется с тем, что и первичная запись в регистратуре всегда идёт к конкретному врачу |
| Аномально долгая консультация — предупреждение и обязательная причина | Если приём (`идёт_приём`) длится дольше `порог_аномальной_длительности_приёма_минут` (настраиваемый параметр), система показывает врачу предупреждение; подписать консультацию сверх порога можно только заполнив `причина_длительной_консультации` |
| Заказ на исследование — не через собственную таблицу | Назначение лабораторного/инструментального исследования всегда идёт через публичный метод `LAB`/`INSTR` — этот модуль не хранит заказы и результаты напрямую, только ссылку в консультации |
| Результат исследования подтягивается автоматически | Как только `LAB`/`INSTR` публикует событие о готовом результате, он становится виден в ЭМК и в консультации, без ручного «импорта» врачом |
| Доступ к ЭМК — по функциональной необходимости | Принцип минимальных привилегий `[SYS-RBAC]`, `[SYS-PDATA]`; любое открытие/изменение ЭМК — под аудит-трейл `[SYS-AUDIT]` |

## 4. Роли и права
> В привязке к общей RBAC-модели (`AUTH`).

- **Врач (амбулаторный)** — приём пациентов по очереди, ведение консультации, подписание, направления к другим врачам, назначение исследований, просмотр истории болезни своих и (при праве) чужих пациентов.
- **Администратор** *(настройка модуля, не роль клиники)* — справочник МКБ-10, шаблоны документов (совместно с `DOCT`), настройка специализаций для направлений.

## 5. Пользовательские сценарии

**1. Приём пациента, записанного через регистратуру**
- *Предусловия:* талон пациента в статусе `ожидает_врача` для этого врача (см. «Электронная очередь»).
- *Основной поток:* врач на «Рабочем столе» видит список ожидающих пациентов → нажимает «Вызвать пациента» → создаётся/активируется `Encounter(ожидает)` → пациент заходит, врач подтверждает начало → `идёт_приём`.

**2. Консультация с созданием медицинского документа**
- *Основной поток:* врач заполняет `Consultation` (жалобы, анамнез, осмотр, диагноз МКБ-10, план) → при необходимости формирует печатный документ (`ClinicalDocument`, шаблон из `DOCT`) → подписывает консультацию → `Encounter` переходит в `завершён`, талон электронной очереди — в `завершён`.

**3. Направление пациента к другому врачу**
- *Основной поток:* в рамках открытой консультации врач создаёт `Referral` — выбирает конкретного целевого врача, указывает причину → текущий `Encounter` завершается, но визит в клинику (`visit_id`) остаётся открытым → у целевого врача появляется новый `Encounter(ожидает)` с тем же `visit_id` и тем же `talon_id` («Электронная очередь» переадресует номерок, новый не выдаётся) → после консультации у целевого врача `Referral` переходит в `выполнено`.

**4. Назначение лабораторного/инструментального исследования**
- *Основной поток:* врач в консультации выбирает «Назначить исследование» → указывает тип (лабораторное/инструментальное) и перечень → система вызывает публичный метод `LAB.СоздатьЗаказ`/`INSTR.СоздатьЗаказ` → в консультации сохраняется ссылка на заказ со статусом «назначено».
- *Продолжение:* когда `LAB`/`INSTR` публикует результат, событие обновляет ссылку в консультации на «результат готов»; врач видит его в ЭМК без дополнительных действий.

**5. Просмотр истории болезни пациента**
- *Основной поток:* врач открывает ЭМК пациента → видит ленту всех предыдущих консультаций (по датам, с врачами и диагнозами), документов, направлений и результатов исследований → открывает нужную запись для деталей.

**6. Работа с пациентом, впервые пришедшим в клинику**
- *Основной поток:* до первого визита ЭМК пациента показывается врачу только в общем виде (ФИО, дата рождения), без истории — истории попросту ещё нет; после подписания первой консультации `дата_первого_визита` заполняется, и ЭМК становится полноценно видимой при последующих визитах.

**7. Экстренный приём по триажу**
- *Основной поток:* пациент с экстренным случаем обращается в регистратуру (в амбулаторном сегменте) → регистратура оценивает ситуацию и вызывает `REG.ОткрытьЭкстренныйПриём` (указывает пациента и, если известно, конкретного врача) → создаётся `Encounter(идёт_приём, триаж_уровень = экстренный)` напрямую, минуя обычный порядок вызова из очереди → далее приём ведётся как обычно (консультация, направления, назначения).
- *Уточнение:* в стационаре аналогичную роль будет играть приёмный покой — этот сценарий будет уточнён при описании стационарного модуля.

**8. Предупреждение об аномально долгой консультации**
- *Основной поток:* приём находится в статусе `идёт_приём` дольше `порог_аномальной_длительности_приёма_минут` → врач видит системное предупреждение → при попытке подписать консультацию система требует заполнить `причина_длительной_консультации` → после заполнения подписание становится доступно.

## 6. Экраны (для дизайн-документации)

> Полная проработка экранов (wireframes, кнопки, состояния, edge cases) — см. [1.11 Module Amb doctor](../../1.%20Modules%20and%20pages/1.11%20Module%20Amb%20doctor/Amb%20Doctor.md).

**«Рабочий стол врача»**
- Список приёмов на сегодня (талоны в очереди этого врача): пациент, время постановки в очередь, статус.
- Кнопка «Вызвать пациента» (см. `Electronic Queue.md`).
- Быстрый доступ к истории пациента до вызова (просмотр без открытия приёма).

**«Консультация»**
- Шапка: ФИО пациента, возраст, аллергии (всегда на виду, выделены цветом), хронические заболевания.
- Форма записи: жалобы, анамнез заболевания/жизни, объективный осмотр (структурированные поля + свободный текст), диагноз (поиск по МКБ-10), план лечения/рекомендации.
- Кнопки: «Направить к врачу», «Назначить исследование», «Сформировать документ», «Подписать и завершить приём».
- Боковая панель — сокращённая ЭМК (см. ниже), доступная не покидая консультацию.

**«ЭМК пациента»** *(решено — максимальная информация о пациенте: история болезни, персональные данные, «что представляет собой пациент»)*
- Шапка: персональные/демографические данные (ФИО, дата рождения, ПИНФЛ, контакты), аллергии, хронические заболевания, группа крови.
- Лента визитов: хронологический список консультаций (дата, врач, диагноз, статус) с разворачиванием деталей.
- Вкладки: Документы, Направления, Результаты исследований (лабораторные/инструментальные, с датой и статусом).
- Поиск/фильтр по ленте (по врачу, периоду, диагнозу).

**«Направление к врачу»**
- Выбор целевого врача (с индикацией занятости, как в «Booking Page») — обязателен, выбора «только специализация» нет.
- Поле «Причина направления» (обязательно).

**«Назначение исследования»**
- Переключатель «Лабораторное / Инструментальное».
- Список доступных исследований (из справочника `LAB`/`INSTR`), мультивыбор.

**«Настройки модуля»** *(администратор)* — переключатель «Ограничивать длительность приёма» + число минут (опционально, своё значение для каждой клиники; при выключенном переключателе `причина_длительной_консультации` никогда не запрашивается).

**«Открыть экстренный приём»** *(регистратура)* — форма: пациент (поиск), целевой врач (если известен), причина экстренности → создаёт `Encounter(идёт_приём, триаж_уровень = экстренный)` в обход обычного вызова из очереди.

## 7. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| `PHYS-FR-01` | Приём пациента у врача создаётся по вызову из электронной очереди | must |
| `PHYS-FR-02` | Врач видит на «Рабочем столе» список пациентов, ожидающих приёма именно у него | must |
| `EMR-FR-01` | ЭМК пациента без ни одного визита в клинику отображается врачу в ограниченном виде (без содержательной истории) | must |
| `EMR-FR-02` | Поиск пациента доступен по ФИО, номеру ЭМК и ПИНФЛ | must |
| `EMR-FR-03` | Консультация после подписания недоступна для прямой правки — только для создания новой версии | must |
| `EMR-FR-04` | Аллергии пациента отображаются на экране консультации без дополнительных действий врача | must |
| `PHYS-FR-03` | Врач может создать направление к конкретному целевому врачу из открытой консультации | must |
| `PHYS-FR-04` | Направление автоматически создаёт приём у целевого врача в рамках того же визита в клинику, с тем же номерком очереди | must |
| `PHYS-FR-07` | Экстренный приём по триажу открывается регистратурой напрямую, минуя обычный порядок вызова из электронной очереди | must |
| `PHYS-FR-08` | Приём, превысивший настраиваемый порог длительности (если он включён клиникой), требует обязательного указания причины перед подписанием консультации | must |
| `PHYS-FR-09` | Ограничение по длительности приёма — опциональное, каждая клиника включает/выключает и задаёт значение самостоятельно | must |
| `PHYS-FR-05` | Врач может назначить лабораторное и/или инструментальное исследование из консультации | must |
| `PHYS-FR-06` | Результат исследования автоматически появляется в консультации/ЭМК без ручного импорта | must |
| `EMR-FR-05` | Врач видит полную историю болезни пациента (все подписанные консультации, документы, направления, результаты) | must |
| `EMR-FR-06` | Каждое открытие и изменение ЭМК фиксируется в аудит-трейле | must |

## 8. Публичный интерфейс

**Синхронные методы, которые вызывает `PHYS`/`EMR`:**

| Метод | Модуль-владелец | Назначение |
|---|---|---|
| `REG.ВызватьПациентаКВрачу` | `REG` | кнопка «Вызвать пациента» (уже описан в `Electronic Queue.md`) |
| `LAB.СоздатьЗаказ` / `INSTR.СоздатьЗаказ` | `LAB` / `INSTR` | назначение исследования из консультации |
| `DOCT.ПолучитьШаблон` | `DOCT` | формирование печатного документа консультации |

**Синхронные методы, которые предоставляет `PHYS` (вызывает `REG`):**

| Метод | Вход | Выход | Назначение |
|---|---|---|---|
| `PHYS.ОткрытьЭкстренныйПриём` | пациент_id, врач_id (опц.), причина | `Encounter(идёт_приём, триаж_уровень=экстренный)` | Регистратура открывает приём вне очереди по результатам оценки экстренности (§3, §5 сценарий 7) |

**Синхронные методы, которые предоставляет `EMR`/`PHYS`:**

| Метод | Вход | Выход | Назначение |
|---|---|---|---|
| `EMR.ПолучитьКраткуюЭМК` | пациент_id | демография, аллергии, хронические заболевания (без полной истории) | Для регистратуры/кассы — административные данные без клинической части |
| `EMR.ПроверитьПервыйВизит` | пациент_id | `дата_первого_визита` заполнена/нет | Для реализации правила ограниченной видимости ЭМК |

**Публикуемые доменные события:**

| Событие | Когда | Полезная нагрузка |
|---|---|---|
| `EMR.КонсультацияПодписана` | врач завершил и подписал приём | `consultation_id, эмк_id, приём_id` |
| `PHYS.ПриёмЗавершён` | приём переведён в `завершён` | `приём_id, visit_id` |
| `PHYS.НаправлениеСоздано` | оформлено направление к другому врачу | `referral_id, целевой_врач_id, visit_id` |

**Зависимости от других модулей:** `REG` (визит, компонент «Электронная очередь»), `HR` (врач, специализация), `CLINIC` (кабинет), `LAB`/`INSTR` (заказы и результаты исследований — модули ещё не детализированы), `DOCT`/`FILE` (шаблоны и хранение печатных документов), `CASH` (счёт визита «дорастает» при направлении к другому врачу), `AUTH` (роли и права), `NOTIF` (уведомление врача о готовом результате исследования).

## Решённые вопросы (зафиксировано в этой версии)

- **Приём вне электронной очереди** — возможен как экстренный приём по триажу (`Encounter.триаж_уровень = экстренный`), минуя обычный порядок вызова (§1.2, §2, §3, §5 сценарий 7, `PHYS-FR-07`).
- **Что переиспользуется при направлении к другому врачу** — тот же визит в клинику и тот же номерок очереди; закрывается только текущий приём у конкретного врача, а не весь визит. Введена явная терминология «визит в клинику» vs «приём у врача» (см. раздел «Терминология» в начале документа, §1.2, §2, §3, §5 сценарий 3).
- **Аномально долгая консультация** — система предупреждает врача и требует обязательную причину перед подписанием, если приём превысил настраиваемый порог длительности (§1.2, §3, §5 сценарий 8, §6 «Настройки модуля», `PHYS-FR-08`).
- **Владелец справочника МКБ-10** — это не предмет спора о владении данными: код МКБ-10 — просто поле медицинского документа, которое врач заполняет при консультации, справочник кодов — внешний международный стандарт (§1.1, Консультация).
- **Состав «максимально информативной» ЭМК** — история болезни, персональные/демографические данные, максимум информации о пациенте — отражено в экране «ЭМК пациента» (§6); точный состав виджетов остаётся гибким, а не фиксированным узким списком.
- **Направление без конкретного врача** — невозможно; направление всегда на конкретного целевого врача, поле `целевая_специализация` убрано из модели (§1.2, §3, §6, `PHYS-FR-03`).
- **Кто проводит триаж и инициирует экстренный приём** — в амбулаторном сегменте (этот релиз) это регистратура, через `PHYS.ОткрытьЭкстренныйПриём`; в стационаре аналогичную роль будет играть приёмный покой — уточнится при описании стационарного модуля (§1.2, §3, §5 сценарий 7, §8).
- **Значение порога аномальной длительности приёма** — не фиксируется на уровне системы; это опциональная настройка, которую администратор каждой клиники включает/выключает и задаёт по-своему (§1.2, §3, §6, `PHYS-FR-09`).
