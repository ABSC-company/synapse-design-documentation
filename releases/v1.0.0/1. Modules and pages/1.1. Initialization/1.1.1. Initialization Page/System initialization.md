# 1.1.1. System Initialization Page

## Спецификация страницы интерфейса

* Название страницы: Инициализация системы
* Модуль: Management
* Тип страницы: Первичная настройка клиники
* Основные роли: Администратор клиники, Оператор клиники
* Статус документа: Draft (актуализировано по бизнес-логике)
* Платформа: веб-приложение для медицинской системы

---

# Назначение страницы и сценарии использования

Страница предназначена для первоначальной настройки системы после установки. С помощью страницы пользователь может:

1. Подключить клинику на основании конфигурационных данных (файл или QR-код).
2. Создать первого администратора системы.
3. Настроить первый филиал.
4. Настроить системные параметры, включая выбор и кастомизацию справочников-стандартов (например, классификация пола, специализации и категории врачей).
5. Завершить процесс активации системы.

После успешного завершения инициализации пользователь получает возможность авторизоваться и работать в системе.

**Важная особенность логики:** все данные, введённые пользователем на шагах 1–4, накапливаются в рамках мастера и не применяются по отдельности на каждом шаге. Реальное применение всех данных происходит единомоментно на последнем шаге, при нажатии кнопки «Завершить настройку». Это значит, что до завершения мастера пользователь может свободно возвращаться назад и менять любые ранее введённые данные — ничего не считается окончательным, пока процесс не завершён целиком. Если в процессе завершения возникает ошибка — ни одна часть данных не сохраняется, система остаётся неинициализированной, и пользователь может повторить попытку.

Исключение — логотип клиники (см. Шаг 2): он сохраняется сразу при загрузке, отдельно от остального процесса.

# Layout страницы

Страница представляет собой полноэкранный мастер настройки (Wizard). Состоит из следующих областей:

## Header

Содержит:

* Название страницы
* Краткое описание текущего этапа
* Индикатор общего прогресса настройки

---

## Stepper

Расположен под Header.

Тип:

* Horizontal Stepper

Этапы:

1. Подключение клиники
2. Администратор
3. Филиал
4. Настройки системы
5. Завершение

Состояния этапов:

* Completed
* Active
* Disabled
* Error

Пользователь всегда видит:

* текущий этап;
* завершённые этапы;
* общее количество этапов.

---

## Content Area

Основная рабочая область страницы. Отображает контент текущего шага. Контент может прокручиваться независимо от остальных элементов страницы.

---

## Footer

Закреплён в нижней части страницы. Содержит кнопки управления мастером:

* Назад
* Далее
* Завершить настройку

Правила:

* На первом шаге кнопка «Назад» скрыта.
* На промежуточных шагах отображаются кнопки «Назад» и «Далее».
* На последнем шаге отображается кнопка «Завершить настройку».
* Переход на следующий шаг невозможен при наличии ошибок валидации.
* Переход между шагами 1–4 не приводит к окончательному сохранению данных — это происходит только на последнем шаге (см. «Назначение страницы»).

# User Flows

## Flow 1. Полная настройка системы

1. Пользователь открывает страницу.
2. Загружает данные клиники.
3. Создаёт администратора.
4. Создаёт первый филиал.
5. Настраивает параметры системы.
6. Завершает настройку — все данные применяются одновременно.
7. Получает доступ к авторизации.

---

## Flow 2. Подключение клиники

1. Пользователь загружает JSON-файл или сканирует QR-код.
2. Система локально считывает идентифицирующие данные клиники из файла/кода.
3. Пользователь видит подтверждение, что файл/код считан корректно.
4. Полноценная проверка подлинности этих данных и подтверждение клиники происходит на последнем шаге, в момент завершения настройки (см. Шаг 5).
5. Пользователь переходит к следующему шагу.

> **Изменение логики:** ранее карточка клиники с названием, типом лицензии, страной и статусом подключения отображалась сразу после загрузки файла. Теперь эти данные становятся известны и подтверждёнными только после полной верификации, которая происходит при завершении всего мастера, а не сразу после загрузки. На Шаге 1 можно показать только то, что считалось из файла локально (например, идентификатор клиники), без статуса подтверждения.

---

## Flow 3. Создание администратора

1. Пользователь вводит данные администратора.
2. Указывает название первой административной роли.
3. Переходит к следующему этапу. Учётная запись фактически создаётся только при завершении настройки на последнем шаге.

---

## Flow 4. Настройка филиала

1. Пользователь указывает данные филиала.
2. Настраивает местоположение.
3. Настраивает расписание работы и перерывы.
4. Подтверждает данные и переходит дальше. Филиал фактически создаётся только при завершении настройки на последнем шаге.

---

## Flow 5. Настройка системы

1. Пользователь открывает разделы настроек.
2. Настраивает необходимые параметры.
3. Для настроек, которые являются изменяемыми справочниками-стандартами, пользователь может выбрать из списка нужный вариант, либо создать собственный (см. раздел «Кастомизируемые настройки-стандарты»).
4. Переходит дальше. Все значения настроек применяются только при завершении настройки на последнем шаге, **за исключением** управления вариантами стандартов — создание, изменение и удаление собственного варианта стандарта применяется сразу, в момент действия пользователя, не дожидаясь завершения мастера.

---

## Flow 6. Загрузка логотипа клиники

1. Пользователь на Шаге 2 загружает изображение логотипа.
2. Логотип сохраняется в системе сразу, как только пользователь выбрал файл — это единственный элемент мастера, который не дожидается финального завершения.
3. Если загрузка логотипа не удалась — отображается соответствующая ошибка, но переход к следующему шагу не блокируется, так как логотип необязателен.

---

# Шаг 1. Подключение клиники

## Назначение

Считывание идентифицирующих данных клиники из конфигурационного файла или QR-кода. Полноценное подтверждение подлинности этих данных происходит позже, при завершении всего мастера.

## Элементы интерфейса

### Блок загрузки

Содержит:

* Drag & Drop область
* Кнопку загрузки JSON-файла
* Поддерживаемые форматы файлов

---

### Блок сканирования

Содержит:

* Область камеры
* Кнопку запуска сканирования
* Индикатор успешного считывания QR-кода

---

### Карточка считанных данных

После успешного локального чтения файла/QR отображается карточка с минимальным набором данных, которые можно прочитать без обращения к серверу:

Поля:

* Идентификатор клиники
* Статус считывания (успешно / ошибка формата файла или QR-кода)

> Поля «Название клиники», «Тип лицензии», «Страна» и «Статус подключения» на этом шаге не отображаются — они становятся достоверно известны только после подтверждения подлинности данных, которое происходит при завершении настройки (Шаг 5).

---

# Шаг 2. Администратор

## Назначение

Ввод данных первого администратора системы. Учётная запись фактически создаётся только при завершении настройки.

## Поля

* Логин
* Пароль
* Подтверждение пароля
* Имя
* Фамилия
* Название роли

> **Изменение состава полей:** поля «Email» и «Телефон» временно исключены из формы, так как соответствующая бизнес-логика обработки этих данных при создании администратора не предусмотрена. Если эти поля нужны продукту, требуется отдельно согласовать их добавление в логику инициализации, прежде чем возвращать их в дизайн.

## Дополнительные элементы

* Загрузка логотипа клиники — применяется сразу при выборе файла, а не дожидается завершения всего мастера (см. Flow 6).

---

# Шаг 3. Филиал

## Назначение

Создание первого филиала организации. Филиал фактически создаётся только при завершении настройки.

## Поля

### Основная информация

* Название филиала
* Адрес


### Региональные настройки

* Язык системы
* Часовой пояс

### Рабочий график

Расписание задаётся в виде диапазонов, а не индивидуально для каждого дня недели:

* Количество рабочих дней в неделе
* Диапазон рабочих дней (например, «с понедельника по пятницу»)
* Диапазон рабочих часов (время открытия — время закрытия)

> **Уточнение логики:** ранее предполагалось, что время открытия и закрытия указывается отдельно для каждого дня недели. По актуальной бизнес-логике расписание формируется как единый диапазон дней и единый диапазон часов работы, без индивидуальной настройки каждого дня в рамках этого мастера.

### Перерывы

Возможность добавить один или несколько временных интервалов. Каждый интервал содержит:

* Время начала
* Время окончания

# Шаг 4. Настройки системы

## Назначение

Первичная настройка глобальных параметров системы. Значения настроек применяются при завершении настройки на последнем шаге, **кроме** действий по управлению вариантами стандартов, которые применяются сразу (см. ниже).

---

### Описание

Краткое описание назначения настройки. Пример: Название организации, отображаемое в системе.

Тип зависит от конфигурации настройки.

---

### Ошибка валидации

Отображается под полем.

### Number Input

Поле для ввода чисел.

Переключатель Да / Нет.

Одиночный выбор значения из списка.

## Кастомизируемые настройки-стандарты — новая бизнес-логика

Отдельные настройки в разделах (например, в разделе HR) представляют собой не просто значение, а ссылку на **активный вариант справочника-стандарта**, который клиника может кастомизировать под себя. Примеры таких настроек:

* Классификация пола сотрудника
* Справочник специализаций врачей
* Справочник квалификационных категорий врачей

### Поведение в UI

1. Такая настройка отображается как обычный список выбора (select) с текущим выбранным активным вариантом.
2. Рядом с полем дополнительно отображается элемент «Управление вариантами» — он появляется только у настроек, которые поддерживают кастомизацию.
3. По нажатию открывается окно управления вариантами справочника:
   * Список существующих вариантов с пометкой происхождения — «Системный» (поставляется вместе с платформой) или «Кастомный» (создан клиникой).
   * Системные варианты доступны только для просмотра: их нельзя редактировать или удалять.
   * Кастомные варианты можно редактировать и удалять.
   * Кнопка «Создать новый вариант» — открывает форму с полями: название варианта, описание (опционально), таблица значений (пары «отображаемое значение» / «код значения», с возможностью задать порядок отображения).
   * Удалить вариант, который сейчас выбран как активный, нельзя — пользователю нужно сначала переключиться на другой вариант.
4. Выбор активного варианта в списке — это обычное изменение настройки, которое применяется вместе со всеми остальными настройками при завершении мастера.
5. Создание, изменение и удаление кастомного варианта — это действия, которые применяются **сразу**, в момент их выполнения пользователем, независимо от того, на каком шаге мастера сейчас находится пользователь, и не дожидаются финального завершения настройки.

> Это значительно расширяет бизнес-логику Шага 4 по сравнению с предыдущей версией дизайна, где раздел настроек описывал только статичные типы полей без концепции управляемых клиникой справочников.

---

## Правила отображения

* Интерфейс должен поддерживать неограниченное количество разделов.
* Интерфейс должен поддерживать неограниченное количество настроек.
* Header, Stepper и Footer остаются фиксированными.
* Прокручивается только область контента.
* Изменение значения происходит без перезагрузки страницы.
* Каждая настройка отображает название, описание и текущее значение.
* Для настроек-стандартов дополнительно отображается элемент управления вариантами (см. выше).

---

# Шаг 5. Завершение

## Назначение

Финальный шаг, на котором происходит фактическое применение всех данных, введённых пользователем на предыдущих шагах: подключение клиники, создание администратора, создание филиала и сохранение настроек.

## Экран ожидания

В момент нажатия «Завершить настройку» отображается индикатор процесса, информирующий пользователя о происходящем (без необходимости точного прогресс-бара по процентам), например, в виде последовательности этапов:

1. Подтверждение клиники.
2. Создание администратора.
3. Создание филиала.
4. Сохранение настроек.

На этом экране элементы формы заблокированы, повторная отправка недоступна.

## Экран подтверждения (Success)

Отображается итоговая информация:

* Клиника подключена
* Администратор создан
* Филиал создан
* Настройки сохранены

## Экран частичного успеха

> Новое состояние, отсутствовавшее в прежней версии документа: возможна ситуация, когда часть данных применена успешно, а часть — нет (частичный успех). В этом случае отображается предупреждающий (не критичный) баннер с описанием, какие шаги выполнены успешно, а какие требуют ручной проверки администратором после первого входа в систему. Переход к авторизации остаётся доступным.

## Кнопка «Завершить настройку»

Активна только при отсутствии ошибок валидации на всех предыдущих шагах. После успешного завершения (включая частичный успех) превращается в кнопку перехода на страницу авторизации.

---

# Состояния

## Empty

* Данные отсутствуют.
* Следующий шаг недоступен.

---

## Loading

* Выполняется операция (загрузка логотипа, изменение варианта стандарта, либо завершение настройки на последнем шаге).
* Элементы формы заблокированы.
* Отображается индикатор выполнения.

---

## Error

Возможные причины ошибок, на которые должен реагировать интерфейс:

* Ошибка считывания файла или QR-кода клиники.
* Ошибка подтверждения подлинности данных клиники при завершении настройки — пользователь возвращается к Шагу 1 с понятным сообщением и может загрузить файл/код заново.
* Ошибка проверки данных (валидация полей).
* Ошибка создания администратора или филиала при завершении настройки — данные не сохраняются, пользователь может повторить попытку завершения.
* Ошибка загрузки логотипа — не блокирует переход дальше.
* Ошибки при создании, изменении или удалении кастомного варианта стандарта:
  * попытка создать вариант с уже существующим названием/идентификатором;
  * некорректные или неполные данные варианта;
  * попытка удалить системный вариант или вариант, который сейчас активен.
* Ошибка, если страница инициализации открыта повторно после того, как система уже была настроена — пользователь должен быть перенаправлен на страницу авторизации, мастер не показывается.

Сообщение должно содержать понятное описание проблемы.

---

## Success

* Данные успешно сохранены.
* Пользователь получает уведомление.

---

## Offline

* Отсутствует соединение с сервером.
* Отображается предупреждение.
* Отправка данных недоступна.

---

# Interaction & Animation

* Переключение между шагами происходит плавно.
* Активный шаг визуально выделяется.
* Ошибки отображаются без смещения интерфейса.
* Загрузка сопровождается индикатором прогресса.
* Завершение настройки сопровождается уведомлением об успехе (или предупреждением при частичном успехе).
* Все переходы между состояниями должны быть плавными и предсказуемыми.

---

# Acceptance Criteria

* Пользователь может пройти все этапы настройки в рамках одного мастера; данные шагов 1–4 не применяются по отдельности — окончательное применение происходит только при завершении всего мастера.
* Поддерживается загрузка JSON-файла и сканирование QR-кода для считывания данных клиники, с полноценным подтверждением подлинности только на финальном шаге.
* Логотип клиники применяется сразу при загрузке, не дожидаясь завершения мастера.
* Можно создать первого администратора с полями: логин, пароль, подтверждение пароля, имя, фамилия, название роли (без email и телефона).
* Можно создать первый филиал с расписанием в виде диапазонов дней/часов и отдельным списком перерывов.
* Можно настроить системные параметры, включая выбор активного варианта справочника-стандарта и управление кастомными вариантами (создание/редактирование/удаление) для настроек, которые это поддерживают.
* Управление кастомными вариантами стандартов применяется немедленно, не дожидаясь завершения мастера.
* Все обязательные поля проходят валидацию.
* Все состояния (включая ошибки подтверждения клиники, частичный успех, ошибки управления вариантами) отображаются корректно.
* После успешного завершения настройки (включая частичный успех) пользователь перенаправляется на страницу авторизации.
* Если система уже была настроена ранее, страница мастера не отображается — выполняется перенаправление на страницу авторизации.
