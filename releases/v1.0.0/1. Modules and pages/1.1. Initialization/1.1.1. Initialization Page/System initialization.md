# 1.1.1. System Initialization Page

## Спецификация страницы интерфейса

* Название страницы: Инициализация системы
* Модуль: Management
* Тип страницы: Первичная настройка клиники
* Основные роли: Администратор клиники, Оператор клиники
* Статус документа: Draft
* Платформа: веб-приложение для медицинской системы

## Назначение страницы и сценарии использования

Страница предназначена для первоначальной настройки системы после установки.

С помощью страницы пользователь может:

1. Интегрировать клинику на основе конфигурационных данных.
2. Создать первого администратора.
3. Настроить первый филиал.
4. Заполнить основные системные настройки.
5. Завершить активацию системы.

Сценарии использования:

1. Пользователь открывает страницу инициализации.
2. Загружает конфигурационный файл или сканирует QR-код.
3. Заполняет данные администратора.
4. Настраивает первый филиал.
5. Указывает необходимые настройки системы.
6. Запускает процесс инициализации.
7. После успешного завершения получает доступ к работе в системе.

## User Flows / Task Flows для страницы

### Flow 1. Полная настройка системы

1. Пользователь открывает страницу.
2. Система отображает пошаговую форму настройки.
3. Пользователь проходит все этапы настройки.
4. Нажимает кнопку «Завершить настройку».
5. Система выполняет процесс инициализации.
6. Пользователь получает подтверждение успешной настройки.

### Flow 2. Подключение клиники

1. Пользователь загружает JSON-файл или сканирует QR-код.
2. Система считывает данные клиники.
3. Отображает информацию о найденной клинике.
4. Пользователь подтверждает переход к следующему шагу.

### Flow 3. Создание администратора

1. Пользователь вводит данные администратора.
2. Заполняет название роли.
3. При необходимости загружает логотип клиники.
4. Переходит к следующему этапу.

### Flow 4. Настройка филиала

1. Пользователь заполняет информацию о филиале.
2. Указывает адрес и местоположение на карте.
3. Настраивает рабочие дни и расписание.
4. Подтверждает данные.

### Flow 5. Настройка системы

1. Пользователь просматривает доступные настройки.
2. Изменяет необходимые параметры.
3. Сохраняет настройки.
4. Завершает процесс инициализации.

## Структура: Wireframes

Страница представляет собой многошаговую форму (Stepper).

Основной layout:

* Header с названием страницы и описанием.
* Stepper со списком этапов настройки.
* Центральная область с формой текущего шага.
* Footer с кнопками управления.

### Шаг 1. Подключение клиники

Элементы:

* Загрузка JSON-файла.
* Сканер QR-кода.
* Карточка информации о клинике.
* Индикатор успешной проверки данных.

### Шаг 2. Администратор

Элементы:

* Поле логина.
* Поле пароля.
* Поле имени.
* Поле фамилии.
* Поле названия роли.
* Загрузка логотипа.

### Шаг 3. Филиал

Элементы:

* Название филиала.
* Страна.
* Город.
* Адрес.
* Выбор координат на карте.
* Часовой пояс.
* Язык системы.
* Настройка рабочего графика.
* Настройка перерывов.

### Шаг 4. Настройки системы

Назначение:

Первичная настройка параметров системы.

Отображение:

- Настройки группируются по разделам.
- Каждый раздел отображается отдельным блоком, вкладкой или аккордеоном.
- Состав разделов заранее неизвестен и формируется динамически.
- Количество настроек внутри раздела не ограничено.

Поддерживаемые элементы:

- Текстовое поле.
- Числовое поле.
- Переключатель.
- Выпадающий список.
- Выбор даты.
- Выбор времени.
- Другие типы компонентов, предусмотренные системой настроек.

Требования к интерфейсу:

- Интерфейс должен поддерживать любое количество разделов.
- Интерфейс должен поддерживать любое количество настроек внутри раздела.
- Длинные списки настроек должны корректно прокручиваться.
- Изменение значения должно происходить без перезагрузки страницы.
- Каждая настройка должна отображать название, описание и текущее значение.

## Визуал: High-Fidelity mockups

Визуальные правила:

1. Интерфейс должен выглядеть как мастер первоначальной настройки системы.
2. Каждый шаг визуально отделён от остальных.
3. Пользователь всегда видит текущий этап и общий прогресс.
4. Все обязательные поля должны быть явно обозначены.
5. Ошибки отображаются рядом с соответствующими полями.
6. Кнопка завершения является основным CTA-элементом страницы.
7. Состояние выполнения отображается через индикатор прогресса.
8. Сообщения об успехе и ошибках визуально различаются.
9. Интерфейс должен быть адаптирован под большие формы.
10. Все элементы должны следовать единой дизайн-системе продукта.

## Состояния и edge cases

### error

* Ошибка проверки данных клиники.
* Ошибка заполнения обязательных полей.
* Ошибка выполнения инициализации.
* Пользователь получает понятное описание проблемы.

### empty

* Данные ещё не заполнены.
* Кнопка завершения недоступна.
* Обязательные поля отмечены визуально.

### loading

* Выполняется процесс настройки.
* Все элементы формы заблокированы.
* Отображается индикатор выполнения.

### offline

* Отсутствует подключение к интернету.
* Пользователь получает уведомление.
* Отправка данных недоступна.

### permissions

* Страница доступна только до завершения первоначальной настройки системы.
* После успешной инициализации пользователь перенаправляется на страницу авторизации.

## Interaction & Animation

Правила поведения:

* Переключение между шагами происходит плавно.
* Активный шаг выделяется визуально.
* Ошибки отображаются без смещения макета.
* Загрузка сопровождается индикатором прогресса.
* Успешное завершение сопровождается уведомлением.
* Все переходы между состояниями должны быть плавными и предсказуемыми.

## Acceptance Criteria

* Пользователь может пройти все этапы настройки на одной странице.
* Поддерживается загрузка JSON-файла и сканирование QR-кода.
* Можно создать первого администратора.
* Можно настроить первый филиал.
* Можно изменить системные настройки.
* Процесс настройки запускается после заполнения обязательных полей.
* Все состояния отображаются корректно.
* После успешного завершения пользователь может начать работу в системе.
