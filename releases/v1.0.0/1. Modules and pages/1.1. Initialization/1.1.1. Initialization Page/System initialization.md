# 1.1.1. System Initialization Page

## Спецификация страницы интерфейса

- Название страницы: Инициализация системы
- Модуль: Инициализация / Setup
- Тип страницы: Первичное подключение и настройка клиники
- Основные роли: оператор клиники, администратор клиники
- Статус документа: Draft
- Платформа: веб-приложение для управления медицинской системой

## Назначение страницы и сценарии использования

Страница предназначена для первого запуска и первичной настройки системы в клинике.
Инициализация выполняется единым запросом, проходящим четыре последовательных этапа:
провизионный шаг, создание администратора, создание филиала и установка глобальных настроек.

Сценарии использования:

1. Клиника получает провизионный JSON-файл или QR-код с `token` и `clinicId`.
2. Оператор запускает страницу и загружает данные провизирования.
3. Система выполняет проверку провизионного токена, получает тип филиала.
4. Оператор вводит данные первого администратора.
5. Система создаёт администратора, роль и филиал, затем применяет глобальные настройки.
6. При успешной инициализации клиника переходит в рабочий режим.

## User Flows / Task Flows для страницы

### Flow 1. Полный запуск и инициализация системы

1. Пользователь открывает страницу "Инициализация системы".
2. Система отображает шаги и форму для запуска инициализации.
3. Оператор загружает провизионный JSON-файл или сканирует QR-код.
4. Система извлекает `provisioning.token` и `provisioning.clinicId`.
5. Пользователь заполняет данные первого администратора и настройки филиала.
6. Нажимает кнопку "Запустить инициализацию".
7. Система отправляет единый запрос на API, включающий данные всех четырех этапов.
8. Сервер выполняет провизионный шаг, создание администратора, создание филиала и изменение глобальных настроек.
9. При успешном завершении пользователь получает подтверждение и доступ к дальнейшей настройке.

### Flow 2. Провизионный шаг и получение PFX

1. Пользователь загружает провизионный JSON-файл или QR-код.
2. Система формирует тело запроса с `provisioning.token` и `provisioning.clinicId`.
3. Сервер компании Synapse проверяет данные.
4. При успехе сервер возвращает статус 201.
5. При ошибке сервер возвращает `PROVISIONING_TOKEN_INVALID`.

### Flow 3. Создание администратора и роли

1. Пользователь вводит данные администратора: имя, email, логин, пароль, название роли.
2. ManagementModule вызывает AuthModule через DI.
3. Создаётся пользователь с привязанным `role_id`.
4. При успехе процесс переходит к следующему шагу.

### Flow 4. Инициализация первого филиала

1. После успешного создания администратора ManagementModule вызывает StructureModule.
2. Тип филиала выбирается на основании ответа этапа 1 (основной или дочерний).
3. В главном филиале автоматически создаются стандартные типы отделов.
4. Создаётся отдел "Администрация" первого уровня.

### Flow 5. Применение глобальных настроек

1. После создания филиала система предлагает настройки поведения при удалении отдела и должности.
2. Оператор выбирает политики переноса сотрудников и должностей.
3. Система сохраняет глобальные параметры конфигурации.
4. Инициализация считается завершённой.

## Структура: Wireframes

Страница должна быть разделена на четыре шага (steps):

1. Провизионный шаг — загрузка токена и клиники.
2. Создание администратора и роли.
3. Инициализация первого филиала.
4. Глобальные настройки.

Основной layout:

- Верхняя панель с названием страницы и кратким описанием.
- Блок шагов слева или сверху с явной последовательностью.
- Основная область содержит форму для текущего шага.
- Блок статуса выполнения справа или внизу.
- Кнопка "Запустить инициализацию" доступна при заполнении всех обязательных данных.

Элементы wireframe:

- Шаг 1: поле загрузки JSON / QR-сканер, отображение `token` и `clinicId`.
- Шаг 2: поля администратора и название роли.
- Шаг 3: поля филиала, адрес, координаты, расписание.
- Шаг 4: политики удаления отдела и должности.
- Панель прогресса или индикатор шагов.
- Зону вывода ошибок и подсказок.

## Визуал: High-Fidelity mockups

Визуальные правила:

1. Использовать чистый современный интерфейс с минимумом отвлекающих элементов.
2. Страница должна выглядеть как профессиональное рабочее пространство для настройки.
3. Шаги визуально разделены карточками или вкладками.
4. Текущий активный шаг должен быть выделен.
5. Валидация полей и ошибки показываются в контексте каждого шага.
6. Кнопка отправки выглядит как primary action.
7. Состояние загрузки отображается анимацией на кнопке.
8. Информационные сообщения отображаются мягким фоном и читаемым шрифтом.
9. Компонент QR-сканера оформлен как отдельная область с инструкцией.
10. Сообщения об успешной и неуспешной инициализации имеют различный цвет.

Визуальные компоненты:

- Шаги / шаговый навигационный компонент.
- Форма шага с полями и подсказками.
- Панель статуса запроса.
- Блок отображения результата провизии и типа филиала.
- Баннер ошибок и подтверждений.

## Состояния и edge cases

### error

- `PROVISIONING_TOKEN_INVALID` при неверном токене или `clinicId`.
- Ошибка создания пользователя или роли.
- Ошибка создания филиала.
- Ошибка сохранения глобальных настроек.
- На странице отображается понятное сообщение и предложение повторить попытку.

### empty

- Если ни один шаг не заполнен, кнопка отправки неактивна.
- Поля обязательных данных отмечены звездочкой и подсвечены при пустоте.
- При пустом провизионном токене выводится подсказка "Загрузите JSON-файл или сканируйте QR-код".

### loading

- При отправке единого запроса кнопка "Запустить инициализацию" переходит в состояние загрузки.
- Поля блокируются, пока запрос выполняется.
- Индикатор шага показывает прогресс выполнения этапов.

### offline

- Если отсутствует сеть, отображается баннер "Нет подключения к сети".
- Инициализация не может быть запущена.
- Пользователь может повторить попытку после восстановления связи.

### permissions

- Страница доступна только неавторизованным пользователям или пользователям с правом инициализации.
- Авторизованный пользователь перенаправляется на рабочую страницу или получает сообщение о недостаточности прав.

## Interaction & Animation

Правила поведения:

- Переход между шагами должен быть плавным и понятным.
- При переключении шага активный шаг выделяется с плавной анимацией.
- Кнопка "Запустить инициализацию" должна менять состояние при наведении и нажатии.
- Ошибки появляются плавно, без резких изменений макета.
- Индикация загрузки шага отображается через spinner или progress bar.
- При вводе данных поля мягко подсвечиваются.
- Сообщения об успешном результате появляются с subtle fade-in.

## Зависимости и ограничения

- API:
  - POST /system/init — единый запрос инициализации, содержащий все этапы.
  - POST /system/provisioning/validate — проверка провизионного токена (в рамках единого запроса).
  - POST /auth/users — создание администратора и роли через AuthModule.
  - POST /structure/branches — создание филиала через StructureModule.
  - PATCH /settings/global — запись глобальных параметров.
- Роли:
  - доступ к странице имеет пользователь с правом инициализации или неавторизованный пользователь при первом запуске.
  - при отсутствии прав возвращается ошибка доступа.
- Feature flags:
  - `system_init_enabled` для включения процесса инициализации.
- Технические ограничения:
  - Инициализация должна выполняться одним единственным запросом.
  - Доступность: форма и шаги должны работать на разных разрешениях и устройствах.
  - Безопасность: провизионный токен передаётся только в теле запроса, PFX сохраняется в защищённом хранилище.

## Acceptance criteria

- Страница отображает четыре шага и организует процесс инициализации.
- Шаг 1 принимает `token` и `clinicId` из провизионного JSON или QR-кода.
- При запуске выполняется единый запрос, включающий все этапы.
- Провизионный шаг проверяет токен и клинику, возвращает PFX и тип филиала.
- При неверном токене отображается ошибка `PROVISIONING_TOKEN_INVALID`.
- Создаётся первый администратор и роль с правами `*_MANAGE`.
- Создаётся главный филиал и отдел "Администрация" первого уровня.
- Применяются глобальные настройки политики удаления отделов и должностей.
- Состояния error, empty, loading, offline и permissions обработаны.
- Интерфейс ведёт себя плавно и понятно при переходе между шагами.
