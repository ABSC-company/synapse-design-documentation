## 1.1 Реестр сотрудников

### Назначение страницы

Список всех сотрудников клиники с возможностью поиска, фильтрации и перехода в профиль. Точка входа для приёма нового сотрудника.

### User Flows

**Flow 1. Просмотр и поиск сотрудников**
1. Пользователь открывает «Сотрудники» в боковом меню.
2. Система показывает список сотрудников текущего филиала (или всех, если есть право на просмотр нескольких филиалов).
3. Пользователь вводит запрос в поиск → список фильтруется по имени/фамилии.
4. Пользователь кликает на строку → открывается профиль сотрудника.

**Flow 2. Фильтрация списка**
1. Пользователь нажимает «Фильтр».
2. Открывается панель (Driver) с параметрами.
3. Пользователь выбирает значения → нажимает «Применить».
4. Список обновляется, на кнопке «Фильтр» отображается счётчик активных фильтров.

**Flow 3. Приём нового сотрудника**
1. Пользователь нажимает «+ Принять сотрудника» в шапке.
2. Открывается форма создания сотрудника (см. 1.1.4).
3. Пользователь заполняет обязательные поля, сохраняет.
4. Система создаёт сотрудника со статусом `at_work`, перенаправляет в его профиль на вкладку «Должности» с подсказкой назначить должность.

### 1.1.1 Header

```
┌─────────────────────────────────────────────────────────────────┐
│  Сотрудники                              [+ Принять сотрудника] │
└─────────────────────────────────────────────────────────────────┘
```

- Заголовок слева, кнопка действия справа.
- Кнопка видна только при наличии права `HR_EMPLOYEE_CREATE`.

### 1.1.2 Панель управления списком

```
┌─────────────────────────────────────────────────────────────────┐
│  [Все] [На работе] [Вне работы] [В отпуске] [В командировке]     │
│        [Уволенные →]  ← ведёт на отдельную страницу архива       │
├─────────────────────────────────────────────────────────────────┤
│  🔍 Поиск по имени, фамилии...           [Фильтр (2) ▾]         │
└─────────────────────────────────────────────────────────────────┘
```

- Вкладки фильтруют по статусу сотрудника (раздел 6 тех. задания): `at_work`, `away`, `on_leave`, `on_trip`. Статус `dismissed` НЕ отображается в этих вкладках — для уволенных есть отдельная ссылка «Уволенные →», ведущая в раздел архива (см. `05-dismissal-and-archive.md`), так как уволенные сотрудники физически переносятся в архивную таблицу.
- Поиск — по `first_name`, `second_name`, `third_name`.
- Кнопка «Фильтр» показывает количество активных фильтров бейджем.

### 1.1.4 Карточка/строка сотрудника в списке

Список отображается как таблица (рекомендуется) либо карточки — решение дизайнера, но обязательный набор данных в строке:

```
┌────┬─────────────────────┬───────────────┬──────────────┬──────────┬────┐
│Фото│  ФИО                │  Должность    │  Отдел       │  Статус  │ ⋮  │
├────┼─────────────────────┼───────────────┼──────────────┼──────────┼────┤
│ 🖼 │ Иванов Иван Иванович│  Хирург       │  Хирургия    │ ● На раб.│ ⋮  │
└────┴─────────────────────┴───────────────┴──────────────┴──────────┴────┘
```

- Фото — круглый аватар (placeholder-иконка, если фото не загружено).
- Должность — основная/первая из назначенных, если несколько (см. 1.2.2).
- Статус — бейдж по цветовой семантике (см. `00-overview.md`).
- Контекстное меню `⋮`: «Открыть профиль», «Изменить статус», «Уволить» (последнее — только при `HR_ARCHIVE_MANAGE`).
- Клик по строке (не по `⋮`) — переход в профиль.

### 1.1.5 Форма создания сотрудника

Модальное окно или отдельная страница (рекомендуется модальное окно по аналогии с остальными формами модуля, так как все обязательные поля помещаются в один экран).

```
┌──────────────────────────────────────────┐
│  Принять сотрудника                  [✕] │
├──────────────────────────────────────────┤
│                                          │
│  Фото                                    │
│  [Загрузить файл]  ← паттерн File Module │
│                                          │
│  Имя *           Фамилия *               │
│  ┌─────────────┐ ┌─────────────┐         │
│  └─────────────┘ └─────────────┘         │
│  Отчество                                │
│  ┌──────────────────────────────────┐    │
│  └──────────────────────────────────┘    │
│                                          │
│  Системный пользователь *                │
│  ┌──────────────────────────────────┐    │
│  │ Выберите пользователя ▾          │    │  ← справочник Auth Module
│  └──────────────────────────────────┘    │
│                                          │
│  Филиал *                                │
│  ┌──────────────────────────────────┐    │
│  │ Выберите филиал ▾                │    │  ← справочник Structure Module
│  └──────────────────────────────────┘    │
│                                          │
│  Тип занятости *                         │
│  ○ Сотрудник   ○ Врач                    │
│                                          │
│  Дата рождения *      Гражданство *      │
│  ┌─────────────┐      ┌─────────────┐    │
│  └─────────────┘      └─────────────┘    │
│                                          │
│  Пол *                                   │
│  [значения зависят от глобального        │
│   стандарта клиники — см. ниже]          │
│                                          │
│  Страна проживания *   Регион *          │
│  ┌─────────────┐       ┌─────────────┐   │
│  └─────────────┘       └─────────────┘   │
│  Адрес                                   │
│  ┌──────────────────────────────────┐    │
│  └──────────────────────────────────┘    │
│                                          │
│  MRZ-код паспорта      Серия/номер       │
│  ┌─────────────┐       ┌─────────────┐   │
│  └─────────────┘       └─────────────┘   │
│                                          │
├──────────────────────────────────────────┤
│              [Отмена]      [Создать]     │
└──────────────────────────────────────────┘
```

**Примечания для дизайнера:**

- Поле «Пол» — рендерится по-разному в зависимости от глобальной настройки клиники (стандарт `ISO/IEC 5218` — обычно 4 значения: не указано/мужской/женский/иное, либо `simplified` — 2 значения мужской/женский). Дизайнеру нужно предусмотреть вариант select с 2 и с 4 опциями, не хардкодить количество.
- «MRZ-код паспорта» и «Серия/номер удостоверения» — необязательные технические поля, можно скрыть под «Показать дополнительные поля» / разворачивающийся блок, чтобы не перегружать форму при первом приёме.
- После создания сотрудника поля филиала/типа занятости неизменяемы через эту форму (для типа занятости — изменение зацепит вкладки специализаций/допусков, обсуждается отдельно, в MVP считаем неизменяемым после создания).
- Поле «Системный пользователь» — обязательно, привязка к Auth Module. Если пользователь без аккаунта — нужно сначала создать аккаунт (вне HR-модуля), форма должна явно подсказывать это в пустом состоянии селекта («Нет доступных пользователей. Создайте аккаунт в разделе Пользователи»).

## 1.2 Профиль сотрудника

### Назначение страницы

Единая карточка сотрудника со всей информацией. Точка управления статусом, должностями, контактами, рабочими местами, графиком и (для врачей) медицинскими данными.

### Layout

```
┌───────────────────────────────────────────────────────────────────┐
│  ← Назад к списку                                                  │
├───────────────────────────────────────────────────────────────────┤
│  ШАПКА ПРОФИЛЯ                                                      │
│  🖼   Иванов Иван Иванович                    ● На рабочем месте ▾  │
│       Хирург · Хирургия · Филиал на Юнусобод               [⋮]    │
├───────────────────────────────────────────────────────────────────┤
│  [Основное] [Должности] [Контакты] [Рабочие места] [График]        │
│  [Специализации]* [Допуски]* [Аттестации] [Документы]               │
│  [Инструктажи] [Учёт времени]                                       │
├───────────────────────────────────────────────────────────────────┤
│                                                                     │
│              содержимое активной вкладки                           │
│                                                                     │
└───────────────────────────────────────────────────────────────────┘
```

### 1.2.1 Шапка профиля

- Фото (крупный аватар, placeholder если нет).
- ФИО.
- Под именем — основная должность + отдел + филиал (через точку-разделитель). Если должностей несколько, показывается основная (is_primary рабочего места задаёт контекст, либо первая назначенная должность — уточнить с продуктом при реализации; для дизайна предусмотреть один read-only текст).
- Статус — кликабельный бейдж-дропдаун: клик открывает список допустимых переходов (см. 1.2.1.1).
- `⋮` — контекстное меню: «Редактировать данные», «Уволить» (см. `05-dismissal-and-archive.md`).

#### 1.2.1.1 Изменение статуса

При клике на статус-бейдж открывается мини-дропдаун со списком допустимых переходов (не все статусы — только разрешённые из текущего, согласно матрице переходов тех. задания п.6.2):

```
Текущий статус: ● На рабочем месте

┌──────────────────────────┐
│  Изменить статус на:      │
│  ○ Вне рабочего места     │
│  ○ В отпуске              │
│  ○ В командировке         │
│  ───────────────────────  │
│  ○ Уволить →               │  ← ведёт в отдельный флоу увольнения
└──────────────────────────┘
```

- Если текущий статус `away` / `on_leave` / `on_trip` — единственный доступный переход: «На рабочем месте».
- Пункт «Уволить» визуально отделён (разделитель) и оформлен как отдельное необратимое действие — открывает флоу увольнения (`05-dismissal-and-archive.md`), а не просто меняет статус инлайн.
- После выбора простого перехода (away/on_leave/on_trip/at_work) — подтверждение не требуется, статус меняется сразу, бейдж обновляется, появляется toast «Статус обновлён».

### 1.2.2 Вкладка «Основное»

Просмотр и редактирование персональных данных сотрудника (поля `employees`, кроме служебных).

```
┌─────────────────────────────────────────────────────────┐
│  Персональные данные                  [Редактировать]    │
├─────────────────────────────────────────────────────────┤
│  Имя              Иван                                    │
│  Фамилия          Иванов                                  │
│  Отчество         Иванович                                │
│  Дата рождения    15.03.1985                               │
│  Гражданство      Узбекистан                                │
│  Пол              Мужской                                  │
│  Страна проживания Узбекистан                              │
│  Регион            Ташкент                                  │
│  Адрес             ул. Examplе, 12                          │
│  Тип занятости      Врач                                    │
│  Филиал             Филиал на Юнусобод                      │
│                                                            │
│  Документы личности (служебно)                            │
│  MRZ-код            P<UZB...                                │
│  Серия/номер        AB1234567                                │
└─────────────────────────────────────────────────────────┘
```

- Кнопка «Редактировать» открывает ту же форму, что и создание (1.1.5), без полей «Системный пользователь» и «Филиал» (неизменяемые после создания, см. примечание выше) — предзаполненную текущими значениями.
- Доступна только при `HR_EMPLOYEE_UPDATE`.

### 1.2.3 Вкладка «Должности»

Управление текущими должностями сотрудника. Несколько должностей у одного сотрудника допустимо (с разными ставками).

```
┌─────────────────────────────────────────────────────┐
│  Должности                       [+ Назначить должность]│
├─────────────────────────────────────────────────────┤
│  Хирург · Отдел хирургии              Ставка: 1.0   │
│                                          [✏] [✕]      │
├─────────────────────────────────────────────────────┤
│  Консультант · Отдел терапии          Ставка: 0.5   │
│                                          [✏] [✕]      │
└─────────────────────────────────────────────────────┘
```

- `[✏]` открывает форму редактирования ставки (только ставка изменяема, должность для смены — снять+назначить новую, не редактирование).
- `[✕]` — снятие должности, с подтверждающим модальным окном: «Снять должность «Хирург»? Это освободит ставку в штатном расписании.» Это кадровое действие (категория `transfer`/`dismissal` по смыслу) — после подтверждения снятия показывается блок «Переход в ЭДО» (см. `00-overview.md`) с возможностью сразу создать приказ о переводе/снятии, либо пропустить.

**Форма «Назначить должность»:**

```
┌────────────────────────────────────────┐
│  Назначить должность                [✕]│
├────────────────────────────────────────┤
│  Должность *                            │
│  ┌──────────────────────────────────┐   │
│  │ Выберите должность ▾             │   │  ← Structure Module; показывает
│  └──────────────────────────────────┘   │     отдел и доступную ставку рядом
│  Доступно ставок: 0.5 из 1.0             │  ← подсказка из staffing_schedule
│                                          │
│  Рабочая ставка *                        │
│  ○ 1.0   ○ 0.5   ○ 0.25                  │  ← недопустимые при нехватке —
│                                          │     disabled с подсказкой
│  ┌──────────────────────────────────┐   │
│  │  Кадровый документ                │   │
│  │  Назначение на должность          │   │
│  │  обычно сопровождается приказом   │   │
│  │  о приёме/переводе.               │   │
│  │  [Создать документ в ЭДО →]       │   │
│  └──────────────────────────────────┘   │
├────────────────────────────────────────┤
│              [Отмена]      [Назначить]  │
└────────────────────────────────────────┘
```

- Если выбранная должность имеет базовую роль (`base_role_id` в `staffing_schedule`), при назначении эта роль автоматически выдаётся сотруднику — для дизайнера это инфо-строка под выбором ставки: «При назначении сотруднику будет выдана роль: [название роли]» (если задана).
- Опции ставки, превышающие доступный остаток, показываются неактивными.

│  ○ Телефон   ○ Email   ○ Соц. сеть      │

### 1.2.5 Вкладка «Рабочие места»

```
┌─────────────────────────────────────────────────────┐
│  Рабочие места                    [+ Назначить рабочее место]│
├─────────────────────────────────────────────────────┤
│  Кабинет 204, Хирургия           ★ Основное   [✏][✕]│
├─────────────────────────────────────────────────────┤
│  Кабинет 110, Терапия                          [✏][✕]│
└─────────────────────────────────────────────────────┘
```

- Звезда `★ Основное` — бейдж у рабочего места с `is_primary = true`. У сотрудника может быть только одно основное рабочее место.
- `[✏]` на неосновном месте показывает опцию «Сделать основным» — при выборе предыдущее основное автоматически снимает отметку (показать это пользователю явно: «Кабинет 204 будет снят с пометки «Основное»» в подтверждении или инлайн-подсказке).
- `[✕]` — снятие рабочего места, простое подтверждение без ЭДО (это не кадровый приказ).

**Форма «Назначить рабочее место»:**

```
┌────────────────────────────────────────┐
│  Назначить рабочее место            [✕]│
├────────────────────────────────────────┤
│  Рабочее место *                         │
│  ┌──────────────────────────────────┐   │
│  │ Выберите рабочее место ▾          │   │  ← Structure Module
│  └──────────────────────────────────┘   │
│                                          │
│  [ ] Сделать основным                    │
│  Если отмечено и у сотрудника уже есть   │
│  основное место — оно будет снято.       │
├────────────────────────────────────────┤
│              [Отмена]      [Назначить]  │
└────────────────────────────────────────┘
```

### 1.2.6 Вкладка «График»

Рабочий график сотрудника: дни недели, рабочие промежутки времени, перерывы.

```
┌───────────────────────────────────────────────────────────┐
│  График работы                              [Редактировать]│
├───────────────────────────────────────────────────────────┤
│         Пн      Вт      Ср      Чт      Пт      Сб   Вс    │
│  ───────────────────────────────────────────────────────   │
│  Раб.    ✓       ✓       ✓       ✓       ✓        –    –   │
│  Время  09:00-  09:00-  09:00-  09:00-  09:00-               │
│         18:00   18:00   18:00   18:00   18:00               │
│  Перерыв 13:00-  13:00-  13:00-  13:00-  13:00-               │
│          14:00   14:00   14:00   14:00   14:00               │
└───────────────────────────────────────────────────────────┘
```

- Если график не задан — пустое состояние: «График не установлен» + кнопка «Установить график».
- Визуально рекомендуется недельная таблица (как на скетче выше), а не список интервалов, чтобы график читался наглядно.

**Форма «Редактировать график»:**

```
┌────────────────────────────────────────────────────┐
│  График работы                                  [✕]│
├────────────────────────────────────────────────────┤
│  Рабочие дни *                                       │
│  [Пн✓] [Вт✓] [Ср✓] [Чт✓] [Пт✓] [Сб ] [Вс ]            │  ← toggle-кнопки
│                                                      │
│  Рабочие промежутки                                  │
│  ┌──────────┐ – ┌──────────┐         [+ Добавить]   │
│  │ 09:00    │   │ 18:00    │              [✕]       │
│  └──────────┘   └──────────┘                         │
│                                                      │
│  Перерывы                                             │
│  ┌──────────┐ – ┌──────────┐         [+ Добавить]   │
│  │ 13:00    │   │ 14:00    │              [✕]       │
│  └──────────┘   └──────────┘                         │
│                                                      │
├────────────────────────────────────────────────────┤
│                    [Отмена]      [Сохранить]        │
└────────────────────────────────────────────────────┘
```

- Рабочие дни — multi-toggle (Пн–Вс), выбранные подсвечены.
- Рабочие промежутки и перерывы — динамические списки пар time-picker'ов, можно добавлять несколько интервалов (массив в БД — `timestamp[][]`).
- Время перерыва должно визуально попадать внутрь рабочего промежутка — валидация на уровне формы (если перерыв выходит за рабочие рамки — инлайн-ошибка).

| Состояние | Поведение |
|-----------|-----------|
| Empty | Для каждой вкладки — свой текст пустого состояния («Должности не назначены», «Контакты не добавлены» и т.д.) + кнопка создания |

## Acceptance Criteria

- Реестр сотрудников фильтруется вкладками по статусу (без `dismissed`) и панелью фильтров (филиал, отдел, должность, тип занятости, гражданство).
- Поиск работает по имени/фамилии/отчеству.
- Ссылка «Уволенные →» ведёт в отдельный раздел архива, а не в общий список с фильтром.
- Форма приёма сотрудника содержит все обязательные поля из `employees`, включая привязку к системному пользователю и филиалу.
- Поле «Пол» рендерится с количеством опций, зависящим от глобальной настройки гендерного стандарта клиники.
- Профиль сотрудника — одна страница с вкладками, шапка показывает фото, ФИО, текущую должность/отдел/филиал, статус.
- Клик на статус-бейдж показывает только допустимые переходы согласно матрице статусов; «Уволить» визуально отделён как необратимое действие.
- Вкладка «Должности» позволяет назначать/снимать должности, показывает доступный остаток ставки при назначении и подсказку о базовой роли, если она задана у должности.
- Вкладка «Контакты» валидирует значение в зависимости от выбранного типа контакта.
- Вкладка «Рабочие места» не допускает двух основных мест одновременно; смена основного места явно предупреждает о снятии метки с предыдущего.
- Вкладка «График» отображает рабочие дни, промежутки и перерывы в наглядном недельном виде; форма редактирования поддерживает несколько интервалов на день.
- Действия, требующие кадрового приказа (назначение/снятие должности), показывают блок перехода в ЭДО без встраивания самого редактора документов.
- Для каждой вкладки предусмотрены состояния: пусто, загрузка, ошибка, нет прав.
