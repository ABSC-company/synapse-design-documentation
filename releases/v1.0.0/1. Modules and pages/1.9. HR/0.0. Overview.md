# 0. HR Module — Обзор и навигация

## Техническое задание для дизайнера

* **Модуль:** HR Module (управление персоналом)
* **Платформа:** веб-приложение медицинской системы (внутренняя админка клиники)
* **Основные роли:** Администратор / HR-менеджер, Руководитель отдела, Сотрудник (просмотр своих данных)
* **Статус документа:** Draft

## Что НЕ входит в этот пакет

* **Создание и загрузка файлов** (фото, сканы документов, архивы) — реализованы отдельным File-модулем. В ТД эти места отмечены как стандартный паттерн «Загрузить файл», без описания внутреннего устройства загрузчика.
* **Документооборот (EDM)** — кадровые приказы (приём, перевод, отпуск, командировка, увольнение) создаются и подписываются в отдельном модуле ЭДО. В этом ТД указаны только точки входа — кнопки и переходы в ЭДО, без описания самого редактора документов и реестра.
* **Орг. структура клиники** (отделы, должности, рабочие места, субординация) — управляется в Structure Module. HR Module только использует справочники оттуда (выпадающие списки выбора отдела/должности/рабочего места).

## Карта навигации HR-модуля

```
┌────────────────────────────────────────────────────────────┐
│  Боковое меню (общее для всей системы)                     │
│                                                              │
│  HR / Персонал                                              │
│   ├─ Сотрудники            → 01. Реестр сотрудников         │
│   ├─ Штатное расписание    → 04. Штатное расписание         │
│   ├─ Шаблоны инструктажей  → 03. Реестр шаблонов инструктажей│
│   └─ Архив уволенных       → 05. Архив                      │
└────────────────────────────────────────────────────────────┘
```

При открытии карточки сотрудника из реестра пользователь попадает на **Страницу профиля сотрудника** — единую страницу с вложенными вкладками (tabs). Все разделы, связанные с конкретным сотрудником (должности, контакты, график, специализации врача, документы, инструктажи, учёт времени), находятся внутри профиля, а не на отдельных роутах.

```
Реестр сотрудников
       │
       │ клик по строке/карточке
       ▼
Профиль сотрудника  ──────────────────────────────────┐
 ┌───────────────────────────────────────────────────┐ │
 │ Шапка профиля (фото, ФИО, статус, branch)          │ │
 ├───────────────────────────────────────────────────┤ │
 │ [Основное] [Должности] [Контакты] [Рабочие места]  │ │
 │ [График] [Специализации]* [Допуски]* [Аттестации]  │ │
 │ [Документы] [Инструктажи] [Учёт времени]           │ │
 └───────────────────────────────────────────────────┘ │
                                                        │
   * вкладки «Специализации» и «Допуски» видны          │
     только если employment_type = doctor               │
                                                        ▼
                                          Действие «Увольнение» →
                                          см. 05-dismissal-and-archive.md
```

## Общие UI-паттерны (используются во всех разделах)

Чтобы дизайн был консистентным во всех экранах HR-модуля, ниже зафиксированы общие правила. В каждом последующем файле эти паттерны переиспользуются по ссылке, без повторного описания.

### Паттерн «Реестр» (список сущностей)

Применяется к: списку сотрудников, штатному расписанию, шаблонам инструктажей, архиву уволенных.

```
┌─────────────────────────────────────────────────────────────────┐
│  Заголовок страницы                          [+ Кнопка создания]│
├─────────────────────────────────────────────────────────────────┤
│  [Вкладки/табы, если есть]                                       │
│  🔍 Поиск...                              [Фильтр ▾] [Сортировка]│
├─────────────────────────────────────────────────────────────────┤
│  Таблица / список карточек                                       │
│  ...                                                              │
├─────────────────────────────────────────────────────────────────┤
│  Пагинация                                                       │
└─────────────────────────────────────────────────────────────────┘
```

- Поиск — debounce 300–500 мс, ищет по релевантным полям (имя, фамилия и т.д. — указано в каждом разделе отдельно).
- Фильтр открывается в Driver-компоненте (боковая выезжающая панель) или dropdown-панели — конкретный вид указан в каждом разделе.
- Списки с потенциально большим объёмом строк (сотрудники, учёт времени) — серверная пагинация или infinite scroll (решение дизайнера/фронтенда, в ТД зафиксирована необходимость пагинации, конкретный механизм не предписан).

### Паттерн «Карточка-строка реестра»

- Содержит: ключевые идентифицирующие поля сущности + статус (если применимо) + контекстное меню или быстрые действия справа.
- При клике на строку — переход в карточку/профиль или открытие модального окна с деталями (указано отдельно по разделу).
- Hover на строку — лёгкая подсветка фона.

### Паттерн «Форма создания/редактирования» (модальное окно)

Применяется ко всем CRUD-операциям внутри вкладок профиля (контакты, рабочие места, график, специализации, допуски, аттестации, документы) — везде, где сущность простая и не требует отдельной страницы.

```
┌────────────────────────────────────────┐
│  Заголовок действия                [✕] │
├────────────────────────────────────────┤
│                                        │
│  Поле 1 *                              │
│  Поле 2                                │
│  ...                                   │
│                                        │
├────────────────────────────────────────┤
│              [Отмена]      [Сохранить] │
└────────────────────────────────────────┘
```

- Обязательные поля помечены `*`.
- Кнопка «Сохранить» неактивна, пока форма не валидна.
- Удаление сущности — отдельное подтверждающее модальное окно («Вы уверены? Это действие нельзя отменить» + [Отмена] / [Удалить]), если удаление физически необратимо. Если в тех. задании сущность не удаляется физически, а архивируется/деактивируется — кнопка подписана соответствующим действием, не «Удалить».

### Паттерн «Статус-бейдж»

Используется для отображения статуса сотрудника, статуса аттестации, статуса допуска и т.д. Бейдж — небольшая капсула (pill) с цветной заливкой и текстом статуса.

Цветовая семантика (единая для всего модуля):

| Семантика | Где используется | Цвет (рекомендация) |
|-----------|-------------------|-----------------------|
| Активно / в порядке | `at_work`, аттестация `scheduled`-в-будущем, документ действующий | Зелёный |
| Внимание / скоро истекает | `on_leave`, `on_trip`, документ истекает в ближайшие N дней | Жёлтый/янтарный |
| Критично / просрочено | `overdue`, `failed`, документ просрочен | Красный |
| Нейтрально / неактивно | `away`, `dismissed`, бессрочный документ | Серый |

> Конкретное сопоставление статус → цвет см. в соответствующем разделе (статусы сотрудника заданы в тех. задании п.6, статусы аттестации — п.5.8).

### Паттерн «Пустое состояние» (Empty state)

Везде, где список/таблица не содержит записей:

```
┌─────────────────────────────────────┐
│                                     │
│          [иллюстрация/иконка]       │
│       Пока здесь ничего нет         │
│   [Краткое поясняющее подзаголовок] │
│        [+ Кнопка создания]          │
│                                     │
└─────────────────────────────────────┘
```

### Паттерн «Загрузка» (Loading state)

- Скелетоны (skeleton screens), повторяющие форму контента — для списков, карточек, профиля.
- Спиннер — для точечных действий внутри модальных окон (сохранение формы).

### Паттерн «Ошибка»

- Для ошибки загрузки страницы/списка — блок с иконкой ошибки, текстом и кнопкой «Повторить».
- Для ошибки валидации формы — инлайн-текст под полем, красная обводка поля.
- Для ошибки сетевого действия (сохранение не удалось) — toast-уведомление снизу/сверху экрана.

### Паттерн «Нет прав доступа»

Если у пользователя нет права на просмотр раздела (см. таблицу прав в тех. задании, раздел 13) — вкладка/раздел не отображается в навигации вовсе (а не показывается заблокированной). Если право есть на просмотр, но нет на редактирование — все интерактивные элементы (кнопки добавить/редактировать/удалить) скрываются или показываются disabled с tooltip «Недостаточно прав».

### Паттерн «Загрузка файла» (точка интеграции с File Module)

Везде, где требуется приложить файл (фото сотрудника, скан документа, сертификат специализации, сертификат допуска) — используется единый компонент:

```
┌─────────────────────────────────────────┐
│   📎  Перетащите файл или нажмите        │
│       для выбора                        │
│   PDF, JPG, PNG до N МБ                  │
└─────────────────────────────────────────┘
```

После загрузки — превью файла (миниатюра для изображений, иконка типа файла для документов) + название + кнопки [Заменить] [Удалить]. Поведение загрузки (прогресс, обработка ошибок загрузки) определяется File Module и в данном ТД не детализируется.

### Паттерн «Переход в ЭДО» (точка интеграции с EDM Module)

Везде, где кадровое действие должно сопровождаться приказом (приём, перевод, отпуск, командировка, увольнение — см. тех. задание, раздел 7), в форме действия присутствует блок:

```
┌─────────────────────────────────────────┐
│  Кадровый документ                       │
│  Это действие требует приказа.           │
│  [Создать документ в ЭДО →]              │
└─────────────────────────────────────────┘
```

- Кнопка открывает создание документа в модуле ЭДО (выбор шаблона категории `hr` / создание с нуля — экран самого ЭДО, не входит в этот пакет).
- В каждом конкретном разделе (приём, перевод, увольнение и т.д.) указано, обязательна ли привязка документа для завершения действия, или документ можно создать позже.

## Acceptance criteria — общие (для всех экранов модуля)

- Навигация HR-модуля в боковом меню содержит 4 пункта: Сотрудники, Штатное расписание, Шаблоны инструктажей, Архив уволенных.
- Профиль сотрудника — единая страница с вкладками, без переходов на отдельные роуты для должностей/контактов/графика/документов и т.д.
- Вкладки «Специализации» и «Допуски к услугам» отображаются в профиле только если сотрудник имеет тип `doctor`.
- Все реестры (списки) поддерживают поиск, фильтрацию, пагинацию и три состояния: пусто / загрузка / ошибка.
- Статус-бейджи во всех разделах используют единую цветовую семантику (зелёный/жёлтый/красный/серый).
- Все формы создания/редактирования — модальные окна с обязательными полями, помеченными `*`, и неактивной кнопкой сохранения до прохождения валидации.
- Удаление необратимых сущностей сопровождается подтверждающим модальным окном.
- Везде, где нужно приложить файл, используется единый паттерн загрузки (drag & drop + клик), без дублирования логики самого File Module.
- Везде, где кадровое действие требует приказа, присутствует кнопка перехода в ЭДО, без встраивания самого редактора документов в HR-экраны.
- Элементы, на которые у пользователя нет прав, либо скрыты (если нет прав на просмотр), либо неактивны с поясняющим tooltip (если нет прав на изменение).
