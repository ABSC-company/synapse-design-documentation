## Пользовательские сценарии

**1. Открытие смены**
- *Основной поток:* при первом входе в кассовый интерфейс система требует открыть смену (кассир указывает сумму размена наличных) → создаются `Shift(открыта)`, `ZReport(открыт)` и первый `XReport(открыт)` дня.
- *Альтернативный поток:* попытка перейти к любому кассовому действию без открытой смены → блокирующее сообщение с предложением открыть смену.

**2. Визит доходит до кассира**
- *Основной поток:* пациент прошёл регистрацию и получил направление на услугу → переходит к отдельному кассиру, либо (в зависимости от бизнес-процесса клиники) тот же сотрудник регистратуры переключается в режим «Касса» на своём рабочем месте (см. §4, «Совмещение с регистратурой») → визит появляется в списке «ЭМК к оплате» с суммой задолженности.

**3. Поиск визита/ЭМК к оплате**
- *Основной поток:* кассир открывает список ЭМК, готовых к оплате — по умолчанию все визиты клиники с непогашенной задолженностью (не только своего отделения/кассы, для скорости обслуживания любого пациента), либо ищет конкретный визит по любому из параметров (ФИО, ПИНФЛ, номер ЭМК, номер визита, дата, отделение).

**4. Приём оплаты**
- *Основной поток:* кассир открывает найденный визит → видит перечень оказанных услуг и сумму долга по счёту (`Invoice`) → нажимает «Оплатить» → перед ним выбор способа оплаты:
  - *наличные* — ввод полученной суммы, расчёт сдачи, немедленное подтверждение;
  - *терминал* — инициируется операция на POS-терминале, ожидание подтверждения оплаты картой;
  - *онлайн (Payme/Click/Uzum)* — генерация ссылки/QR, платёж переходит в `ожидает_подтверждения` до колбэка от платёжной системы.
- *Частичная оплата:* кассир может ввести сумму меньше остатка по счёту — счёт переходит в `частично_оплачен`, визит остаётся в списке «ЭМК к оплате» с уменьшенным долгом; пациент донесёт остаток позже (тем же или другим способом).
- *Результат (полное погашение):* платёж `оплачен`, счёт погашен полностью → авто-печать чека → задолженность по визиту снимается, визит пропадает из списка «ЭМК к оплате» (событие наружу).
- *Альтернативный поток:* платёжная система вернула отказ или не ответила за отведённое время → платёж `отменён`, кассиру приходит уведомление через `NOTIF`, он предлагает пациенту другой способ, долг остаётся.

**5. Отложить оплату**
- *Основной поток:* кассир помечает платёж как `отложен` без закрытия визита (например, пациент вышел доплатить) → визит остаётся в «Отложенных платежах» текущей смены, долг за ним сохраняется.

**6. Отменить платёж**
- *Основной поток:* кассир отменяет черновик/отложенный платёж с указанием причины, до перехода в `оплачен`; долг остаётся.

**7. Возврат оплаты (по сумме или по конкретной услуге)**
- *Предусловия:* платёж в статусе `оплачен`.
- *Основной поток:* старший кассир открывает оплаченный визит → инициирует возврат → выбирает один из двух вариантов:
  - *по сумме* — вводит произвольную сумму возврата (полную или частичную) и причину;
  - *по услугам* — отмечает в перечне счёта конкретную услугу (например, одну из группы оказанных), сумма возврата считается автоматически по отмеченным позициям.
- *Результат:* чек аннулируется/выпускается чек-возврат, платёж переходит в `возвращён` (при полном возврате) либо счёт возвращается в `частично_оплачен` с новым остатком долга (при частичном/по услугам).

**8. Перерыв / пересменка — закрытие X-отчёта**
- *Основной поток:* кассир отходит от рабочего места или передаёт кассу другому кассиру → нажимает «Закрыть X-отчёт» → принтер печатает текущий X-отчёт (итоги за этот период) → кассир закрывает кассу и уходит.
- *Продолжение:* система немедленно открывает новый X-отчёт в рамках той же смены — либо для этого же кассира при возврате, либо для заступившего кассира; Z-отчёт и смена продолжают накапливать итоги без перерыва.

**9. Закрытие Z-отчёта (конец рабочего дня)**
- *Основной поток:* кассир (в одиночной кассе) или старший кассир (в мультикассовом режиме) закрывает смену → система автоматически формирует Z-отчёт с итогами по способам оплаты и печатает его на принтере → смена переходит в `закрыта`, дальнейший приём платежей по ней недоступен.

**10. Восстановление ошибочно закрытого Z-отчёта**
- *Предусловия:* Z-отчёт/смена закрыты преждевременно по ошибке кассира.
- *Основной поток:* ответственный сотрудник (сам кассир — если касса в клинике одна; старший кассир — если касс несколько, см. §4) открывает «Управление сменами» → находит закрытую смену → «Восстановить» → указывает причину → смена и Z-отчёт возвращаются в статус `открыта`/`открыт`, работа продолжается; факт восстановления фиксируется в аудите.

**11. Обработка колбэка от платёжной системы (технический сценарий)**
- *Предусловия:* платёж в статусе `ожидает_подтверждения`, ранее отправлен запрос в Payme/Click/Uzum с `idempotency_key`.
- *Основной поток:* платёжная система присылает колбэк с результатом и `транзакция_id` → система находит `Payment` по `idempotency_key` → сверяет сумму → при успехе переводит платёж в `оплачен` (см. сценарий 4, «Результат») → при неуспехе — в `отменён` с фиксацией причины из ответа.
- *Альтернативный поток:* повторный колбэк с тем же `транзакция_id` (платёжные системы могут слать колбэк более одного раза) → обрабатывается идемпотентно, повторного перехода статуса и повторного чека не происходит.
- *Альтернативный поток:* колбэк не пришёл в течение таймаута → платёж остаётся в `ожидает_подтверждения`, кассир получает уведомление через `NOTIF`, ему доступно ручное «Проверить статус» (повторный запрос статуса у платёжной системы) или отмена и повтор оплаты другим способом. Автоматического повтора запроса система не делает.

**12. Сверка наличности при закрытии смены (инкассация, опционально)**
- *Основной поток:* при закрытии Z-отчёта система показывает расчётную сумму наличных (`сумма_наличных_расчётная` = размен + наличные платежи − наличные возвраты); поле фактической суммы необязательно — кассир может закрыть смену, не заполняя его.
- *Если сверка проводится:* кассир/старший кассир пересчитывает физический денежный ящик и вводит `сумма_наличных_фактическая` → при расхождении с расчётной система требует обязательный комментарий (`комментарий_расхождения`) перед закрытием → всё фиксируется вместе с Z-отчётом.
