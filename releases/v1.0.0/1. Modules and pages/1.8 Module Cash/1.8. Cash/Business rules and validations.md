##  Бизнес-правила и валидации

| Правило | Проверка / поведение | Пример сообщения об ошибке |
|---|---|---|
| Обязательное открытие смены | Любое кассовое действие (поиск к оплате, приём платежа, отчёты) заблокировано, пока для текущего кассира/рабочего места нет смены в статусе `открыта` | «Откройте смену, чтобы продолжить работу с кассой» |
| Приём оплаты только в открытую смену/Z-отчёт | Проведение `Payment` в статус `оплачен` невозможно, если Shift/ZReport не в статусе `открыта`/`открыт` | «Смена закрыта, приём оплаты недоступен» |
| Оплата снимает задолженность | После перехода платежа в `оплачен` — печатается чек; если счёт (`Invoice`) полностью погашен, он переходит в статус `оплачен`, визит пропадает из списка «ЭМК к оплате» (событие наружу, см. §9); это не клиническое закрытие ЭМК, только административный статус оплаты | — |
| Частичная оплата поддерживается | Кассир может провести `Payment` на сумму меньше остатка по счёту — счёт переходит в `частично_оплачен`, визит остаётся в списке «ЭМК к оплате» с уменьшенной суммой долга | — |
| Счёт может дорастать новыми услугами того же визита | Если в рамках одного визита пациенту оказали ещё одну услугу, `EMR`/`REG` может добавить её в уже открытый (не оплаченный/не отменённый) `Invoice` — `сумма_итого` увеличивается, долг растёт соответственно. Это удобный вариант по умолчанию — если для конкретной клиники неудобно так вести учёт, услугу оформляют новым визитом, и тогда для него создаётся отдельный, независимый счёт | — |
| Список «к оплате» показывает только должников, по всей клинике | В «ЭМК к оплате» по умолчанию — все визиты клиники с непогашенной задолженностью (не только своего отделения/кассы), чтобы кассир мог быстро обслужить любого пациента; доступны фильтры по отделению/дате для сужения; как только долг закрыт — визит исчезает из списка | — |
| Подтверждение онлайн-оплаты — только по колбэку | Статус `оплачен` для `онлайн_*` не выставляется по действию кассира — только по ответу платёжной системы; повторный колбэк с тем же `idempotency_key`/`транзакция_id` не создаёт второй платёж | «Ожидание подтверждения от платёжной системы» |
| Таймаут/отказ онлайн-оплаты — уведомление, не авто-повтор | Если колбэк не пришёл в течение таймаута или система вернула отказ, кассир получает уведомление через `NOTIF`; автоматического повтора платежа нет — кассир выбирает следующий шаг вручную (повторить/сменить способ) | «Оплата не подтверждена платёжной системой — уведомление отправлено» |
| Отложенный платёж не закрывает визит | Визит остаётся доступным для оплаты позже, кассир находит его через поиск/«Отложенные платежи» | — |
| Отмена — только до оплаты | Платёж в статусе `оплачен` нельзя отменить — только через `Refund` | «Платёж уже проведён, для отмены оформите возврат» |
| Возврат обязателен с причиной | `Refund` не создаётся без заполненного поля `причина` | «Укажите причину возврата» |
| Возврат — по сумме или по конкретным услугам | `Refund` может либо указывать произвольную сумму (`тип_возврата = по_сумме`), либо ссылаться на конкретные позиции счёта (`тип_возврата = по_услугам`) — например, вернуть одну услугу из группы, оставив остальные оплаченными | — |
| Возврат не может превышать остаток по счёту | Сумма всех `Refund` по одному `Payment`/`Invoice` (нарастающим итогом) ≤ фактически оплаченной сумме | «Сумма возврата превышает оплаченную сумму» |
| Разбивка отчётов по способам оплаты | X- и Z-отчёт агрегируют суммы отдельно по каждому способу (наличные / терминал / Payme / Click / Uzum) — для сверки с фактической выручкой | — |
| Закрытие X-отчёта не закрывает смену, новый открывается сам | Кассир печатает текущий X-отчёт и уходит/передаёт кассу; система тут же открывает следующий X-отчёт той же смены — учёт смены непрерывен, ручного «начать заново» не требуется | — |
| Закрытие Z-отчёта закрывает смену | После закрытия дальнейший приём платежей по этой смене невозможен, пока смена не восстановлена | «Смена уже закрыта» |
| Восстановление Z-отчёта — правило зависит от режима кассы | Мультикассовый режим включается автоматически при **2 и более** кассовых рабочих местах **в рамках одного филиала/корпуса** (не по организации в целом — в мультифилиальной клинике режим определяется отдельно для каждого филиала). При одной кассе в филиале восстановить ошибочно закрытый Z-отчёт может сам кассир; при мультикассовом режиме — только роль «Старший кассир». В обоих случаях требуется указание причины и фиксация в аудите | «Недостаточно прав для восстановления Z-отчёта» |
| Кассиру не показывается клиническая часть ЭМК | Из ЭМК в кассовый интерфейс передаются только пациент, перечень оказанных услуг и сумма — принцип минимальных привилегий `[SYS-RBAC]`, `[SYS-PDATA]` | — |
| Цены фиксируются на момент выставления счёта | `invoice.перечень_услуг` не пересчитывается, даже если прайс в `SVC` изменится до фактической оплаты | — |
| Сверка наличности — опциональна, комментарий обязателен при расхождении | Ввод `сумма_наличных_фактическая` при закрытии смены необязателен. Если введена и не совпадает с `сумма_наличных_расчётная` — поле `комментарий_расхождения` обязательно | «Укажите комментарий к расхождению суммы наличных» |
| Фискализация чеков обязательна | Каждый `Receipt` и итоговый `ZReport` формируются через онлайн-ККМ (фискальный регистратор) — печать без фискализации не допускается (см. §7) | — |
| Аудит всех денежных операций | Открытие/закрытие смены, платежи, возвраты, восстановление Z-отчёта — под неизменяемый аудит-трейл `[SYS-AUDIT]` | — |
