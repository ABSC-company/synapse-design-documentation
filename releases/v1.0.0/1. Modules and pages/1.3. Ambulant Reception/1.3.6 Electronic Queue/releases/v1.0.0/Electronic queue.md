# 1.3.6. Electronic Queue

## Спецификация сценария интерфейса

- Сценарий: Electronic Queue (Электронная очередь)
- Модуль: Амбулаторная регистратура
- Тип сценария: Сквозной пользовательский сценарий
- Основные роли: Пациент, регистратор, врач, старший регистратор, администратор регистратуры
- Статус документа: Draft

---

# Назначение сценария и сценарии использования

Сценарий **Electronic Queue** предназначен для организации движения пациента внутри клиники с момента получения талона электронной очереди до начала приёма у врача.

Сценарий объединяет несколько интерфейсов, которые работают как единая система и автоматически синхронизируются между собой.

В состав сценария входят:

1. **Kiosk** — терминал выдачи талонов электронной очереди.
2. **Табло вызова** — публичный экран в зоне ожидания.
3. **Табло у кабинета врача**.
4. **Кнопка вызова пациента** — элемент рабочего места врача.

Основная задача сценария — обеспечить пациенту понятный маршрут по клинике без необходимости обращаться к сотрудникам за дополнительными инструкциями.

## Ключевые задачи сценария

1. Получение номера электронной очереди через терминал самообслуживания.
2. Управление очередью регистратором.
3. Информирование пациентов о вызове к окну регистрации.
4. Передача номера электронной очереди в визит пациента.
5. Информирование пациента возле кабинета врача.
6. Вызов следующего пациента врачом.

---

## Общий сценарий

1. Пациент подходит к терминалу электронной очереди.
2. Получает талон с номером очереди.
3. Номер автоматически появляется в электронной очереди регистратора.
4. Регистратор вызывает следующего пациента.
5. На публичном табло отображается соответствие **«номер талона → номер окна»**.
6. Пациент подходит к указанному окну.
7. После регистрации создаётся визит в клинику и визит к врачу.
8. Номер электронной очереди привязывается к созданному визиту.
9. Пациент направляется к кабинету врача.
10. Когда врач готов принять следующего пациента, он нажимает кнопку **«Вызвать пациента»**.
11. Табло возле кабинета обновляет состояние врача.

# User Flows / Task Flows

## Flow 1. Получение талона электронной очереди

1. Пациент подходит к терминалу.
2. На экране отображается кнопка **«Получить очередь в регистратуру»**.
3. Пациент нажимает кнопку.
4. Система создаёт новый номер электронной очереди.
5. Киоск отправляет талон на печать.
6. После завершения печати пациент получает чек.
7. На экране отображается номер выданного талона.
8. Через несколько секунд интерфейс автоматически возвращается в исходное состояние.

---

## Flow 2. Вызов пациента к окну регистрации

1. Пациент ожидает своей очереди.
2. Регистратор вызывает следующего посетителя.
3. Система автоматически обновляет публичное табло.
4. На табло отображается соответствие **«номер талона → номер окна регистрации»**.
5. Новый вызов визуально выделяется относительно предыдущих.
6. Пациент подходит к указанному окну.

---

## Flow 3. Регистрация пациента

1. Регистратор открывает визит пациента.
2. Выполняет регистрацию.
3. Система связывает номер электронной очереди с визитом.
4. После завершения регистрации пациент получает информацию о кабинете врача.

---

## Flow 4. Ожидание возле кабинета врача

1. Пациент подходит к кабинету.
2. Табло отображает номер кабинета.
3. Табло отображает карточку врача.
4. Пациент ожидает вызова.

Если кабинет временно не используется, отображается состояние **«Кабинет свободен»**.

---

## Flow 5. Вызов пациента врачом

1. Врач завершает текущий приём.
2. Если в очереди есть ожидающие пациенты, кнопка **«Вызвать пациента»** становится активной.
3. Врач нажимает кнопку.
4. Система вызывает следующего пациента.
5. Табло возле кабинета обновляет информацию о состоянии кабинета.

Если ожидающих пациентов нет, кнопка остаётся неактивной.

# Структура: Wireframes

Структурные блоки сценария:

1. **Kiosk** — терминал выдачи талонов.
2. **Табло вызова** — публичный экран зоны ожидания.
3. **Табло у кабинета врача**.
4. **Кнопка вызова пациента** — элемент рабочего места врача.

---

## Структура: Kiosk

Экран состоит из одного основного действия.

### Состояние по умолчанию

- Логотип клиники (опционально).
- Крупная кнопка **«Получить очередь в регистратуру»**.

### Состояние печати

- Индикатор процесса печати.
- Информационное сообщение о подготовке талона.

### Успешная печать

- Крупное отображение номера талона.
- Краткое сообщение об успешной печати.
- Автоматический возврат в начальное состояние.

### Ошибка

- Сообщение о невозможности распечатать талон.
- Рекомендация обратиться к регистратору.

---

## Структура: Табло вызова

Экран состоит из следующих элементов:

- Заголовок.
- Список последних вызовов.
- Текущий вызов.

Структура строки вызова:

| Поле | Описание |
|------|----------|
| Номер талона | Уникальный номер пациента |
| Окно | Номер окна регистрации |

Правила отображения:

1. Последний вызов отображается крупнее остальных.
2. История содержит несколько предыдущих вызовов.
3. Повторный вызов имеет отдельный визуальный акцент.

---

## Структура: Табло кабинета врача

Экран состоит из карточки кабинета.

Карточка отображает:

- номер кабинета;
- фотографию врача;
- ФИО;
- специализацию;
- статус приёма.

Возможные состояния:

- Идёт приём.
- Врач свободен.
- Перерыв.
- Приём окончен.
- Кабинет свободен.

При отсутствии фотографии используется плейсхолдер.

---

## Структура: Кнопка вызова пациента

Элемент располагается в карточке текущего приёма врача.

Содержит:

- кнопку **«Вызвать пациента»**;
- счётчик ожидающих пациентов.

Состояния:

- Active — есть ожидающие пациенты.
- Disabled — очередь отсутствует.


# Визуал: High-Fidelity mockups

## Общие визуальные требования

Все экраны сценария должны использовать единый визуальный язык системы и соответствовать дизайн-системе продукта.

При проектировании необходимо учитывать среду использования каждого интерфейса.

### Kiosk

1. Интерфейс должен содержать только одно основное действие.
2. Кнопка получения талона является главным визуальным элементом экрана.
3. Пользователь должен понимать сценарий использования за 1–2 секунды.
4. Не допускается использование второстепенных CTA или элементов, отвлекающих внимание.
5. Все состояния должны быть максимально простыми и понятными.

### Табло вызова

1. Текущий вызов является главным визуальным элементом экрана.
2. Размер шрифта должен обеспечивать чтение информации с большого расстояния.
3. Новый вызов должен визуально отличаться от истории.
4. История вызовов должна оставаться читаемой, но иметь меньший визуальный приоритет.
5. Интерфейс должен содержать минимальное количество декоративных элементов.

### Табло кабинета врача

1. Карточка врача должна использовать единый стиль с другими разделами системы.
2. Фотография врача имеет фиксированные пропорции.
3. Статус врача должен отображаться в виде цветного бейджа.
4. Номер кабинета должен быть наиболее заметным элементом экрана.
5. При отсутствии фотографии отображается стандартный аватар.

### Кнопка вызова пациента

1. Кнопка должна визуально выделяться среди остальных действий рабочего места врача.
2. Счётчик ожидающих пациентов располагается рядом с кнопкой.
3. Disabled-состояние должно явно показывать недоступность действия.

# Состояния и Edge Cases

## Базовые состояния сценария

| Состояние | Ожидаемое поведение |
|-----------|---------------------|
| loading | Отображается Skeleton соответствующего экрана до получения данных |
| empty | Показывается сообщение об отсутствии данных |
| error | Отображается сообщение об ошибке с возможностью повторить действие |
| offline | Показывается сообщение о временной недоступности сервиса |
| permissions | Недоступные действия скрываются или блокируются согласно роли пользователя |

---

## Edge Cases: Kiosk

1. Во время печати пропало соединение с сервером.
2. Закончилась бумага.
3. Принтер отключён.
4. Принтер завис во время печати.
5. Пользователь нажал кнопку несколько раз подряд.
6. Пользователь покинул терминал до окончания печати.
7. Система успешно создала талон, но печать завершилась ошибкой.
8. Во время печати произошла перезагрузка терминала.
9. Не удалось получить новый номер очереди.
10. Недоступен сервис электронной очереди.

---

## Edge Cases: Табло вызова

1. Нет ожидающих пациентов.
2. Получен повторный вызов пациента.
3. Одновременно вызваны пациенты в разные окна.
4. Очередь приостановлена.
5. Все окна временно закрыты.
6. Табло потеряло соединение с сервером.
7. При восстановлении соединения отображаются актуальные данные.
8. Во время обновления поступил новый вызов.
9. История вызовов превышает максимально допустимое количество строк.
10. Отображается очень длинный номер талона.

---

## Edge Cases: Табло кабинета врача

1. Кабинет временно свободен.
2. У врача отсутствует фотография.
3. Врач ушёл на перерыв.
4. Приём завершён.
5. Произошла смена врача.
6. В кабинете работает другой врач по замещению.
7. В системе отсутствует специализация врача.
8. Не удалось загрузить фотографию врача.
9. Недоступен сервис получения статуса врача.
10. Кабинет временно закрыт.

---

## Edge Cases: Кнопка вызова пациента

1. Очередь отсутствует.
2. Во время вызова следующий пациент уже был вызван другим врачом.
3. Врач нажал кнопку повторно до завершения предыдущего вызова.
4. Во время вызова потеряно соединение.
5. Очередь изменилась другим оператором.
6. Вызов пациента завершился ошибкой API.
7. Очередь была очищена автоматически.
8. Следующий пациент отменил визит до вызова.

---

# Interaction & Animation

## Правила поведения

1. Все экраны автоматически обновляют данные без перезагрузки страницы.
2. Табло вызова обновляется сразу после нового вызова регистратора.
3. Табло кабинета автоматически изменяет статус врача.
4. Кнопка вызова пациента становится активной сразу после появления ожидающего пациента.
5. После успешного вызова количество ожидающих пациентов обновляется автоматически.

---

## Переходы и анимации

### Kiosk

1. После нажатия кнопки отображается индикатор печати.
2. После успешной печати отображается экран подтверждения.
3. Через несколько секунд выполняется автоматический возврат на главный экран.

### Табло вызова

1. Новый вызов появляется с короткой fade-анимацией.
2. Новый вызов кратковременно увеличивается относительно остальных строк.
3. Повторный вызов выделяется дополнительным цветовым акцентом.

### Табло кабинета

1. Изменение статуса сопровождается плавной сменой бейджа.
2. При смене врача карточка обновляется с короткой fade-анимацией.
3. Обновление фотографии происходит без мерцания интерфейса.

### Рабочее место врача

1. После нажатия кнопки отображается состояние загрузки.
2. После успешного вызова обновляется счётчик ожидающих пациентов.
3. При ошибке отображается Toast с причиной ошибки.

1. Kiosk отображает единственную кнопку получения талона.
2. После нажатия создаётся новый номер электронной очереди.
3. После успешной печати отображается номер талона.
4. Интерфейс автоматически возвращается в начальное состояние.
5. При отсутствии бумаги отображается соответствующее сообщение.
6. При ошибке печати пользователь получает инструкцию обратиться в регистратуру.
7. Табло вызова отображает последний вызов крупнее остальных.
8. История вызовов содержит несколько последних записей.
9. Повторный вызов имеет отдельный визуальный акцент.
10. Табло автоматически обновляется без перезагрузки страницы.
11. Табло кабинета отображает номер кабинета.
12. Табло отображает карточку врача.
13. При отсутствии фотографии используется плейсхолдер.
14. При отсутствии врача отображается состояние «Кабинет свободен».
15. Статус врача обновляется автоматически.
16. Кнопка «Вызвать пациента» активна только при наличии очереди.
17. Счётчик отображает количество ожидающих пациентов.
18. После вызова счётчик автоматически обновляется.

21. Все статусы используют единую цветовую систему продукта.
22. Карточка врача соответствует карточкам врача в других модулях.

24. Публичные табло читаемы с большого расстояния.
25. Все экраны автоматически синхронизируются между собой.
