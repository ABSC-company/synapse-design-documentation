## 11. Зависимости и ограничения

### API

1. API получения агрегированных KPI за период и набор фильтров (единый агрегирующий запрос от `RPT`).
2. API получения динамики визитов по временным точкам.
3. API получения динамики выручки по временным точкам.
4. API получения распределения по отделениям.
5. API получения загрузки врачей.
6. API получения топ услуг (по количеству и по выручке).
7. API получения активных предупреждений.
8. API детализации (drill-down) по конкретному KPI/сегменту/строке, с учётом уровня доступа роли.
9. API запуска и получения статуса формирования экспортного файла.
10. API получения оргструктуры для фильтра филиал/подразделение (см. `MGMT`).

### Роли и права

- **Руководство клиники** — полный доступ ко всем показателям всех подразделений, включая персональную детализацию.
- **Администратор** — полный доступ ко всем показателям; фокус на операционных и очередных метриках.
- **Старший регистратор** — доступ к операционным показателям (визиты, очередь, no-show), ограниченный доступ к финансовому блоку.
- **Финансовый отдел** — полный доступ к финансовому блоку и агрегированным операционным показателям, без персональной детализации по пациентам.
- **Руководитель подразделения** — доступ ограничен собственным подразделением; фильтр филиала/отделения не позволяет переключиться на чужие подразделения.
- Персональная детализация (ФИО, контакты пациента в drill-down) доступна только ролям с отдельным правом на просмотр персональных данных — по умолчанию detalization агрегированная.

### Feature flags

1. Включение блока предупреждений.
2. Включение показателя «Пациенты в электронной очереди» (зависит от подключения модуля `QUEUE`).
3. Включение фильтра «Возрастная группа / пол» (зависит от политики данных клиники).
4. Включение экспорта в PDF / Excel по отдельности.
5. Включение drill-down детализации.
6. Включение сравнения с предыдущим периодом на KPI-карточках.

### Технические ограничения

1. Показатели за период — агрегаты с определённой периодичностью обновления (не мгновенный realtime), кроме отдельно помеченных «live»-показателей.
2. Большой произвольный период требует серверной агрегации по укрупнённым интервалам, чтобы не передавать на клиент сырые построчные данные.
3. Детализация (drill-down) и экспорт обязаны применять те же ограничения видимости данных, что и агрегированный экран — расхождений между уровнем детализации на экране и в выгрузке быть не должно.
4. При выгрузке малочисленных групп (например, один врач с единственным визитом за период) агрегирующий слой должен учитывать риск косвенной идентификации пациента и, при необходимости, скрывать или укрупнять такие срезы.
5. Формирование экспортного файла — асинхронная операция с ограничением по времени ожидания и повторной попыткой при таймауте.
